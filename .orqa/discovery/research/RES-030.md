## Problem Statement

The artifact system has hardcoded mappings at every layer:

- **Navigation store**: `GROUP_SUB_CATEGORIES`, `SUB_CATEGORY_LABELS`, `ARTIFACT_ACTIVITIES` — static config that must be updated for every new type
- **Backend**: 10+ separate scan/read commands, each knowing its own directory path
- **Store**: 12+ separate `$state` arrays with per-type loading flags
- **AppLayout**: Per-type `$effect` blocks with different caching logic
- **Activity bar**: Hardcoded icons and group assignments

Adding a new artifact type requires changes to: Rust commands, Tauri command registration, TypeScript types, store arrays, AppLayout effects, navigation store config, ArtifactNav categoryConfig, and the activity bar. This is the opposite of systems thinking.

**Root cause**: The navigation hierarchy is defined in code instead of being derived from the file system.

### README Frontmatter Schema

```yaml
---
role: group | artifacts       # group = contains sub-folders (types), artifacts = contains files
label: "Display Name"         # shown in activity bar and nav
description: "What this contains"
icon: "lucide-icon-name"      # Lucide icon for activity bar / nav headers
sort: 1                       # display order within parent (lower = first)
---

Markdown body = landing content shown when first visiting this group/type.
```

### Discovery Algorithm

The backend walks the file system once and builds a tree:

```
1. Read .orqa/README.md → root node
2. For each subdirectory of .orqa/:
   a. Read README.md frontmatter
   b. If role = "group": recurse into subdirectories, each expected to have role = "artifacts"
   c. If role = "artifacts": scan .md files (excluding README), build DocNode[] leaf list
3. Read docs/README.md → treat as a top-level group (or artifacts if no subdirectories with READMEs)
4. Sort everything by the `sort` field
5. Return the full navigation tree as one IPC response
```

Result: one `NavTree` structure that the frontend renders directly. No hardcoded mappings anywhere.

## The System

```
File System (.orqa/ and docs/ with README.md frontmatter)
    │
    ▼
One Backend Command: artifact_scan_tree() → NavTree
    │  Walks .orqa/ and docs/ recursively
    │  Reads README.md frontmatter at each level
    │  Builds hierarchical NavTree: groups → types → files
    │  Returns everything the frontend needs for navigation
    │
One Backend Command: artifact_read(path) → Artifact
    │  Reads a single file by relative path
    │  Returns full content for the viewer
    │
    ▼
One Store with Two Caches
    │  navTree: NavTree                      // full navigation hierarchy (loaded once)
    │  viewerCache: Map<path, Artifact>      // full content for opened files
    │  activeGroup: NavGroup | null
    │  activeType: NavType | null
    │  activePath: string | null
    │
    ▼
Activity Bar: rendered from navTree.groups[]
    │  Each group gets an icon from its README frontmatter
    │  No hardcoded GROUP_SUB_CATEGORIES
    │
    ▼
Nav Sub-Panel: rendered from activeGroup.types[]
    │  Each type button from the group's children
    │  No hardcoded SUB_CATEGORY_LABELS
    │
    ▼
File Browser: rendered from activeType.nodes[]
    │  DocNode[] — same shape for all types
    │  Title + description for every item
    │  Folders rendered as collapsible sections
    │
    ▼
Viewer: one pipeline for all types
    │  Metadata card → Title → Description → Markdown body
    │
    ▼
File Watcher: invalidates caches on disk changes
    │  Tauri fs::watch on .orqa/ and docs/
    │  On change: re-scan affected branch of the tree
    │  Svelte reactivity triggers re-render
```

### Data Types

```typescript
/** The full navigation tree returned by artifact_scan_tree */
interface NavTree {
  groups: NavGroup[];
}

/** A group folder (e.g. Planning, Governance) */
interface NavGroup {
  label: string;
  description: string;
  icon: string;
  sort: number;
  path: string;             // relative path to the group folder
  readme_content: string;   // README body for landing page
  types: NavType[];         // child artifact types
}

/** An artifact type folder (e.g. Epics, Rules) */
interface NavType {
  label: string;
  description: string;
  icon: string;
  sort: number;
  path: string;             // relative path to the type folder
  readme_content: string;   // README body for landing page
  nodes: DocNode[];         // the artifact files (flat or hierarchical)
}

/** A browsable artifact (leaf) or folder (branch) */
interface DocNode {
  label: string;
  path: string | null;
  description: string | null;
  children: DocNode[] | null;
}
```

### Invariants

1. **Every group and type folder has a README.md** with `role`, `label`, `description`, `icon`, `sort`
2. **One backend command for navigation**: `artifact_scan_tree()` returns the full tree
3. **One backend command for viewing**: `artifact_read(path)` returns one file
4. **Navigation is derived state** — the frontend renders what the backend discovers
5. **Adding a new type = create a folder with a README** — zero code changes
6. **Cache-first with file watcher invalidation** — fast browsing, automatic refresh
7. **README body is the landing content** — shown when a group/type is first selected

### What Disappears

| Current | Replaced By |
|---------|------------|
| `GROUP_SUB_CATEGORIES` hardcoded map | `navTree.groups[].types[]` from scan |
| `SUB_CATEGORY_LABELS` hardcoded map | `navType.label` from README frontmatter |
| `ARTIFACT_ACTIVITIES` hardcoded list | Derived from `navTree` |
| `activityToArtifactType` mapping | Not needed — types identified by path |
| 10+ separate Rust scan/read commands | 2 commands: `artifact_scan_tree` + `artifact_read` |
| 12+ `$state` arrays + loading flags | One `navTree` + one `viewerCache` Map |
| Per-type `$effect` blocks in AppLayout | One `$effect` for nav, one for viewer |
| `categoryConfig` in ArtifactNav | Rendered directly from `NavType` |
| `flatToNodes()` conversion | Backend returns `DocNode[]` for everything |
| `ArtifactSummary` type | `DocNode` is the universal browse shape |
| Per-type viewer routing | One generic viewer pipeline |
| `ActivityView` union type (19 members) | String-based, derived from folder names |
| `ActivityGroup` union type | String-based, derived from group folders |

## Migration: Current → Target

### What's Already Done (This Session)

- All 220 artifact files have `title` + `description` in YAML frontmatter
- Breadcrumb deduplication fixed
- Specialised viewer routing removed — all types use generic pipeline
- `extract_first_paragraph` fallback removed from all scanners
- README filtering added to all scanners

### Phase 2: Restructure .orqa/ Folders

Current `.orqa/` is flat:
```
.orqa/milestones/  .orqa/epics/  .orqa/tasks/  .orqa/agents/  .orqa/rules/  ...
```

Target `.orqa/` is grouped:
```
.orqa/delivery/milestones/  .orqa/delivery/epics/  .orqa/process/agents/  .orqa/process/rules/  ...
```

Steps:
1. Create group folders with README.md frontmatter
2. Move type folders into their groups (git mv)
3. Add README.md frontmatter to each type folder
4. Update any hardcoded `.orqa/` paths in the codebase

### Phase 3: Unified Backend

1. `artifact_scan_tree()` — walks the new folder structure, returns `NavTree`
2. `artifact_read(path)` — reads any file by relative path (mostly exists already)
3. Keep old commands temporarily for backward compat during migration

### Phase 4: Unified Store + Frontend

1. New `ArtifactStore` with `navTree` + `viewerCache`
2. Activity bar rendered from `navTree.groups[]`
3. Nav sub-panel rendered from `activeGroup.types[]`
4. ArtifactNav rendered from `activeType.nodes[]`
5. Remove all hardcoded navigation config

### Phase 5: File Watcher

1. Tauri `fs::watch` on `.orqa/` and `docs/`
2. On file change: re-scan affected branch, update `navTree`
3. On file change in viewed artifact: invalidate `viewerCache` entry

### Phase 6: Cleanup

1. Remove old per-type commands
2. Remove `ArtifactSummary` type (replaced by `DocNode`)
3. Remove `ActivityView` and `ActivityGroup` union types
4. Remove all hardcoded navigation maps

## Open Questions

1. **docs/ lives outside .orqa/** — should it move inside, or does `artifact_scan_tree` scan both roots?
   - Recommendation: scan both roots. docs/ is a convention many projects already have. The README frontmatter makes it discoverable either way.

2. **Skills are directories (skill-name/SKILL.md)** — how does the scanner handle these?
   - Recommendation: if a subfolder of an artifacts folder contains a SKILL.md (or any single .md), treat the folder as one artifact whose content is the .md file.

3. **Hooks are .sh files, not .md** — do they get YAML frontmatter?
   - Recommendation: hooks get a companion .md metadata file, or the scanner reads shell comments as metadata. Alternative: hooks folder is special-cased to scan .sh files.

4. **Cross-references between artifacts** ([EPIC-001](EPIC-001) references [MS-001](MS-001)) — do the paths change after restructuring?
   - The frontmatter IDs ([MS-001](MS-001), [EPIC-001](EPIC-001)) stay the same. Only `rel_path` values change. The `navigateToArtifact(id)` resolver needs to search the tree by ID instead of by hardcoded prefix mapping.
