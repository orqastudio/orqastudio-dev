# AD-6f6ff517: Skills Require Human-Facing Documentation

## Decision

Every skill artifact MUST have at least one `synchronised-with` relationship to a documentation artifact (`doc` type). This is enforced via a `required: true, minCount: 1` constraint on the `synchronised-with` relationship for skills in `core.json`.

## Context

Skills are agent-facing instructions — they tell AI agents how to behave in specific domains. But humans need to know what skills exist, what they do, and what authority they grant to agents. Without a human-facing equivalent, the agent's capabilities are invisible to the people who govern the system.

This creates a self-documenting app: the set of installed skills (and their human-facing docs) defines the app's capabilities in a way that's browsable by both humans and agents.

## How It Works

- Every skill (core, plugin, or project) must link to at least one `doc` artifact via `synchronised-with`
- Multiple skills can link to the same doc — a plugin doc can cover all its skills in a single reference table
- The doc doesn't need to duplicate the skill's content — it can be a summary, a table row, or a section in a larger document
- The integrity scanner flags skills without this relationship as errors

## Examples

**Single skill, dedicated doc:**
```yaml
# In planning.md (skill)
relationships:
  - target: DOC-434e00e1
    type: synchronised-with
```

**Multiple plugin skills, one doc:**
```yaml
# In svelte5-best-practices.md (skill)
relationships:
  - target: DOC-SVE-5d832d1d
    type: synchronised-with

# In svelte5-runes.md (skill)
relationships:
  - target: DOC-SVE-5d832d1d
    type: synchronised-with

# DOC-SVE-5d832d1d has a table listing all svelte skills
```

## Why

1. **Self-documenting capabilities** — the app's functionality is defined by its skills; documenting them makes that visible to humans
2. **Governance transparency** — you can't govern what you can't see; human docs make agent authority reviewable
3. **Pillar alignment** — Clarity Through Structure (PILLAR-569581e0) requires making thinking visible; Purpose Through Continuity (PILLAR-94b281db) requires intent to survive implementation pressure — undocumented skills erode both
4. **Plugin ecosystem quality** — third-party plugins must document what their skills do before they can be installed

## Migration

Existing skills without docs are flagged as integrity errors. The migration creates documentation artifacts for all existing skills, either as dedicated docs or as reference tables grouped by plugin/domain.
