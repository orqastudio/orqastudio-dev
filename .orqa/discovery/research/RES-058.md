## Key Findings

### Current state is extraction-ready

- **Zero circular dependencies** between stores
- **All stores delegate to backend** via typed `invoke()` calls — no direct data access
- **ArtifactGraphSDK** is already a self-contained singleton with subscription API
- **All 13 type files** are fully portable
- **39 Tauri commands** are invoked across 10 stores, cleanly scoped by domain

### Portability assessment

| Component | Portable | Notes |
|-----------|----------|-------|
| ArtifactGraphSDK | Yes | Zero app-specific logic |
| invoke wrapper | Yes | Pure Tauri IPC abstraction |
| All type files (13) | Yes | No UI dependencies |
| frontmatter.ts parser | Yes | Zero dependencies |
| sessionStore | Yes | Pure session CRUD |
| projectStore | Yes | Pure project state |
| artifactStore | Yes | Delegates to SDK |
| enforcementStore | Yes | Pure state management |
| lessonStore | Yes | Pure state management |
| setupStore | Yes | Backend-driven |
| conversationStore | Mostly | One import from UI component (DEFAULT_MODEL) |
| settingsStore | Mostly | Theme applies CSS class to document — needs injectable hook |
| errorStore | Mostly | Browser event handlers (window.onerror) — make optional |
| navigationStore | No | Tightly coupled to app layout, hardcoded view names |

### IPC command inventory (39 commands)

Cleanly grouped by domain: session (6), conversation (4), project (8), artifacts/graph (11), settings/sidecar (4), enforcement (2), lessons (3), setup (5). Each group maps to one store.

## Architecture Options

### Option A: Single SDK package with everything (recommended)

```
@orqastudio/sdk
├── types/          ← all 13 type files
├── ipc/            ← invoke wrapper, channel, error extraction
├── graph/          ← ArtifactGraphSDK
├── stores/         ← all portable stores
├── utils/          ← frontmatter parser
└── index.ts        ← barrel export
```

**Pros:**
- One package to install, one version to track
- Stores and SDK share types without cross-package imports
- Simpler dependency graph for plugins
- Matches how the code is already structured

**Cons:**
- Larger package — plugins that only need types still pull stores
- Version bumps affect all consumers even for unrelated changes

### Option B: Split types from SDK

```
@orqastudio/types   ← types only (tiny, stable)
@orqastudio/sdk     ← graph, stores, ipc (depends on types)
```

**Pros:**
- Types are stable and rarely change — fewer version bumps for type-only consumers
- Integrity validator depends on types, not the full SDK
- Cleaner dependency tree

**Cons:**
- Two packages to version and publish
- Cross-package type imports add complexity

### Option C: Three-layer split

```
@orqastudio/types   ← types only
@orqastudio/ipc     ← invoke wrapper, channel (Tauri-specific)
@orqastudio/sdk     ← graph, stores (depends on types + ipc)
```

**Pros:**
- IPC layer could theoretically be swapped for non-Tauri backends
- Maximum separation of concerns

**Cons:**
- Over-engineering — the IPC layer is 50 lines of code
- Three packages to maintain for minimal benefit
- No realistic use case for non-Tauri backends yet

## Recommendation

**Option B** — split types from SDK. The integrity validator already needs types but not stores. Plugins that only validate artifacts shouldn't pull the full Svelte runtime. The types package is small, stable, and broadly useful.

The SDK package would `import type` from `@orqastudio/types` and re-export the types for convenience, so consumers can choose:
- `import type { ArtifactNode } from "@orqastudio/types"` — minimal
- `import { artifactGraphSDK, type ArtifactNode } from "@orqastudio/sdk"` — full

## Extraction approach

### What moves to @orqastudio/types
- All 13 files from `ui/src/lib/types/`
- `INVERSE_MAP`, `SINGLE_REF_FIELDS`, `ARRAY_REF_FIELDS` constants

### What moves to @orqastudio/sdk
- `artifact-graph.svelte.ts` → `graph/artifact-graph.svelte.ts`
- `invoke.ts` → `ipc/invoke.ts`
- `frontmatter.ts` → `utils/frontmatter.ts`
- All stores except `navigation.svelte.ts`
- `conversationStore`: extract `DEFAULT_MODEL` to config parameter
- `settingsStore`: make `applyThemeToDocument` an injectable callback
- `errorStore`: make browser handlers opt-in

### What stays in the app
- `navigationStore` — too coupled to app layout
- `model-options.ts` — UI component config
- UI utility functions (colors, formatting)

### What changes for plugins
- Plugins `import { sessionStore, artifactGraphSDK } from "@orqastudio/sdk"`
- Plugins get the same stores the app uses — parity guaranteed
- Plugins can subscribe to graph changes via `artifactGraphSDK.subscribe()`
- Plugins get streaming via `conversationStore` and Tauri channels

## Non-portable elements requiring modification

| Element | Current | Fix |
|---------|---------|-----|
| `DEFAULT_MODEL` in conversationStore | Imported from `model-options.ts` component | Accept as constructor param or config |
| `applyThemeToDocument` in settingsStore | Direct DOM manipulation | Accept as callback in `initialize()` |
| `window.onerror` in errorStore | Browser globals | Wrap in `if (typeof window !== "undefined")` |

All three are trivial changes — under 10 lines each.
