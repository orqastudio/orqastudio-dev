## Research Questions

1. What licensing options protect against big tech absorption while enabling open-source participation?
2. Can the concepts/methodology behind OrqaStudio be protected, or only the code?
3. How have other projects handled this problem?
4. What trademark and branding protection is needed?

---

## Findings

### Current State

OrqaStudio ships with **Apache License 2.0** (copyright 2026 Bobbi Byrne-Graham). This is a permissive license that offers **zero protection** against big tech absorption. Any company can take the code, modify it, and ship a competing closed-source product without contributing back. Neither `Cargo.toml` nor `package.json` declares a license field.

### What Can and Cannot Be Protected by License

| Asset | License Protects? | Alternative |
|-------|:-:|---|
| The code (Rust, Svelte, TypeScript) | Yes | Copyright (automatic) |
| Governance framework concept | No | Patent, trade secret, or execution speed |
| Artifact graph pattern | Partially (implementation, not concept) | Patent |
| Plugin architecture | Partially | Patent |
| Enforcement engine concept | No | Stay ahead through execution |
| Methodology (understand-plan-document-implement-review-learn) | No | Brand recognition, community |
| Name "OrqaStudio" | No (not through code license) | Trademark |

**Key insight**: A competitor could read the code, understand the concepts, and reimplement from scratch under any license. Protection comes from execution speed, community, brand, and (optionally) patents.

### License Options Compared

#### Option A: FSL (Functional Source License) — RECOMMENDED

Created by Sentry. Source-available. Can do anything EXCEPT compete commercially with the licensor. Auto-converts to Apache 2.0 after 2 years. Written by Heather Meeker (top open-source attorney).

- Directly prohibits competitors from taking code and shipping a competing product
- Clearest, simplest anti-competition language of all options
- 2-year conversion shows good faith
- Backed by a consortium (Sentry, CodeCov, Liquibase)
- No "betrayal" narrative if set from the start
- **Not OSI-approved** — "source-available" not "open source"

#### Option B: AGPL v3 + Dual License

True OSI-approved open source. Forces distribution of modifications under AGPL. Offer commercial license for enterprises that can't comply. Classic model (MySQL, Qt, GitLab).

- Strongest community signal (true open source)
- Weaker against big tech (internal use without sharing is allowed)
- Requires Contributor License Agreement for dual licensing
- More complex to manage as a solo developer

#### Option C: BSL 1.1 (Business Source License)

Source-available. Cannot offer competing commercial product. Converts to open source after 4 years. Used by HashiCorp.

- Strongest commercial protection for longest period
- Carries the "HashiCorp stigma" — triggered OpenTofu fork and backlash
- More ambiguous than FSL

#### Not Recommended

- **SSPL** — designed for SaaS/cloud, overkill for a desktop app
- **Elastic License 2.0** — narrowly focused on "managed service" restriction
- **Commons Clause** — legally vague, confusing, widely criticized

### Case Studies

| Company | Changed To | Result |
|---------|-----------|--------|
| HashiCorp | BSL 1.1 | Backlash, OpenTofu fork, acquired by IBM |
| Elastic | SSPL + ELv2 | AWS forked as OpenSearch, later added AGPL back |
| Redis | SSPL + RSALv2 | Valkey fork within weeks |
| Sentry | FSL | Smooth transition, no major fork, community respected it |

**Pattern**: Starting with a transparent license (FSL) avoids the "betrayal" narrative that companies switching FROM permissive licenses suffered.

### Trademark Protection

Regardless of code license, **"OrqaStudio" should be trademarked**. Even if someone forks the code, they cannot use the name or branding. Firefox (Mozilla), WordPress (Foundation), and Linux (Torvalds) all use trademark protection as their primary brand defense.

**Recommended actions:**
1. Register "OrqaStudio" trademark in relevant jurisdictions
2. Create a trademark usage policy
3. Works regardless of which code license is chosen

---

## Recommendation

**FSL + Trademark** is the strongest fit for OrqaStudio's situation:

- Solo developer wanting transparency but protection
- Desktop app (not SaaS, so SSPL/Elastic are poor fits)
- Novel concepts that can't be protected by license alone
- Setting terms from day one (no betrayal narrative)
- 2-year auto-conversion to Apache 2.0 shows good faith

If true OSI "open source" label matters more than competition protection, use **AGPL v3 + dual licensing** instead.
