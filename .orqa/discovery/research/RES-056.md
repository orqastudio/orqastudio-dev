## Problem Statement

OrqaStudio's enforcement system is **half-complete across all layers**. Rules declare enforcement entries as if they're active, but critical event types and actions are not consumed by the plugin or connected to the app's agent execution pipeline. The system that should enforce governance during its own development cannot enforce itself.

## Current State

### What Works

| Layer | Event Types | Actions | Status |
|-------|------------|---------|--------|
| Plugin (rule-engine.mjs) | `file`, `bash` | `block`, `warn`, `inject` (skill names only) | Working |
| Plugin (stop-checklist.sh) | N/A | Static checklist | Working but not rule-driven |
| Pre-commit hook | Schema validation, link verification | Block | Working |
| Rust EnforcementEngine | `file`, `bash`, `scan`, `lint` | Block, warn, inject | Complete but disconnected |
| Rust Process Gates | `write`, `stop` | Thinking prompts | Working but separate from rule engine |

### What Doesn't Work

| Gap | Impact | Declared By |
|-----|--------|-------------|
| `stop` events not evaluated by plugin | RULE-532100d9 permission-seeking check never fires | RULE-532100d9, RULE-98682b5e |
| `prompt` events not evaluated by plugin | No enforcement on user prompt patterns | Not yet declared |
| Skill injection returns names, not content | Agent gets "Read: skill-name" instead of actual skill content | RULE-f9d0279c, RULE-130f1f63 |
| Rust engine not connected to agent pipeline | App-context enforcement doesn't fire during agent execution | EPIC-3a8ad459 Phase 2 |
| Process gates separate from rule engine | Two enforcement systems in Rust that don't compose | N/A |
| Plugin stop hook not rule-driven | stop-checklist.sh is hardcoded, ignores enforcement entries | RULE-532100d9 |

### The Numbers

- **11 rules** have `enforcement:` arrays (of 46 total rules)
- **~40 enforcement entries** declared across those rules
- **~24 file entries** — plugin evaluates these (working)
- **~9 bash entries** — plugin evaluates these (working)
- **3 stop entries** — plugin does NOT evaluate (gap)
- **5 lint entries** — declarative only, intentionally not evaluated (correct)
- **~10 inject actions** — plugin returns skill names but not content (partial)

## Root Cause Analysis

### Circular Dependency

```
The orchestrator needs enforcement to maintain governance
  → Enforcement is declared in rule frontmatter
    → The plugin consumes enforcement entries
      → The plugin only handles file/bash events
        → Stop events are never evaluated
          → The orchestrator violates behavioral rules (IMPL-85add0f1, IMPL-945fa6ab, IMPL-e848fb92)
            → The orchestrator needs enforcement to maintain governance...
```

### Why This Happened

EPIC-3a8ad459 Phase 1 built the plugin with file/bash enforcement first (the most mechanical patterns). Stop events, prompt events, and full skill injection were planned but not prioritized because:

1. File/bash enforcement catches the most violations (coding standards, forbidden patterns)
2. Stop events require different context (no file path, no content — just session state)
3. Skill injection was assumed to work via name-only references (agents would load skills themselves)
4. The app's Rust engine was built in parallel but not yet connected to agent execution

The result: **mechanical enforcement works for coding standards but behavioral enforcement remains self-policed**.

## Architecture Assessment

### Plugin Gaps (rule-engine.mjs)

**Line 199**: Hard filter `if (!["Write", "Edit", "Bash"].includes(toolName))` exits without evaluation. Adding stop event support requires:

1. Stop hook calling rule-engine.mjs (currently calls stop-checklist.sh)
2. rule-engine.mjs accepting stop event context (no file_path, different context shape)
3. Pattern matching against session transcript or turn summary

**Skill injection (lines 156-177)**: Returns `"**Read before implementing:**\n${ids.join("\n")}"` — just names. Full injection requires:

1. Resolving skill name to `.orqa/process/skills/{name}/SKILL.md`
2. Reading and parsing the skill file
3. Returning skill body as systemMessage content
4. Deduplication already works via `.injected-skills.json`

### App Gaps (Rust)

**EnforcementEngine** (`backend/src-tauri/src/domain/enforcement_engine.rs`): Complete implementation with file, bash, scan, lint evaluation. Returns `Verdict` with skills array. BUT:

1. Not connected to agent tool approval pipeline
2. Not connected to sidecar communication
3. Process gates (process_gates.rs) are a separate system with different semantics

**Unification opportunity**: Process gates and enforcement engine should compose. Process gates track workflow state (has_written_code, has_run_verification). Enforcement engine evaluates patterns. A unified system would let enforcement entries reference workflow state.

### PostToolUse Hook (graph-guardian.mjs)

The plugin already has a PostToolUse hook (`graph-guardian.mjs`) that fires after Write/Edit calls. This is where RULE-130f1f63's write-time integrity checking would naturally live. Currently it performs basic graph integrity checks. Extending it to run `verify-links.mjs --files <path>` on the modified file would provide efficient, targeted integrity enforcement.

## Proposed Solution

### Priority Order

1. **Stop event support in plugin** — Connect stop hook to rule-engine.mjs, add stop event evaluation
2. **Full skill content injection** — Read skill files and inject content, not just names
3. **graph-guardian.mjs enhancement** — Add bidirectional relationship checking on PostToolUse
4. **App enforcement pipeline** — Wire Rust EnforcementEngine to agent tool approval
5. **Process gate / enforcement engine unification** — Compose workflow state with pattern matching

### Efficiency Considerations

- Stop event evaluation: Runs once per session end — negligible cost
- Skill content injection: Reads skill files on first match, cached for session — low cost
- PostToolUse graph checking: Runs `verify-links.mjs` on single file — ~100ms per write
- App enforcement pipeline: Rust evaluation is fast — main cost is integration complexity
