## Decision

OrqaStudio implements native code search using three integrated components:

1. **DuckDB** — Stores code chunks and embedding vectors. Provides vector similarity search via `array_cosine_similarity()`. Single-file database per project.
2. **ONNX Runtime** — Runs the `bge-small-en-v1.5` embedding model (33M parameters, 384-dimensional vectors) locally. No external API calls for search.
3. **DirectML** — Hardware acceleration layer. Uses NPU/GPU when available, falls back to CPU. No code changes needed between acceleration paths.

The search engine provides three tools: `search_regex` (exact pattern matching), `search_semantic` (meaning-based search via embeddings), and `code_research` (architectural analysis combining search with LLM reasoning).

## Rationale

External search APIs (Algolia, Elasticsearch) add network latency and require running services. A native engine keeps search fully offline, fast (<100ms for semantic queries), and private. DuckDB was chosen over SQLite for vector operations because it has built-in array functions and columnar storage optimized for analytical queries. ONNX Runtime was chosen over direct PyTorch/TensorFlow because it's lighter weight and supports DirectML for Windows NPU/GPU acceleration without CUDA dependency.

The `bge-small-en-v1.5` model was selected for its balance of quality and size — small enough to bundle with the installer (~67MB) while maintaining strong code understanding.

## Consequences

- The ONNX model must be distributed with the app (bundled in installer for production, auto-downloaded from Hugging Face for development)
- DuckDB adds a native dependency per platform
- Code must be chunked at semantic boundaries (functions, classes, imports) rather than fixed-size windows
- Indexing happens at startup and on file changes — background task with progress tracking in the status bar
- The search engine is dual-context: CLI uses ChunkHound MCP server (external tool, separate implementation), app uses this native embedded Rust engine. Both expose the same three tool names (`search_regex`, `search_semantic`, `code_research`) but are completely independent implementations — ChunkHound is NOT integrated into the app; it is a CLI-only companion
