## Open Questions

### Q1: SQLite Schema Design

**Question:** What tables does OrqaStudio need, and how should they be structured?

**Findings:**

#### Complete Schema Design

Nine core tables plus two FTS5 virtual tables. All tables use `INTEGER PRIMARY KEY` (SQLite's built-in rowid alias) for auto-incrementing IDs. Timestamps are stored as ISO 8601 text (`YYYY-MM-DDTHH:MM:SS.sssZ`) for human readability and SQLite datetime function compatibility.

```sql
-- ============================================================
-- PROJECTS
-- ============================================================
CREATE TABLE projects (
    id              INTEGER PRIMARY KEY,
    name            TEXT NOT NULL,
    path            TEXT NOT NULL UNIQUE,
    description     TEXT,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE UNIQUE INDEX idx_projects_path ON projects(path);

-- ============================================================
-- SESSIONS
-- ============================================================
CREATE TABLE sessions (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    user_id         TEXT,                       -- nullable: for future multi-user support
    title           TEXT,
    model           TEXT NOT NULL DEFAULT 'claude-sonnet-4-20250514',
    system_prompt   TEXT,
    status          TEXT NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active', 'completed', 'abandoned', 'error')),
    summary         TEXT,                       -- Claude-generated on session end
    handoff_notes   TEXT,                       -- for session continuity
    total_input_tokens  INTEGER DEFAULT 0,
    total_output_tokens INTEGER DEFAULT 0,
    total_cost_usd  REAL DEFAULT 0.0,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE INDEX idx_sessions_project ON sessions(project_id);
CREATE INDEX idx_sessions_created ON sessions(created_at);
CREATE INDEX idx_sessions_status ON sessions(status);

-- ============================================================
-- MESSAGES
-- ============================================================
-- Stores all conversation messages including tool_use and tool_result.
-- A single API turn may produce multiple content blocks (text + tool_use).
-- Each content block is a separate row for granular storage and search.
CREATE TABLE messages (
    id              INTEGER PRIMARY KEY,
    session_id      INTEGER NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    -- role: 'user', 'assistant', 'system'
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    -- content_type: 'text', 'tool_use', 'tool_result', 'thinking', 'image'
    content_type    TEXT NOT NULL DEFAULT 'text'
                    CHECK (content_type IN ('text', 'tool_use', 'tool_result', 'thinking', 'image')),
    -- For text: the message text
    -- For tool_use: null (structured data in tool_* columns)
    -- For tool_result: the result text or summary
    -- For thinking: the thinking text
    content         TEXT,
    -- Tool-specific fields (null for non-tool messages)
    tool_call_id    TEXT,           -- Claude's tool_use id (e.g. "toolu_01D7...")
    tool_name       TEXT,           -- e.g. "Read", "Edit", "Bash"
    tool_input      TEXT,           -- JSON: the input object Claude sent
    tool_is_error   INTEGER DEFAULT 0,  -- 1 if tool_result is an error
    -- Grouping: messages in the same API turn share a turn_index
    turn_index      INTEGER NOT NULL DEFAULT 0,
    -- Ordering within a turn (text block 0, tool_use block 1, etc.)
    block_index     INTEGER NOT NULL DEFAULT 0,
    -- Streaming state: 'pending' during stream, 'complete' when done, 'error' on failure
    stream_status   TEXT NOT NULL DEFAULT 'complete'
                    CHECK (stream_status IN ('pending', 'complete', 'error')),
    -- Token usage for this specific API response (assistant messages only)
    input_tokens    INTEGER,
    output_tokens   INTEGER,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE INDEX idx_messages_session ON messages(session_id, turn_index, block_index);
CREATE INDEX idx_messages_tool ON messages(tool_name) WHERE tool_name IS NOT NULL;
CREATE INDEX idx_messages_stream ON messages(stream_status) WHERE stream_status = 'pending';

-- ============================================================
-- ARTIFACTS (governance artifact metadata)
-- ============================================================
-- Content lives on disk as markdown files. This table stores metadata only.
CREATE TABLE artifacts (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    -- type: 'agent', 'rule', 'skill', 'doc'
    artifact_type   TEXT NOT NULL CHECK (artifact_type IN ('agent', 'rule', 'skill', 'doc')),
    -- Relative path from project root (e.g. ".orqa/agents/backend-engineer.md")
    rel_path        TEXT NOT NULL,
    -- Extracted from frontmatter or filename
    name            TEXT NOT NULL,
    description     TEXT,
    -- File state tracking
    file_hash       TEXT,               -- SHA-256 of file content for change detection
    file_size       INTEGER,
    file_modified_at TEXT,              -- OS file modification time
    -- OrqaStudio metadata
    last_scanned_at TEXT,
    compliance_status TEXT DEFAULT 'unknown'
                    CHECK (compliance_status IN ('compliant', 'non_compliant', 'unknown', 'error')),
    -- JSON: extracted relationships (references to other artifacts)
    relationships   TEXT,               -- JSON array: [{"type": "references", "target": "path"}]
    -- JSON: arbitrary metadata extracted during scanning
    metadata        TEXT,               -- JSON object
    last_edited_by  TEXT,               -- nullable: for future multi-user support
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE UNIQUE INDEX idx_artifacts_path ON artifacts(project_id, rel_path);
CREATE INDEX idx_artifacts_type ON artifacts(project_id, artifact_type);

-- ============================================================
-- SCANNER RESULTS
-- ============================================================
CREATE TABLE scanner_results (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    scanner_name    TEXT NOT NULL,       -- e.g. "documentation-coverage", "rule-compliance"
    passed          INTEGER NOT NULL,   -- 0 or 1
    score           REAL,               -- optional numeric score (0.0 - 1.0)
    -- JSON: detailed findings, per-item results
    details         TEXT,               -- JSON object with scanner-specific structure
    -- JSON: list of artifact IDs that failed
    failed_items    TEXT,               -- JSON array of artifact IDs or paths
    duration_ms     INTEGER,            -- how long the scan took
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE INDEX idx_scanner_project ON scanner_results(project_id, scanner_name);
CREATE INDEX idx_scanner_created ON scanner_results(created_at);

-- ============================================================
-- METRICS
-- ============================================================
CREATE TABLE metrics (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    metric_name     TEXT NOT NULL,       -- e.g. "doc_coverage", "rule_compliance", "session_count"
    value           REAL NOT NULL,
    unit            TEXT,                -- e.g. "percent", "count", "seconds"
    -- JSON: dimensional breakdown (optional)
    dimensions      TEXT,               -- JSON: {"agent": "backend-engineer", "category": "rules"}
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE INDEX idx_metrics_project ON metrics(project_id, metric_name);
CREATE INDEX idx_metrics_created ON metrics(created_at);

-- ============================================================
-- TASKS
-- ============================================================
CREATE TABLE tasks (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    session_id      INTEGER REFERENCES sessions(id) ON DELETE SET NULL,
    title           TEXT NOT NULL,
    description     TEXT,
    status          TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'in_progress', 'blocked', 'completed', 'cancelled')),
    priority        INTEGER DEFAULT 0,  -- higher = more urgent
    assigned_agent  TEXT,               -- which agent type should handle this
    -- JSON: task-specific metadata (acceptance criteria, linked artifacts, etc.)
    metadata        TEXT,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    completed_at    TEXT
);

CREATE INDEX idx_tasks_project ON tasks(project_id, status);
CREATE INDEX idx_tasks_session ON tasks(session_id) WHERE session_id IS NOT NULL;

-- ============================================================
-- LESSONS
-- ============================================================
CREATE TABLE lessons (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    session_id      INTEGER REFERENCES sessions(id) ON DELETE SET NULL,
    title           TEXT NOT NULL,
    -- The problem pattern this lesson addresses
    pattern         TEXT NOT NULL,
    -- The fix or correct approach
    fix             TEXT NOT NULL,
    -- Recurrence tracking
    occurrence_count INTEGER NOT NULL DEFAULT 1,
    last_occurred_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    -- Severity/impact
    severity        TEXT DEFAULT 'medium'
                    CHECK (severity IN ('low', 'medium', 'high', 'critical')),
    -- JSON: tags, related artifacts, categories
    metadata        TEXT,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE INDEX idx_lessons_project ON lessons(project_id);
CREATE INDEX idx_lessons_recurrence ON lessons(occurrence_count DESC);

-- ============================================================
-- SETTINGS
-- ============================================================
CREATE TABLE settings (
    key             TEXT NOT NULL,
    value           TEXT NOT NULL,       -- JSON value (string, number, object, array)
    -- scope: 'app' for global settings, or a project_id for project-specific
    scope           TEXT NOT NULL DEFAULT 'app',
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    PRIMARY KEY (key, scope)
);
```

#### Conversation Message Schema Design Rationale

**Why one row per content block (not one row per API message):**

A single Claude API response can contain multiple content blocks: text, then tool_use, then more text, then another tool_use. Storing each block as a separate row enables:

1. **Granular FTS indexing** -- index text content separately from tool results
2. **Efficient streaming updates** -- update a single row's content as tokens arrive, rather than appending to a growing JSON blob
3. **Tool-specific queries** -- "show me all Bash tool calls in this project" is a simple WHERE clause
4. **UI rendering** -- each block maps naturally to a UI component (text bubble, tool call card, thinking block)

The alternative (one row per message, tool calls in a JSON array) would require JSON parsing for every query that touches tool data. The `turn_index` and `block_index` columns preserve the original message ordering within a turn.

**Tool call storage:**

The Claude API produces `tool_use` blocks (assistant asks to use a tool) and `tool_result` blocks (user provides the result). These are stored as separate rows:

- `tool_use` row: `content_type='tool_use'`, `tool_call_id` set, `tool_name` set, `tool_input` contains the JSON input object. `content` is null.
- `tool_result` row: `content_type='tool_result'`, `tool_call_id` matches the tool_use row, `content` contains the result text. `tool_is_error=1` if the tool failed.

This matches the Claude API's own structure (see API reference: tool_use has `id`, `name`, `input`; tool_result has `tool_use_id`, `content`, `is_error`).

#### Streaming / Partial Message Handling

**Problem:** During streaming, tokens arrive one at a time. How do we persist a message that is still being generated?

**Strategy: Insert-then-update with `stream_status`.**

1. When streaming begins, insert a row with `stream_status='pending'` and empty `content`.
2. Buffer tokens in memory (Rust side). Periodically (every ~500ms or ~50 tokens) update the row's `content` column with the accumulated text. This is a simple `UPDATE messages SET content = ? WHERE id = ?`.
3. When the stream completes, set `stream_status='complete'` and write the final content.
4. If the stream errors, set `stream_status='error'` and store whatever was received.
5. On app startup, check for `stream_status='pending'` rows -- these represent interrupted streams. Mark them as `'error'` or delete them.

**Why not write every token to SQLite?** At 50-100 tokens/second, that would be 50-100 writes/second per active stream. SQLite in WAL mode can handle this, but it is wasteful. Batching to ~2 writes/second is sufficient for crash recovery while keeping the UI responsive via in-memory state.

**WAL mode is essential for streaming.** Enable with `PRAGMA journal_mode=WAL;` at connection init. WAL allows concurrent reads (UI queries) while streaming writes happen, with no reader/writer blocking.

#### FTS5 Full-Text Search Setup

**Recommendation: External content FTS5 table with triggers.**

This avoids duplicating message content (FTS5 normally stores its own copy). The FTS index references the messages table and stays in sync via triggers.

```sql
-- ============================================================
-- FTS5: Message Search
-- ============================================================
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,                            -- message text / tool result text
    tool_name,                          -- for filtering by tool
    content='messages',                 -- external content table
    content_rowid='id',
    tokenize='porter unicode61'         -- English stemming + Unicode support
);

-- Triggers to keep FTS in sync with messages table
CREATE TRIGGER messages_ai AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content, tool_name)
    VALUES (new.id, new.content, new.tool_name);
END;

CREATE TRIGGER messages_ad AFTER DELETE ON messages BEGIN
    INSERT INTO messages_fts(messages_fts, rowid, content, tool_name)
    VALUES ('delete', old.id, old.content, old.tool_name);
END;

CREATE TRIGGER messages_au AFTER UPDATE OF content ON messages BEGIN
    INSERT INTO messages_fts(messages_fts, rowid, content, tool_name)
    VALUES ('delete', old.id, old.content, old.tool_name);
    INSERT INTO messages_fts(rowid, content, tool_name)
    VALUES (new.id, new.content, new.tool_name);
END;

-- ============================================================
-- FTS5: Artifact Content Search (indexed from disk files)
-- ============================================================
CREATE VIRTUAL TABLE artifacts_fts USING fts5(
    name,
    content,                            -- file content, indexed for search
    content='',                         -- contentless: no content stored, index only
    contentless_delete=1,               -- allow DELETE (SQLite 3.43.0+)
    tokenize='porter unicode61'
);
```

**FTS5 query examples for OrqaStudio:**

```sql
-- Search messages across all sessions in a project
SELECT m.id, m.content, m.session_id, m.role,
       highlight(messages_fts, 0, '<mark>', '</mark>') as highlighted,
       bm25(messages_fts) as rank
FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid
JOIN sessions s ON s.id = m.session_id
WHERE messages_fts MATCH 'authentication AND error'
  AND s.project_id = ?
ORDER BY rank
LIMIT 50;

-- Search only tool results
SELECT m.id, m.content, m.tool_name
FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid
WHERE messages_fts MATCH 'tool_name : Bash AND content : "permission denied"'
ORDER BY bm25(messages_fts);

-- Search artifacts by content
SELECT a.id, a.name, a.rel_path,
       highlight(artifacts_fts, 1, '<mark>', '</mark>') as highlighted
FROM artifacts_fts
JOIN artifacts a ON a.name = artifacts_fts.name
WHERE artifacts_fts MATCH 'error handling strategy'
ORDER BY bm25(artifacts_fts);
```

**FTS5 performance characteristics:**

| Dataset Size | Index Time | Query Time | Index Size |
|-------------|-----------|-----------|-----------|
| 10K messages | < 1s | < 5ms | ~2 MB |
| 100K messages | ~5s | < 15ms | ~20 MB |
| 1M messages | ~60s | < 50ms | ~200 MB |

These are estimates based on published benchmarks. FTS5 uses an inverted index (term-to-rowid mapping) and the query time scales with the number of matching documents, not total documents. A search returning 100 results from 1M rows is nearly as fast as from 10K rows -- the bottleneck is reading matching rows, not scanning the index.

**Tokenizer choice:** `porter unicode61` is recommended because:
- `unicode61` handles international characters properly
- `porter` adds English stemming (searching "authentication" also matches "authenticate", "authenticating")
- For substring matching (e.g., searching partial identifiers), add a separate trigram index on specific columns if needed

**When to rebuild the FTS index:**

```sql
-- After bulk operations (import, migration)
INSERT INTO messages_fts(messages_fts) VALUES('rebuild');
-- Optimize after many updates (merges internal b-trees)
INSERT INTO messages_fts(messages_fts) VALUES('optimize');
```

#### JSON Columns vs Normalized Tables

**Recommendation: Use JSON columns for truly flexible/schemaless data. Use normalized columns for anything you query on.**

| Data | Store As | Rationale |
|------|---------|-----------|
| Tool input (`tool_input`) | JSON column | Structure varies per tool, never queried by sub-fields |
| Scanner details | JSON column | Scanner-specific structure, queried as a whole |
| Artifact relationships | JSON column | Variable structure, queried infrequently |
| Metric dimensions | JSON column | Flexible dimensional breakdown |
| Task metadata | JSON column | Acceptance criteria, links -- varies per task |
| Tool name, tool_call_id | Normalized columns | Queried frequently, indexed |
| Message role, content_type | Normalized columns | Filtered constantly |
| Scanner pass/fail, score | Normalized columns | Aggregated, charted |
| Artifact type, compliance | Normalized columns | Filtered, grouped |

**SQLite JSON best practices:**

```sql
-- Use json_extract() for indexed lookups on JSON columns
CREATE INDEX idx_artifact_meta_category
    ON artifacts(json_extract(metadata, '$.category'));

-- Use json_each() for querying JSON arrays
SELECT a.name FROM artifacts a, json_each(a.relationships) r
WHERE json_extract(r.value, '$.type') = 'references';

-- JSONB (SQLite 3.45.0+, Jan 2024) stores as binary for 3x faster reads
-- Use jsonb() when inserting, json() when reading
INSERT INTO artifacts (metadata) VALUES (jsonb('{"category": "core"}'));
```

**Caution:** SQLite's JSONB is available in SQLite 3.45.0+. Verify the bundled SQLite version in your `rusqlite` or `sqlx` dependency. As of early 2026, `rusqlite` bundles SQLite 3.46+ and `sqlx` depends on the system SQLite or bundled version.

**Migration file organization:**

```
backend/src-tauri/
  migrations/
    001_core_tables.sql      -- projects, sessions, messages, settings
    002_fts_indexes.sql      -- FTS5 virtual tables and triggers
    003_governance.sql       -- artifacts, scanner_results
    004_tracking.sql         -- metrics, tasks, lessons
```

**Why `tauri-plugin-sql` migrations over `sqlx` migrations:**

- Plugin migrations are simpler (version number + SQL string + direction)
- Applied automatically at app startup, no separate migration binary
- Sufficient for a desktop app where you control the deployment
- `sqlx` migrations require a development database connection at compile time for its `query!()` macro -- unnecessary overhead for a desktop app with a local SQLite file

**However:** If you need `sqlx` for compile-time query checking or complex joins in Rust, you can use both: `tauri-plugin-sql` for migrations and the frontend JS API, plus `sqlx` in Rust commands for complex backend queries. They share the same SQLite file.

**Pragmas to set at connection init:**

```sql
PRAGMA journal_mode=WAL;           -- concurrent reads during writes
PRAGMA synchronous=NORMAL;         -- good balance of safety and speed
PRAGMA foreign_keys=ON;            -- enforce FK constraints
PRAGMA busy_timeout=5000;          -- wait 5s on lock instead of failing
PRAGMA cache_size=-64000;          -- 64MB page cache (negative = KB)
PRAGMA mmap_size=268435456;        -- 256MB memory-mapped I/O
```

**Decision:** Use the schema above with `tauri-plugin-sql` migrations. One row per content block for messages. External content FTS5 with triggers. JSON columns for flexible data, normalized columns for anything queried or filtered.

### Q2: File vs DB Boundary

**Question:** Where does the boundary lie between file-based and database-stored data?

**Findings:**

#### What Metadata Does OrqaStudio Need?

Governance artifacts (`.orqa/agents/*.md`, `.orqa/rules/*.md`, `.orqa/skills/**`, `docs/**`) are markdown files committed to git. OrqaStudio needs the following metadata that cannot be derived from the file alone:

| Metadata | Source | Why DB |
|----------|--------|--------|
| Last scanned timestamp | OrqaStudio scanner | Tracks when OrqaStudio last analyzed this artifact |
| Compliance status | OrqaStudio scanner | Pass/fail/unknown from governance scanners |
| File hash (SHA-256) | Computed from content | Change detection without re-reading full content |
| Relationship graph | Parsed from content | Which artifacts reference which others |
| Edit history | OrqaStudio UI | When edits were made through OrqaStudio (not from git log) |
| Scanner findings | OrqaStudio scanner | Detailed issues found during scanning |
| Classification tags | User/auto | Categories beyond the file's directory location |
| Content index | FTS5 | Full-text search over artifact content |

#### Recommendation: Option D -- Hybrid with Indexing

**Metadata in DB, file content indexed for search (FTS5), content always read from disk for display. File watcher triggers re-index.**

This is the same pattern used by Obsidian and VS Code:

**How Obsidian does it:**
- Files are plain markdown on disk (the source of truth)
- A `MetadataCache` (stored in IndexedDB) pre-parses all files and maintains an indexed cache of structural elements (frontmatter, headings, links, tags)
- File watcher detects changes and triggers incremental cache updates (parse-once-query-many)
- Plugins query the cache for fast lookups, but read files directly for display
- On first launch or after external changes, the cache is rebuilt from disk

**How VS Code does it:**
- Files are always on disk (source of truth)
- An in-memory file index supports Go To File, symbol search, etc.
- File watcher (using the `chokidar` library in Node.js, equivalent to Rust's `notify`) detects changes
- Language services parse files and cache symbol tables
- Search uses `ripgrep` directly on disk (no DB-based search)

**How Cursor does it (closest analog to OrqaStudio):**
- Conversation data stored in `state.vscdb` (SQLite) as JSON in a key-value `ItemTable`
- File-backed content remains on disk
- Conversations are local-only, not synced to servers

**OrqaStudio's hybrid approach:**

```
                    Source of Truth         OrqaStudio Metadata         Search Index
                    ================       ================       ==============
Artifacts           Disk (.claude/)   -->  SQLite (artifacts)     FTS5 (artifacts_fts)
                         ^                      |                      |
                         |                      |                      |
                    File Watcher  -----------> Re-scan  -----------> Re-index
                    (notify crate)
```

1. **Content is always read from disk for display.** Never cache full file content in SQLite.
2. **Metadata is stored in SQLite.** File hash, scan results, compliance status, relationships.
3. **Content is indexed in FTS5 for search.** Contentless FTS5 table (stores index only, not content).
4. **File watcher triggers re-scan and re-index** when files change on disk.

#### File Watcher Architecture

**The `notify` crate** (used by `tauri-plugin-fs`) provides cross-platform file watching. Key considerations for OrqaStudio:

**Debouncing:** Use `notify-debouncer-full` (not `notify-debouncer-mini`) for OrqaStudio because it:
- Merges rapid successive events into one (editors often trigger multiple events per save)
- Tracks file renames (important for git operations)
- Emits events with a configurable delay (recommended: 500ms for OrqaStudio)

**Editor temporary files:** Many editors save files by writing a temp file then renaming it. This produces Create + Rename events instead of a single Modify. The debouncer handles this correctly by tracking the final path.

**Git operations:** When `git checkout`, `git merge`, or `git rebase` touches many files simultaneously, the watcher will emit a burst of events. Strategy:
1. The 500ms debounce window collapses rapid events
2. After debounce, batch all changed paths and re-scan them in a single pass
3. Use the file hash (`artifacts.file_hash`) to skip files that have not actually changed

**Platform-specific considerations:**

| Platform | Backend | Caveats |
|----------|---------|---------|
| Linux | inotify | Subject to `max_user_watches` limit (default 8192). Increase or use PollWatcher for large projects. |
| macOS | FSEvents | May miss events on unowned files. Generally reliable for user-owned project directories. |
| Windows | ReadDirectoryChangesW | Reliable. May emit duplicate events for some editors -- debouncer handles this. |
| Network/WSL | None reliable | Use `PollWatcher` with 2-5s interval as fallback. |

**Implementation sketch (Rust side):**

```rust
use notify_debouncer_full::{new_debouncer, DebouncedEvent};
use std::time::Duration;

fn start_artifact_watcher(project_path: &Path, app_handle: AppHandle) {
    let (tx, rx) = std::sync::mpsc::channel();
    let mut debouncer = new_debouncer(Duration::from_millis(500), None, tx).unwrap();

    // Watch .claude/ directory and docs/ directory
    debouncer.watcher().watch(
        &project_path.join(".claude"),
        RecursiveMode::Recursive,
    ).unwrap();
    debouncer.watcher().watch(
        &project_path.join("docs"),
        RecursiveMode::Recursive,
    ).unwrap();

    // Spawn handler thread
    std::thread::spawn(move || {
        for result in rx {
            match result {
                Ok(events) => {
                    let changed_paths: Vec<_> = events.iter()
                        .filter_map(|e| e.paths.first())
                        .filter(|p| p.extension().map_or(false, |ext| ext == "md"))
                        .collect();
                    if !changed_paths.is_empty() {
                        // Batch re-scan changed artifacts
                        rescan_artifacts(&changed_paths, &app_handle);
                        // Notify frontend
                        app_handle.emit("artifacts-changed", &changed_paths).ok();
                    }
                }
                Err(errors) => {
                    log::warn!("File watcher errors: {:?}", errors);
                }
            }
        }
    });
}
```

**Keeping metadata fresh -- the re-scan flow:**

1. File watcher detects change to `.orqa/rules/no-stubs.md`
2. Compute SHA-256 of the file
3. Compare with `artifacts.file_hash` in DB
4. If hash differs:
   a. Update `file_hash`, `file_size`, `file_modified_at` in `artifacts` table
   b. Re-parse file (extract frontmatter, headings, references to other artifacts)
   c. Update `relationships`, `metadata` JSON columns
   d. Re-index content in `artifacts_fts` (delete old index entry, insert new one)
   e. Update `updated_at` timestamp
5. If hash matches: skip (file was touched but content unchanged, common during git operations)

**Why not Option B (DB cache + file watch)?**

Caching full file content in SQLite creates a dual source of truth. When the user edits a file in VS Code or the terminal, the DB cache is stale until the watcher fires. If the watcher misses an event (network filesystem, rapid edits), the UI shows stale content. Reading from disk is fast enough for individual files (< 1ms for a typical markdown file), so the cache buys almost nothing while adding significant complexity.

**Why not Option A (disk-only)?**

No metadata storage means no scan history, no compliance tracking, no relationship graph, and no full-text search. These are core OrqaStudio features.

**Why not Option C (metadata-only DB, no FTS)?**

Searching artifact content is a key feature. Without FTS5 indexing, every search would require reading and scanning every file on disk -- impractical for projects with hundreds of artifacts.

**Decision:** Option D (hybrid with indexing). Metadata + FTS index in SQLite, content always read from disk. File watcher via `notify-debouncer-full` with 500ms debounce triggers re-scan and re-index.

### Q3: Session Persistence Model

**Question:** How much conversation history should OrqaStudio persist, and how?

**Findings:**

#### Storage Size Estimates

**Typical Claude coding session (1 hour):**

Based on Claude Code transcript data and API token counts:

| Component | Quantity per Hour | Avg Size Each | Total |
|-----------|------------------|---------------|-------|
| User messages (prompts) | 20-40 | ~500 bytes | ~15 KB |
| Assistant text responses | 20-40 | ~2 KB | ~60 KB |
| Tool_use blocks | 30-80 | ~500 bytes (input JSON) | ~25 KB |
| Tool_result blocks (small: grep, glob) | 20-50 | ~2 KB | ~60 KB |
| Tool_result blocks (large: file reads) | 5-15 | ~10 KB | ~100 KB |
| Tool_result blocks (very large: bash output) | 2-5 | ~50 KB | ~175 KB |
| Thinking blocks (extended thinking) | 5-10 | ~3 KB | ~25 KB |
| **Total per 1-hour session** | | | **~460 KB** |

**Session size distribution (estimated):**

| Session Type | Duration | Message Count | DB Size |
|-------------|----------|---------------|---------|
| Quick question | 5 min | 5-10 | 10-50 KB |
| Feature implementation | 30-60 min | 50-150 | 200 KB - 1 MB |
| Extended coding session | 2-4 hours | 200-500 | 1-5 MB |
| Marathon session with heavy tool use | 8+ hours | 500+ | 5-20 MB |

**Accumulated storage (per project):**

| Usage Pattern | Sessions/Week | Annual DB Size |
|--------------|---------------|----------------|
| Light (solo dev, occasional) | 5-10 | 50-200 MB |
| Moderate (daily use, single project) | 20-30 | 200 MB - 1 GB |
| Heavy (multiple sessions/day, large tool results) | 50+ | 1-5 GB |

A 1-5 GB SQLite database is well within SQLite's comfort zone (tested to 281 TB). Disk space on modern machines is abundant. **There is no storage reason to avoid full history.**

#### Recommendation: Full History with Smart Tool Result Storage

**Store every message and tool result in full.** Apply size-aware strategies only for very large tool results.

**Rationale:**
- Disk is cheap. A year of heavy use produces < 5 GB.
- Full history enables powerful cross-session search ("when did I last change the auth module?")
- Summarization loses information. You cannot un-summarize.
- Users expect to scroll back through conversation history.
- The FTS5 index enables fast search regardless of total history size.

**Tool result storage strategy (tiered by size):**

```
Tool result size        Strategy
< 10 KB                 Store in full (text messages, small grep results)
10 KB - 100 KB          Store in full (file contents, moderate bash output)
100 KB - 1 MB           Store in full but compress with zlib (large file dumps)
> 1 MB                  Truncate to first 100 KB + store byte count + "truncated" flag
```

Implementation in the messages table:

```sql
-- Add to messages table for large results
ALTER TABLE messages ADD COLUMN content_truncated INTEGER DEFAULT 0;
ALTER TABLE messages ADD COLUMN content_original_size INTEGER;
```

**Why not tiered storage (recent full, older summarized)?**

- Summarization requires calling Claude (costs money, takes time)
- Automated summarization may lose critical details ("the error was on line 47" gets lost)
- The storage savings are minimal compared to the complexity
- If storage ever becomes an issue, implement a "session archive" feature that exports old sessions to a separate SQLite file (detach, re-attach on demand)

**Why not reference-only tool results (re-executable)?**

- Files change. Re-reading `src/auth.rs` today gives different content than when the conversation happened.
- Commands are not always idempotent. Re-running `git log` gives different results.
- The conversation makes sense only with the original tool results in context.

#### Claude's Context Window and OrqaStudio's Role

**Claude's context window management:**

Claude (Sonnet 4, Opus 4) has a 200K token context window. Claude Code auto-compacts when the context reaches ~155-160K tokens (reserving ~40-45K for the response buffer). Auto-compaction summarizes older conversation turns to free up context space.

**OrqaStudio's role:**

OrqaStudio manages its own context independently from Claude's context window. Key distinction:

- **Claude's context** = what Claude sees in a single API call (up to 200K tokens). Managed by OrqaStudio's conversation engine -- it decides what to include in each API request.
- **OrqaStudio's history** = everything ever said, stored in SQLite. Complete and permanent.

OrqaStudio should implement its own context management strategy:

```
When sending a message to Claude:
1. Always include: system prompt + tool definitions (~5-10K tokens)
2. Always include: last N messages (configurable, default ~20 turns)
3. If context budget remains: include session summary (if continuing from earlier)
4. If context budget remains: include relevant earlier messages (selected by similarity or recency)
5. Track cumulative token count. When approaching 150K tokens, summarize older turns.
```

**Important:** OrqaStudio must track token counts per message (stored in `messages.input_tokens` and `messages.output_tokens`) to know when it is approaching context limits. The Claude API returns usage in every response.

#### Cross-Session Search

**FTS5 performance at scale:**

| Messages | FTS5 Index Size | Typical Query Time | Notes |
|----------|-----------------|-------------------|-------|
| 10K | ~2 MB | < 5ms | Instant. A few weeks of use. |
| 100K | ~20 MB | < 15ms | Months of heavy use. No issues. |
| 1M | ~200 MB | < 50ms | Years of use. Still fast. |

FTS5 query time scales with the number of **matching documents**, not total documents. A query that matches 50 rows out of 1M is nearly as fast as 50 out of 10K.

**Search features to build on FTS5:**

```sql
-- Cross-session search with project filtering
SELECT m.id, m.content, s.title as session_title, s.created_at as session_date,
       snippet(messages_fts, 0, '<mark>', '</mark>', '...', 30) as excerpt,
       bm25(messages_fts) as relevance
FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid
JOIN sessions s ON s.id = m.session_id
WHERE messages_fts MATCH ?
  AND s.project_id = ?
ORDER BY relevance
LIMIT 50;

-- Search with date range
SELECT ... FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid
WHERE messages_fts MATCH 'authentication bug'
  AND m.created_at > '2026-01-01'
ORDER BY bm25(messages_fts);

-- Search tool results only
SELECT ... FROM messages_fts
JOIN messages m ON m.id = messages_fts.rowid
WHERE messages_fts MATCH 'tool_name : Bash AND content : "error"';
```

#### Session Continuity: Handoff Notes

**How should a new session pick up where the last one left off?**

**Strategy: Auto-generated handoff notes stored on session end.**

When a session ends (user closes it, starts a new one, or the app closes):

1. OrqaStudio generates a handoff summary by calling Claude with a special prompt:
   ```
   Summarize this conversation for the next session. Include:
   - What was accomplished
   - What was in progress but unfinished
   - Key decisions made
   - Open questions or blockers
   - Files that were being modified
   Keep it under 500 words.
   ```
2. The summary is stored in `sessions.summary` and `sessions.handoff_notes`.
3. When a new session starts for the same project, the handoff notes from the last session are included in the system prompt.

**Cost of handoff generation:** One additional API call at session end (~500-1000 input tokens for the summary prompt + recent messages, ~200-400 output tokens for the summary). At Sonnet pricing: < $0.01 per session end. Worthwhile.

**Alternative (cheaper): Rule-based summary.** Extract from the conversation without Claude:
- Last user message and assistant response
- List of files modified (from tool_use blocks with Write/Edit tools)
- List of tool calls made
- Any task status changes

This is free but less intelligent. **Recommendation: Claude-generated handoff summaries from Phase 1.** Context loss between sessions is the highest-impact pain point for all three personas. The cost is negligible (< $0.01/session via the sidecar) and the value is immediate. Rule-based summaries can serve as a fallback when the sidecar is unavailable.

**Decision:** Full history storage for all messages and tool results. Size-based truncation only for results > 1 MB. FTS5 for cross-session search. Claude-generated handoff summaries on session end from Phase 1 (rule-based fallback when sidecar unavailable). OrqaStudio manages its own context window independently from Claude.

## Additional Research: Multi-User Considerations

### Future Direction: Collaborative Access

OrqaStudio is initially a single-user desktop application, but the architecture should not preclude multi-user collaborative access in the future (e.g., a small team sharing a OrqaStudio instance where a PM, Tech Lead, and Developer each have visibility into the project's sessions and governance).

**Schema implications (build for single-user, don't preclude multi-user):**

- **Sessions table:** Include a nullable `user_id TEXT` column from Phase 1. Default to `NULL` (single-user mode). When multi-user is implemented, this becomes the session author.
- **Messages table:** No per-message user_id needed — the session's user_id identifies the human participant. AI messages belong to the session.
- **Artifacts table:** Include a nullable `last_edited_by TEXT` column. Tracks who last modified a governance artifact through OrqaStudio.
- **Settings table:** The existing `scope` column (`'app'` or project_id) can be extended to include user-specific scopes (`'user:<id>'`).

**What multi-user would require (not Phase 1):**

- User identity and authentication (local accounts or SSO)
- A `users` table with profile, role (PM, Tech Lead, Developer), and permissions
- Session visibility controls (own sessions, team sessions, all sessions)
- Artifact edit locking or conflict resolution
- Shared vs. personal settings
- Possibly a server component (SQLite is single-writer; concurrent multi-user access from separate machines would need a server or a shared database like PostgreSQL)

**What we do now:** Add nullable `user_id` and `last_edited_by` columns to avoid a migration-heavy schema change later. Leave them NULL in Phase 1. No user management, no authentication, no access controls.

## Additional Research: ORM / Query Layer

### Recommendation: `rusqlite` with `tauri-plugin-sql` for Frontend Queries

**Evaluated options:**

| Library | Type | Async | Compile-time checks | SQLite fit | Notes |
|---------|------|-------|-------------------|-----------|-------|
| `tauri-plugin-sql` (uses sqlx) | Plugin | Yes | No (from JS) | Good | Frontend JS API for simple queries |
| `rusqlite` | Wrapper | No (sync) | No | Excellent | Best SQLite wrapper in Rust ecosystem. Bundles SQLite. |
| `sqlx` | Query builder | Yes | Yes (`query!()` macro) | Good | Requires dev DB at compile time for macro |
| `diesel` | ORM | No | Yes (schema.rs) | Good | Heaviest abstraction, steepest learning curve |
| `sea-orm` | ORM | Yes | Partial | Good | Async-first, now has rusqlite backend |

**Recommendation for OrqaStudio: Use both `tauri-plugin-sql` (frontend) and `rusqlite` (backend).**

**Why:**

1. **`tauri-plugin-sql`** handles migrations and provides a simple JS/TS API for the frontend. Simple CRUD operations (list sessions, get settings, insert messages) can go directly through the plugin from Svelte components via `db.select()` and `db.execute()`. No need to write Rust commands for every simple query.

2. **`rusqlite`** handles complex backend operations in Rust. The streaming pipeline, FTS5 queries, batch artifact re-scanning, and context management logic all live in Rust commands. `rusqlite` is the most ergonomic SQLite library in Rust -- it bundles SQLite (no system dependency), supports all SQLite extensions (FTS5, JSON), and has zero abstraction overhead.

3. **Why not `sqlx`?** The `query!()` macro requires a running SQLite database at compile time. For a desktop app where the database does not exist until first launch, this creates an awkward development workflow (maintain a "dev" database file, keep it in sync with migrations). `rusqlite` avoids this entirely -- queries are runtime-checked.

4. **Why not `diesel` or `sea-orm`?** OrqaStudio's queries are mostly straightforward (CRUD + FTS5 + JSON functions). A full ORM adds learning curve and abstraction for minimal benefit. SQLite-specific features (FTS5 `MATCH`, `bm25()`, `json_extract()`) are easier to express in raw SQL than through an ORM's query builder.

**Shared database access pattern:**

```rust
// In Rust commands, access the SQLite connection directly
use rusqlite::Connection;
use std::sync::Mutex;

struct AppState {
    db: Mutex<Connection>,
}

#[tauri::command]
fn search_messages(
    state: tauri::State<'_, AppState>,
    project_id: i64,
    query: String,
) -> Result<Vec<SearchResult>, String> {
    let db = state.db.lock().map_err(|e| e.to_string())?;
    let mut stmt = db.prepare(
        "SELECT m.id, m.content, s.title,
                snippet(messages_fts, 0, '<mark>', '</mark>', '...', 30) as excerpt
         FROM messages_fts
         JOIN messages m ON m.id = messages_fts.rowid
         JOIN sessions s ON s.id = m.session_id
         WHERE messages_fts MATCH ?1
           AND s.project_id = ?2
         ORDER BY bm25(messages_fts)
         LIMIT 50"
    ).map_err(|e| e.to_string())?;

    let results = stmt.query_map(rusqlite::params![query, project_id], |row| {
        Ok(SearchResult {
            id: row.get(0)?,
            content: row.get(1)?,
            session_title: row.get(2)?,
            excerpt: row.get(3)?,
        })
    }).map_err(|e| e.to_string())?
      .filter_map(|r| r.ok())
      .collect();

    Ok(results)
}
```

**Can `tauri-plugin-sql` and `rusqlite` share the same file?**

Yes, but with care. Both open the SQLite file independently. SQLite's WAL mode supports multiple concurrent connections. However:
- Set `PRAGMA busy_timeout=5000` on both connections
- Set `PRAGMA journal_mode=WAL` on both connections
- The plugin connection handles frontend queries and migrations
- The rusqlite connection handles backend operations

Alternatively, use a single `rusqlite` connection managed in Rust state, and route all frontend queries through Rust `invoke()` commands. This is simpler but requires writing a Rust command for every query type. **Start with the dual-connection approach; refactor to single-connection if contention issues arise.**

### Connection Pooling

**Not needed for a desktop app.** Connection pooling (r2d2, deadpool) solves the problem of many concurrent users competing for database connections. OrqaStudio has exactly one user. A single connection (or two: one for the plugin, one for Rust) is sufficient.

The only scenario where pooling might help: if multiple Tauri commands run concurrently and all need DB access, a `Mutex<Connection>` serializes them. But SQLite itself serializes writes anyway (one writer at a time), so a pool of 2-4 connections provides marginal benefit at best. **Do not add a connection pool unless profiling shows `Mutex` contention is a bottleneck.**

## Summary of Decisions

| Question | Decision | Rationale |
|----------|----------|-----------|
| **Q1: Schema** | 9 tables + 2 FTS5. One row per content block. External content FTS5 with triggers. | Granular storage, efficient search, natural UI mapping |
| **Q1: JSON** | JSON for flexible data (tool_input, metadata, details). Normalized columns for queried fields. | Query performance where needed, flexibility where needed |
| **Q1: Streaming** | Insert pending row, batch-update content ~2/sec, finalize on complete | Balance crash recovery with write efficiency |
| **Q1: FTS5** | Porter + Unicode61 tokenizer, external content table, sync triggers | English stemming, no content duplication, auto-sync |
| **Q1: Migrations** | `tauri-plugin-sql` migrations with `include_str!()` SQL files | Simple, automatic, sufficient for desktop app |
| **Q2: File/DB** | Hybrid (Option D): metadata + FTS index in DB, content from disk | Obsidian-like pattern. Fresh content, rich metadata, searchable |
| **Q2: Watcher** | `notify-debouncer-full` with 500ms debounce, hash-based change detection | Handles editor temp files, git batch ops, rapid saves |
| **Q3: History** | Full history, truncate only > 1 MB tool results | Disk is cheap, full history enables search, summarization loses info |
| **Q3: Search** | FTS5 over all messages, performs well to 1M+ rows | < 50ms queries at 1M rows, scales with matches not total rows |
| **Q3: Continuity** | Claude-generated handoff summaries from Phase 1 (rule-based fallback) | Context loss is the #1 pain point; cost is negligible via sidecar |
| **Q3: Context** | OrqaStudio manages its own context window, independent of Claude's | Full history in DB, selective context in API calls |
| **ORM** | `rusqlite` (Rust) + `tauri-plugin-sql` (frontend JS) | Best SQLite fit, no compile-time DB requirement, shared file |
| **Pooling** | Not needed | Single user, 1-2 connections sufficient |
