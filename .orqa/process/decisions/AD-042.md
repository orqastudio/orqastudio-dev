## Decision

OrqaStudio's governance layer is restructured around a **knowledge maturity pipeline**. Every piece of governance knowledge has a position in a six-stage lifecycle:

```
Observation → Understanding → Principle → Practice → Enforcement → Verification
```

Artifacts at each stage carry **mandatory relationships** to their upstream and downstream neighbors, enforced by schema validation. Threads are not explicit artifacts — they emerge from traversing these relationships, the same way a file path emerges from traversing directory links.

### The Core Question

OrqaStudio is structured therapy for problem-solving. Where therapy asks you to examine situations from different perspectives to ground emotions, OrqaStudio's learning loop constantly checks against the overarching question: **"What is it you are actually trying to do here?"**

The pipeline makes this question structural. When enforcement exists without a principle, the system asks: "What are you actually enforcing and why?" When observations accumulate without becoming principles, the system asks: "What did you actually learn from this?" When the pipeline runs backwards — constraints created without learning feeding them — the system surfaces: "You're generating rules without understanding. Step back."

This is agile, systems-based thinking and retrospective learning, codified and automated by AI.

### The Six Stages

| Stage | Purpose | Artifact Type |
|-------|---------|--------------|
| **Observation** | "What happened?" — raw capture of an event | Lesson (observation) |
| **Understanding** | "Why did it happen?" — root cause and context | Lesson (analyzed) |
| **Principle** | "What's the general rule?" — abstract, transferable | Decision |
| **Practice** | "How do you follow the rule?" — concrete guidance | Skill |
| **Enforcement** | "How does the system prevent violation?" — automated | Rule enforcement entry |
| **Verification** | "Is it working? What's next?" — evidence, reflection, evolution | Project-configured (see below) |

### Configurable Edges Array

Every artifact carries a uniform `relationships` array. Each entry defines the type of relationship, what it connects to, and why:

```yaml
relationships:
  - type: grounded
    target: AD-774cc3d0
    rationale: "Enforces component purity — display components must not fetch data"
  - type: observes
    target: IMPL-e53df28b
    rationale: "Discovered when invoke() was found in a display component"
```

**Relationship types (bidirectional):**

Every relationship type has an outgoing and incoming variant, making the graph fully traversable in both directions.

| Outgoing | Incoming | Pipeline Stage | Meaning |
|----------|----------|----------------|---------|
| `observes` | `observed-by` | Observation | "What was noticed that led to this" / "What emerged from this observation" |
| `grounded` | `grounded-by` | Principle | "What principle this is grounded in" / "What this principle grounds" |
| `grounded-by` | `grounded` | Practice | "How to follow this in practice" / "What this practice serves" |
| `enforces` | `enforced-by` | Enforcement | "What prevents violation of this" / "What this enforcement protects" |
| `enforces` | `enforced-by` | Verification | "What evidence shows this works" / "What this verification measures" |
| `informs` | `informed-by` | Cross-cutting | "What knowledge shaped this" / "What this knowledge shaped" |

**Required relationships per artifact type:**

| Artifact | Required Relationship Types |
|----------|---------------------------|
| Lesson (maturity: understanding) | `grounded` (what principle did we learn?) |
| Decision | `grounded-by`, `enforces` (nullable with rationale when legitimately absent) |
| Rule | `grounded` (what principle does this enforce?) |
| Skill (category: domain/tool) | `grounded` → Decision |
| Skill (category: methodology) | `grounded` → Pillar |

When `target` is null, an `intended` field is required:

```yaml
relationships:
  - type: enforces
    target: null
    rationale: "Enforced by build system (cargo), not governance rules"
    intended: true
```

- `intended: false` (or absent) → surfaces on the dashboard as an **unresolved tension** that needs human review
- `intended: true` → human has reviewed and confirmed this is intended — visible but not flagged

During AI-assisted backfill, the AI proposes connections. Where it can't find one, it sets `target: null, intended: false` with a rationale. The dashboard shows all unresolved tensions. The human works through them: some get resolved (a connection found or created), some get intended (genuinely no connection needed). The system never has silent gaps.

Schema validation enforces required relationship types per artifact type, but the `relationships` array structure is uniform across all types. This means:
- **One field pattern** across all schemas, not different fields per type
- **Extensible** — new relationship types don't require schema changes
- **Co-located rationale** — the "why" lives next to the connection
- **Configurable** — valid relationship types can be defined per project
- **Multiple relationships of same type** — a rule can connect to two principles naturally

The `rationale` field is always required. You cannot create an edge without explaining why it exists. This is the purposeful friction — the AI helps you answer the question, but the question must be answered.

### Emergent Threads

A thread is what you see when you walk the graph:

```
IMPL-e53df28b →(principle)→ AD-774cc3d0 →(practice)→ component-extraction
                               →(enforcement)→ RULE-b49142be
```

No thread artifact exists. The pipeline view in the app traverses relationships and renders the flow. The system surfaces:
- **Disconnected nodes** — enforcement with no principle, principle with no practice
- **Stuck observations** — lessons that haven't matured into principles
- **Failing enforcement** — rules that keep firing (practice isn't working)
- **Backwards pipeline** — enforcement created without upstream learning

### Self-Enforcing Property

The pipeline is self-enforcing through **structural visibility**, not through more hooks or regex. When the system can show "you've created 12 new enforcement rules this week with zero observations feeding them," that signal prevents drift — the same drift that led to this decision.

### Knowledge That Doesn't Come From Mistakes

Not all principles originate from observations. Deliberate design choices (e.g., "use Tauri," "IPC boundary") enter the pipeline at the Principle stage directly. These threads have no Observation/Understanding stages — they start at Principle. The pipeline accommodates this: a thread's first stage can be any stage. What matters is that downstream stages exist (practice, enforcement, verification).

### Verification Is Project-Configured, Not a Fixed Artifact

Verification is shaped by the nature of the project. This is determined at project setup.

**The fundamental question: "Can this project be completed, or is it an ongoing concern?"**

| Project Nature | Verification Focus | OrqaStudio's Role |
|---------------|-------------------|-------------------|
| **Completable** | All questions answered, all work done | Guide the user through what needs answering: what can be automated, what can be researched by AI, what needs human input for knowledge or verification |
| **Ongoing** | Things keep working, things keep evolving | Plugin architecture for integration, configurable check-in cadence, gates for human acknowledgement |

Ongoing projects already handle completable phases through milestones — no new concept needed.

**Within ongoing concerns, three verification patterns:**

| Pattern | What it looks like | OrqaStudio's mechanism |
|---------|-------------------|----------------------|
| **Automated learning** | Quantifiable metrics exist — test coverage, error rates, deployment frequency | Plugin integration to ingest data from external tools. Gates at appropriate places for human verification and acknowledgement. The learning is automated; the human confirms the learning is meaningful. |
| **Monitoring** | Things need periodic attention — "is this architectural decision still right?" | Configurable cadence for AI to surface reflection questions. Plugin integration to set parameters on data that requires urgent notification. The AI asks "is this still working?" on a schedule the user defines. |
| **Evolution** | The project needs to grow — "what should change next?" | AI instruction for knowing when to ask "what next?" Not just compliance checking but direction-setting. Surfacing when the pipeline shows saturation (all threads verified, no new observations) or stagnation (observations accumulating, no new principles). |

**Two verification paths:**

| Path | Trigger | What happens | Creates |
|------|---------|-------------|---------|
| **Automated** | Cadence fires, plugin data source is configured | System requests snapshot from plugin, compares against parameters | VER-NNN record (snapshot + parameters + pass/fail). If fail → also creates TASK for human attention. |
| **Human** | Reflect cadence fires, or automated verification fails | AI surfaces the prompt, user investigates | TASK-NNN with required fields from verification definition |

**Verification definition** lives on the artifact being verified (rule, decision):

```yaml
verification:
  mode: monitor                         # set-and-forget | monitor | reflect
  cadence: "30d"
  trigger: [schedule, project-start]
  source: "plugin:clippy-metrics"       # plugin data endpoint (null = human-only)
  parameters:                           # for automated comparison
    violation-count: { max: 5 }
    trend-direction: { expected: "declining" }
  prompt: "Are violations declining?"   # for human tasks / AI reflection
  required-fields: [action-needed, notes]  # shape of human verification tasks
```

**Relationships** grow over time as the audit trail:

```yaml
relationships:
  - type: verified-by
    target: VER-012
    rationale: "March 2026 — automated, 3 violations (threshold: 5), PASS"
  - type: verified-by
    target: VER-008
    rationale: "February 2026 — automated, 8 violations, FAIL"
  - type: verified-by
    target: TASK-439fa554
    rationale: "February 2026 — human investigation from failed VER-008"
```

**VER-NNN** is an automated data snapshot record: when, what was measured, what was expected, pass/fail. Storage is project-configured — file-based for low-volume, SQLite for high-volume. The relationship references the VER ID regardless of storage backend.

### Enforcement Tooling as Plugins

In software projects, each linting/quality tool is a **plugin** that provides three capabilities:

| Plugin Capability | What it provides | Example |
|------------------|------------------|---------|
| **Skill** | How the tool works, how to configure it for this codebase | "Clippy pedantic catches unwrap/expect. Fix violations like this." |
| **Data collection** | Metrics endpoint for automated verification snapshots | Violation counts, trend data, coverage numbers |
| **Enforcement** | Git hooks, Claude hooks, or both | Pre-commit runs clippy; Claude hook blocks writes with unwrap() |

The same plugin that enforces also provides the data for automated verification. The loop closes within the plugin:

```
Principle (Decision: no unwrap)
  → Practice (clippy-plugin skill: how to comply)
  → Enforcement (clippy-plugin hook: blocks violations)
  → Verification (clippy-plugin data: violation snapshot → VER-NNN)
  → fail? → Task for human → potentially new Observation
```

OrqaStudio core knows nothing about specific tools. It provides:
- Skill loading from plugins
- Hook execution from plugins
- Data snapshot requests from plugins
- Parameter comparison for automated verification
- VER-NNN record creation and task creation on failure

Plugins provide the specifics. Software projects install clippy, ESLint, svelte-check plugins. Non-software projects install different plugins or rely on human verification only. Projects without any quantifiable tooling use reflect-mode exclusively.

**What this means for the pipeline:**

1. **VER-NNN exists** — as automated data snapshot records, not governance documents. Created by the system when plugins provide data.
2. **Verification definition lives on the artifact** — each rule/decision declares what verification looks like.
3. **Plugins close the enforcement-verification loop** — skill + hooks + data in one package.
4. **Two paths, one audit trail** — automated (VER-NNN) and human (TASK-NNN) both recorded as `enforced-by` relationships.
5. **Project setup configures triggers and defaults** — cadence, trigger types, storage backend.
6. **Tasks only created when human attention needed** — automated passes are silent; failures and reflections create tasks.

## Rationale

### Why not independent categories?

The current model (decisions, rules, lessons, skills as independent types with optional cross-references) failed to create a self-enforcing loop. Evidence:

1. **Lessons are write-only** — 16 IMPL files exist, none have been promoted despite [RULE-551bde31](RULE-551bde31) requiring it
2. **Decisions are disconnected** — [AD-774cc3d0](AD-774cc3d0) (component purity) has no structural link to the rule or skill that enforces it
3. **Rules mix concerns** — [RULE-e120bb70](RULE-e120bb70) has enforcement entries for deferral language detection AND workflow gates, because "enforcement" is the only mechanism with teeth
4. **No feedback** — nothing measures whether any of this is working

The root cause: artifacts are islands. The relationships between them are optional and implicit. The pipeline makes relationships mandatory and visible.

### Why keep separate artifact types?

A single "Knowledge" type would be clean but loses semantic distinctions that matter:
- Decisions need rationale and alternatives sections
- Skills need actionable guidance with examples
- Rules need enforcement triggers and actions
- Lessons need context and recurrence tracking

These are genuinely different document shapes. Forcing them into one type would either bloat the schema or lose information. Mandatory relationships connect them without merging them.

### Why mandatory relationships, not explicit thread artifacts?

Explicit thread artifacts (THREAD-NNN.md) would add another type to maintain — exactly the over-engineering pattern this decision was born from. Threads emerge naturally from walking mandatory relationships. The graph IS the thread. Adding a separate artifact to represent it would be redundant metadata about metadata.

### Why visibility over automation?

The drift we experienced wasn't a rule violation — it was a gradual shift in attention. No regex can catch "you've been creating enforcement without learning." But a dashboard showing pipeline flow makes the problem obvious. Self-enforcement through visibility scales better than self-enforcement through constraints, because constraints can be gamed or worked around, while visibility changes what you pay attention to.

## Consequences

### Positive

- **Learning becomes visible** — you can see where knowledge is stuck in the pipeline
- **Drift detection** — the system surfaces "pipeline running backwards" signals
- **Existing artifacts preserved** — no mass migration, additive changes only
- **The loop closes** — verification stage provides feedback on whether enforcement works
- **Principled enforcement** — every rule traces to a principle, every principle traces to learning (or deliberate design)

### Negative

- **Schema changes** — lesson, decision, skill, and rule schemas need new edge fields
- **New artifact type** — verification records need a schema and storage
- **Visualization work** — the pipeline dashboard is significant UI/UX work
- **Retroactive relationship population** — existing artifacts need relationships added (can be incremental)
- **Relationship maintenance** — mandatory relationships add friction to artifact creation (but that friction is the point — it forces connection to purpose)

### Migration Planning Process

This decision is being migrated step by step, constantly checking back against the core question ("What is it you are actually trying to do here?") and the source documents ([RES-132fbb38](RES-132fbb38), this AD) to prevent the drift pattern that motivated it.

**The process:**

1. **Understand current state** — Map every artifact type's schema, count artifacts, identify existing relationships
2. **Test the model against reality** — Do the proposed relationships actually fit all 150 artifacts? What doesn't fit? Why?
3. **Resolve tensions** — Where the model doesn't fit reality, decide: change the model or change the artifacts?
4. **Design schemas** — Concrete schema changes with exact field definitions
5. **Validate against purpose** — Re-read [RES-132fbb38](RES-132fbb38) and this AD. Does the schema serve the learning loop? Or have we drifted into implementation again?
6. **Migration plan** — How to get from current state to target state incrementally
7. **Implement** — Schema changes, edge population, visibility

At each step, check: **"Is this still serving the learning loop, or are we generating implementation details?"**

**Resolution (Step 2→3): Purposeful friction with AI-assisted backfill.**

Edges are **mandatory** — schema validation blocks creation without them. This is intended friction: it forces the creator to connect the artifact to its purpose in the pipeline before it exists. The friction is made nearly free by the AI:
- **Without AI**: Stop, find or create a decision, link it, come back. Feels like bureaucracy.
- **With AI**: "This rule needs a principle. It relates to error handling — connect to [AD-774cc3d0](AD-774cc3d0) or create a new decision?" Feels like a 5-second confirmation.

The system blocks but offers to do the connection work. The human confirms intent. This applies to both the app (AI suggests connections during artifact creation) and the CLI (the orchestrator/plugin suggests connections when an agent creates an artifact).

For the existing 150 artifacts: the AI performs a one-time backfill migration, proposing connections for each disconnected artifact. The human reviews and approves in batches.

**Step 3: Resolve tensions** — COMPLETE

Now that the edge model is decided (mandatory + AI-assisted), the remaining tensions:

**Tension 1: Artifacts that don't have all pipeline neighbors.** RESOLVED.

Edges are required but nullable with a rationale. If a decision has no governance rule enforcing it, set `enforcement: null` with `enforcement-rationale: "enforced by build system, not governance"`. The friction is preserved (you had to think about it) and the rationale captures genuinely useful information about how enforcement actually works.

**Tension 2: Rules that mix enforcement concerns.** RESOLVED.

Initial assessment was that [RULE-e120bb70](RULE-e120bb70)'s two entries are "different triggers for the same principle." On deeper examination (reframing the question): they serve **different principles**.

- Deferral language detection → **honesty**: don't quietly move scope out
- Completion verification → **rigour**: prove the work is done before closing it

An honest team can lack rigour. A rigorous team can lack honesty. These are independent failure modes.

**Resolution:** Honesty and rigour are distinct principles that both need enforcement. The current rule structure conflates them. This doesn't need to be fixed now — the backfill migration will naturally surface it. When the AI connects [RULE-e120bb70](RULE-e120bb70)'s enforcement entries to principles, it will ask "these serve different principles — split or link separately?" The purposeful friction surfaces the design problem at the right moment.

**Broader implication:** Other rules likely conflate principles too. The migration backfill will surface these systematically. This is the pipeline working as designed — forcing connection to purpose reveals structural problems that were invisible when artifacts were islands.

**Tension 3: Skills serve different roles.** RESOLVED.

Skills get a new `category` field that determines what kind of principle edge they require:

| Category | Principle edge points to | Example |
|----------|------------------------|---------|
| `methodology` | Pillar (`PILLAR-NNN`) | planning → [PILLAR-569581e0](PILLAR-569581e0) |
| `domain` | Decision (`AD-NNN`) | orqa-ipc-patterns → [AD-774cc3d0](AD-774cc3d0) |
| `tool` | Decision (`AD-NNN`) | tauri-v2 → [AD-e513c9e4](AD-e513c9e4) |

The category is set at creation time. Schema validates that the `principle` edge matches the expected pattern for that category (`PILLAR-NNN` for methodology, `AD-NNN` for domain/tool). This removes ambiguity — the AI's first question when creating a skill is "is this methodology, domain practice, or tool usage?" and the answer determines what it connects to.

**Emerging insight: "Friction is the feature."** The purposeful friction of mandatory relationships — forcing connection to purpose before creation — may be significant enough to become a pillar or foundational principle. Captured here for future discussion. AI makes the friction nearly free, but the friction itself is non-negotiable.

**Migration Requirements** (must be satisfied before migration is considered complete):

1. **Verification stage must exist.** RESOLVED. Verification has two paths: automated (VER-NNN snapshot records from plugin data, compared against parameters) and human (TASK-NNN created on failure or for reflection). Verification definitions live on artifacts. Enforcement tooling is plugin-based — each tool plugin provides skill + hooks + data collection. The loop closes within the plugin.

2. **Therapeutic question patterns must be encoded.** The questions asked during this planning conversation are themselves a demonstration of the learning loop. The AI should be able to replicate them during artifact creation, planning, and review. Key patterns discovered during this conversation:

   | Moment | Question Pattern | Purpose |
   |--------|-----------------|---------|
   | Artifact creation | "What principle does this serve?" | Forces connection to purpose (mandatory edge friction) |
   | Drift detection | "What are you actually trying to do here?" | Pulls back from implementation to purpose |
   | Assumption testing | "Can you reframe the question so I get a different perspective?" | Challenges first instinct, surfaces hidden assumptions |
   | Principle separation | "Is this one thing or two things wearing one hat?" | Detects conflated concepts (honesty vs rigour) |
   | Scope check | "Does this need resolving now, or will the process surface it naturally?" | Prevents over-engineering by trusting the system |
   | Verification | "Is this still serving the loop, or have we drifted?" | Periodic purpose alignment check |

   And the meta-question that connects the loop to itself:

   | Loop closure | "Before we move on — how does this inform what we're about to do?" | Ensures each step feeds the next rather than being processed in isolation |

   **Key insight: The questions ARE the pipeline, expressed conversationally.**

   Each question maps to a pipeline stage:

   | Question | Pipeline Stage |
   |----------|---------------|
   | "What happened?" | Observation |
   | "Why?" / "Can you reframe this?" | Understanding |
   | "What's the general rule?" / "Is this one thing or two?" | Principle |
   | "How do we encode this?" / "How does this inform what we're about to do?" | Practice |
   | "What are you actually trying to do here?" | Enforcement (of purpose) |
   | "Is this still serving the loop?" | Verification |

   The pipeline isn't a structure with questions bolted on. The pipeline is what these questions look like when they're made structural. The questions aren't a feature — they're the system's core mechanism expressed in two forms: conversational (AI asks them) and structural (artifacts connected by relationships).

   **Encoding approach: RESOLVED.**

   The questions don't need a separate encoding mechanism. They are solved through three complementary approaches:

   **a) Edge rationale as required schema fields.** Every pipeline edge carries a rationale field (e.g., `principle-rationale`) that documents WHY the edge exists. The rationale IS the stage question, answered. The schema requires it — you can't create an edge without explaining its purpose. The friction of answering "why does this connection exist?" is the system asking the pipeline's question. This makes the questions visible in the artifact itself.

   **b) Reframing as an AI behaviour.** The system prompt includes instruction to think about reframing — to offer alternative perspectives when users are making decisions. When reframing produces new understanding that changes existing artifacts, a skill handles the migration. The questions evolve through the learning loop itself, not through a static definition.

   **c) Domain-specific framing as AI skills.** Different projects need different questions. Pre-defined skills handle framing for different domains (software, research, personal planning). Project setup includes a "I don't know, help me figure it out" option — the AI helps discover what questions the project needs. This is an AI capability, not a data structure.

   **Together:** The schema enforces that relationships have rationale (making questions structural). The AI evolves how it asks the questions (making them adaptive). Domain skills shape which questions are most relevant (making them contextual). No new artifact types, no separate question store.

**Tension 4: Enforcement tooling in software projects.** RESOLVED.

Each linting/quality tool is a **plugin** providing three capabilities: skill (how the tool works), data collection (metrics for automated verification), and enforcement (git hooks / Claude hooks). The same plugin that enforces also provides the data for automated verification — the loop closes within the plugin.

OrqaStudio core provides the interfaces: skill loading, hook execution, data snapshot requests, parameter comparison, VER/TASK creation. Plugins provide the specifics. Non-software projects use different plugins or human-only verification.

**Project initialisation**: Two discovery paths during setup:

1. **Automated detection** — the project-inference skill scans the codebase for tooling (languages, frameworks, linters, build tools). Detected tools → suggest verification plugins → once the plugin system has matured, offer to generate the plugin automatically.
2. **User interview** — ask the user: "Do you use any metrics or monitoring tooling for this project?" (e.g., Grafana, Datadog, SonarQube, Linear, Jira, custom dashboards). Same flow: match to existing plugins → install, or offer to create one.

3. **Recommendation** — if the user has no existing tooling, the setup agent recommends options based on project context: project type, scale, budget, team size, existing infrastructure. "Based on your Rust/Svelte project, you might benefit from X for code quality metrics and Y for deployment monitoring. Here are free/paid options."

All three paths converge: detect/ask/recommend → match to existing plugin → install → or generate a new one. The structure and requirements for a tooling plugin (skill + hooks + data collection) are well-defined enough that AI-assisted plugin generation becomes a realistic capability.

**Phase 0 detail — rule-overrides mechanism:**

A task or epic can declare `rule-overrides` in its frontmatter — a list of rules that are scoped-suspended for that work, with rationale:

```yaml
rule-overrides:
  - rule: RULE-a764b2ae
    reason: "Schema fields are being migrated — frontmatter will temporarily diverge"
  - rule: RULE-7b770593
    reason: "Artifact lifecycle is being restructured"
```

This must be recognized at two enforcement points:

1. **Prompt injection** (CLI plugin + app system prompt builder): When loading rules into agent context for a task, check the task's `rule-overrides`. Suspended rules are still loaded but annotated: "This rule is suspended for this task because: [reason]." The agent sees the rule, understands it exists, and knows why it doesn't apply here.

2. **Pre-commit hook**: When validating staged artifacts, check if any in-progress task (via `tmp/session-state.md` or a task reference in the commit message) carries `rule-overrides` that suspend the failing validation. If so, warn but don't block.

The `rule-overrides` field is added to both task and epic schemas. All subsequent migration tasks depend on this mechanism being in place.

**Inheritance model:** Task with no `rule-overrides` inherits from its epic. Task with its own `rule-overrides` **replaces** the epic's entirely — it declares the complete set of suspensions for that task. A rule absent from the task's list is active, even if the epic suspended it. No merge, no special "restore" action — absence means active.
