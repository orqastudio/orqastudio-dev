# Tauri Dev Process Management on Windows

## Problem Statement

`make stop` is unreliable on Windows. The app frequently gets stuck in a state where port 1420 is occupied by an orphaned Vite/node process, preventing `make dev` from starting. This creates a recurring cycle of manual process hunting that breaks development flow.

## Root Causes (3 independent issues)

### 1. Tauri CLI Does Not Kill `beforeDevCommand` Reliably

This is a **known upstream bug** in Tauri. Multiple issues document it:

- [#10023](https://github.com/tauri-apps/tauri/issues/10023): `beforeDevCommand` is not killed when compilation fails with `--no-watch` (Windows, Tauri v2)
- [#2794](https://github.com/tauri-apps/tauri/issues/2794): `beforeDevCommand` is not killed on app exit
- [#4262](https://github.com/tauri-apps/tauri/issues/4262): Development server not killed after window exited
- [#1626](https://github.com/tauri-apps/tauri/issues/1626): Dangling dev server process ŌĆö "sometimes when Tauri exits, the beforeDevCommand still runs in the background"

**Impact**: When `cargo tauri dev --no-watch` exits (crash, compile error, window close), the Vite node process often survives as an orphan. It holds port 1420, blocking the next `make dev`.

A [fix was merged](https://github.com/tauri-apps/tauri/pull/2883) for Tauri v1 (`terminate the beforeDevCommand`), but the problem persists in v2, especially with `--no-watch` on Windows.

### 2. MSYS2 Path Mangling Breaks `taskkill`

Git Bash (MSYS2) automatically converts arguments that look like Unix paths. When `taskkill /F /PID 1234` is called from Git Bash, MSYS2 converts `/F` and `/PID` to `C:/Program Files/Git/F` and `C:/Program Files/Git/PID`.

This is [a known MSYS2 behavior](https://github.com/msys2/MSYS2-packages/issues/84) and is the root cause of `taskkill` hanging or failing silently from Git Bash.

**Workaround**: Set `MSYS2_ARG_CONV_EXCL="*"` to disable path conversion:
```bash
MSYS2_ARG_CONV_EXCL="*" taskkill /F /PID 1234
```
Or use `//F //PID` (double-slash) which is a Git Bash specific escape ŌĆö but this is fragile and not always reliable.

**Better workaround**: Don't use `taskkill` from Git Bash at all. Delegate to PowerShell or a Node.js script that doesn't have path mangling issues.

### 3. Process Tree Not Killed

`taskkill /F /PID <pid>` kills only the target process, not its children. `cargo tauri dev` spawns a tree:

```
cargo tauri dev
Ōö£ŌöĆŌöĆ npm run dev (beforeDevCommand)
Ōöé   ŌööŌöĆŌöĆ node (Vite dev server)        ŌåÉ holds port 1420
ŌööŌöĆŌöĆ cargo run (Rust app)
    ŌööŌöĆŌöĆ orqa-studio.exe               ŌåÉ locks the .exe file
```

Killing `cargo tauri dev` leaves the Vite node process alive. On Windows, `taskkill /T /F /PID` is supposed to kill the tree, but it's unreliable from MSYS2 (see issue #2).

## Solutions Evaluated

### Option A: PowerShell Script (Windows only)

**Approach**: `make stop` delegates to a `.ps1` script on Windows.

**Pros**: `Stop-Process` is reliable, `Get-NetTCPConnection` finds port holders, `Get-CimInstance Win32_Process` walks the process tree.

**Cons**: Calling PowerShell from Git Bash via `-File` flag can also hang due to path mangling. Need `MSYS2_ARG_CONV_EXCL` or use `-Command` with inline script.

**Verdict**: Viable but fragile at the bashŌåÆPowerShell boundary.

### Option B: Node.js Script (cross-platform)

**Approach**: A `.mjs` script using Node's built-in `child_process` and `net` modules, plus `process.kill()`. No npm dependencies needed.

**Pros**:
- Node is already installed (project dependency)
- No MSYS2 path mangling ŌĆö Node runs natively on Windows
- Same script works on all platforms
- `process.kill(pid, 'SIGTERM')` works cross-platform
- Can use `execSync('taskkill /T /F /PID ...')` on Windows for tree kill with no mangling since it's called from Node, not bash
- Can poll ports with `net.createServer().listen()` to verify they're free

**Cons**: Adds a script to maintain. But it's simpler than platform-specific shell scripts.

**Verdict**: Best option. Avoids all three root causes.

### Option C: npm Package (cross-port-killer, fkill, kill-port)

**Approach**: Use an existing npm package like `kill-port`, `fkill`, or `cross-port-killer`.

Packages evaluated:
- [`kill-port`](https://www.npmjs.com/package/kill-port) ŌĆö 189 dependents, cross-platform
- [`fkill`](https://github.com/sindresorhus/fkill) ŌĆö tree kill is [Windows-only](https://github.com/sindresorhus/fkill/issues/21)
- [`cross-port-killer`](https://www.npmjs.com/package/cross-port-killer) ŌĆö cross-platform, actively maintained
- [`kill-port-process`](https://www.npmjs.com/package/kill-port-process) ŌĆö recently updated

**Pros**: Battle-tested, handles edge cases.

**Cons**: Adds a dependency for a dev-only task. Some packages don't handle process trees. Dependency maintenance burden.

**Verdict**: Overkill. A 50-line Node script handles our specific case better.

### Option D: Fix `taskkill` with `MSYS2_ARG_CONV_EXCL`

**Approach**: Prefix all `taskkill` calls with `MSYS2_ARG_CONV_EXCL="*"`.

**Pros**: Minimal change to existing Makefile.

**Cons**: Still doesn't solve the process tree issue. Still platform-specific. Relies on an env var that developers might not understand.

**Verdict**: Band-aid. Fixes symptom #2 but not #1 or #3.

## Recommendation

**Use a Node.js script (Option B)** with the following design:

1. **Port-based killing**: Find processes on ports 1420 and 5173, kill them and their children
2. **Name-based killing**: Also kill `orqa-studio` and `cargo-tauri` by name (catches processes not yet listening)
3. **Tree killing on Windows**: Use `taskkill /T /F` from within Node (no MSYS2 mangling)
4. **Tree killing on Unix**: Use `kill -TERM -<pgid>` for process group kill
5. **Verification**: Poll ports until free (max 10s timeout), exit non-zero if still occupied
6. **Makefile integration**: `stop: node scripts/stop.mjs`

### Why Node over PowerShell

| Factor | Node.js | PowerShell |
|--------|---------|------------|
| Cross-platform | Yes | Windows only (PowerShell Core exists but rare on CI) |
| MSYS2 mangling | Not affected | `-File` flag gets mangled |
| Already installed | Yes (project dep) | Yes (Windows built-in) |
| Maintainability | One script for all OS | Need .ps1 + .sh + Makefile conditionals |
| Process tree kill | Via `taskkill /T` on Win, `kill -pgid` on Unix | Via `Get-CimInstance` on Win only |

## Task Manager Grouping (AppUserModelID)

Windows groups processes in Task Manager and the taskbar using [AppUserModelIDs](https://learn.microsoft.com/en-us/windows/win32/shell/appids). If multiple processes share the same AppUserModelID, they appear grouped under one entry.

**For production builds**: Tauri sets the AppUserModelID automatically from `tauri.conf.json` `identifier`. The built `orqa-studio.exe` gets proper Windows metadata and appears as a single app entry.

**For dev builds**: `cargo tauri dev` spawns `cargo-tauri.exe`, `node.exe`, and `orqa-studio.exe` ŌĆö three different executables. We don't control `node.exe` or `cargo.exe`, so we can't set their AppUserModelID. Dev process grouping in Task Manager is not solvable without upstream changes to the Tauri CLI.

**Conclusion**: AppUserModelID grouping is a production concern (already handled by Tauri). Dev workflow relies on `debugger/dev.mjs` for reliable process management.
