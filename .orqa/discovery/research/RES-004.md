## Q1: Markdown Rendering + Editing

**Question:** What component should handle rendering and editing markdown in OrqaStudio?

**Requirements:**
- Render markdown beautifully (conversation messages, documentation panels)
- Support inline editing (click to edit, save back to file)
- Syntax highlighting in code blocks
- Frontmatter support (YAML metadata in governance artifacts)
- Performance with large documents
- Live preview during editing

### Evaluated Options

#### Option A: CodeMirror 6 (Editor) + Renderer

- **Packages:** `codemirror` (meta), `@codemirror/lang-markdown`, `@codemirror/view`, `@codemirror/state`
- **Svelte wrapper:** `svelte-codemirror-editor` v2.1.0 (published ~Oct 2025, supports Svelte 5 runes via `$state`)
- **Alternative wrappers:** `sveltemirror` (thin wrapper, less maintained), direct integration (CodeMirror's API is vanilla-JS-friendly)
- **Bundle size:** ~250 KB minified for a minimal editor setup, ~75 KB gzipped. `@codemirror/lang-markdown` is 72 KB uncompressed.
- **GitHub stars:** CodeMirror org has ~10k+ stars. svelte-codemirror-editor has ~300+.
- **Last release:** CodeMirror core packages actively maintained (weekly releases). svelte-codemirror-editor v2.1.0 ~Oct 2025.
- **Svelte 5:** svelte-codemirror-editor v2.x explicitly supports Svelte 5 runes. v1.x was for Svelte 3/4/5 without runes. Direct integration also works since CM6 is framework-agnostic.

**Strengths for OrqaStudio:**
- Battle-tested editing engine (used by Obsidian, VS Code web, Firefox DevTools)
- Markdown language mode with syntax highlighting built in
- Extension system: can add YAML frontmatter parsing, custom widgets, folding, search
- Hybrid editing possible: community extensions like Ixora render preview for unfocused lines while keeping active line as raw markdown (Obsidian "Live Preview" style)
- CodeMirror has vim/emacs keybinding extensions (power users)
- Excellent performance with large documents (virtual scrolling built in)

**Weaknesses:**
- Not WYSIWYG by default — requires either split-pane preview or hybrid rendering extensions
- Steeper learning curve for customization (extension API is powerful but involved)
- Svelte wrapper ecosystem is thin (but direct integration is viable and arguably better)

#### Option B: Carta

- **Package:** `carta-md` v4.11.1 (published ~Dec 2025)
- **Plugin packages:** `@cartamd/plugin-code` (syntax highlighting), `@cartamd/plugin-emoji`, `@cartamd/plugin-slash`, `@cartamd/plugin-attachment`, `@cartamd/plugin-anchor`, `@cartamd/plugin-component`
- **Bundle size:** Very small — explicitly designed to avoid CodeMirror/ProseMirror dependencies. Powered by unified/remark/rehype under the hood.
- **Svelte 5:** Full support. Documentation examples use `$state()`, `$props()` syntax. Two components: `MarkdownEditor` (full editor) and `Markdown` (read-only renderer).
- **SSR:** Supports server-side pre-rendering via `carta.render()`

**Strengths for OrqaStudio:**
- Built specifically for Svelte — not a port from React/Vue
- Lightweight: no ProseMirror or CodeMirror dependency
- Split view editor with live preview built in
- Plugin system for extensions (code highlighting, math, slash commands)
- Dual-mode: editor component and read-only renderer from same library
- remark/rehype plugin ecosystem available

**Weaknesses:**
- Smaller community than CodeMirror/TipTap
- Not WYSIWYG — split-pane editing style (source left, preview right)
- May struggle with very large documents (no virtual scrolling like CM6)
- Plugin ecosystem smaller than CodeMirror's

#### Option C: Milkdown (WYSIWYG on ProseMirror)

- **Package:** `@milkdown/kit` v7.18.0 (published ~Jan 2026)
- **Sub-packages:** `@milkdown/core`, `@milkdown/preset-commonmark`, `@milkdown/preset-gfm`, `@milkdown/theme-nord`, `@milkdown/utils`
- **Bundle size:** Heavy — includes full ProseMirror stack
- **Svelte 5:** Uncertain. Official docs have Svelte recipes but community reports difficulties with custom node components in Svelte (one developer switched to TipTap because custom nodes were easier there). Framework-agnostic/headless design.
- **GitHub stars:** ~8k+

**Strengths for OrqaStudio:**
- True WYSIWYG markdown editing — renders as you type
- Plugin-driven architecture (everything is a plugin)
- Y.js support for collaborative editing (future use)
- Headless — no opinionated CSS

**Weaknesses:**
- Custom node components harder in Svelte than React/Vue (community reports)
- Svelte 5 compatibility unconfirmed for advanced use cases
- Heavy bundle (full ProseMirror + Milkdown)
- Opaque debugging — ProseMirror internals are complex

#### Option D: TipTap / Tipex (WYSIWYG on ProseMirror)

- **TipTap package:** `@tiptap/core` (official, framework-agnostic)
- **Svelte 5 wrappers:**
  - `@friendofsvelte/tipex` v0.0.9 (published ~Dec 2025) — Svelte 5 runes, built on TipTap/ProseMirror
  - `svelte-tiptap` — community wrapper, Svelte 5 support via issue #57 (in progress)
  - `sv5tiptap` — community project with Svelte 5 + TipTap 2 + Tailwind 4
- **Bundle size:** Medium-heavy — TipTap core + ProseMirror + extensions
- **Svelte 5:** TipTap officially lists Svelte support. Tipex explicitly uses Svelte 5 runes and snippets.

**Strengths for OrqaStudio:**
- Headless — maximum styling control
- TipTap has enormous extension ecosystem (200+ extensions)
- Rich text editing with markdown serialization
- Tipex provides a Svelte-native starting point with runes
- Better Svelte custom node support than Milkdown (community reports)

**Weaknesses:**
- Tipex is v0.0.9 — early/unstable
- TipTap's Svelte support is secondary to React/Vue
- ProseMirror complexity underneath
- Markdown round-tripping (HTML to markdown and back) can lose formatting nuances

#### Option E: @humanspeak/svelte-markdown (Read-Only Renderer)

- **Package:** `@humanspeak/svelte-markdown` v0.8.13 (published ~Feb 2026)
- **Original:** Fork of `svelte-markdown` (Pablo Berganza), now maintained by Humanspeak Inc.
- **Built on:** marked
- **Svelte 5:** Yes — built with Svelte 5 runes, strict TypeScript
- **Bundle size:** Small (marked has zero dependencies)

**Strengths for OrqaStudio:**
- Intelligent token caching: 50-200x speedup for repeated content, cache hits under 1ms
- Image lazy loading by default
- Svelte 5 with strict TypeScript
- Good for read-only rendering (conversation messages, preview panels)

**Weaknesses:**
- Read-only — no editor component
- Would need pairing with a separate editor for editing use cases

#### Option F: unified/remark/rehype Pipeline (Read-Only Renderer)

- **Packages:** `unified`, `remark-parse`, `remark-rehype`, `rehype-stringify`, `rehype-sanitize`
- **Svelte integrations:** `mdsvex` (compile-time preprocessor), `sveltex-unified` (reactive markdown)
- **Bundle size:** Moderate (multiple packages, but tree-shakeable)

**Strengths for OrqaStudio:**
- Most flexible pipeline: parse markdown to AST, transform, render to HTML
- Enormous plugin ecosystem: remark-gfm, remark-frontmatter, rehype-highlight, rehype-shiki
- Can extract frontmatter YAML as structured data during rendering
- Industry standard for markdown processing

**Weaknesses:**
- Rendering only — no editor
- More plumbing required than a drop-in component
- mdsvex is compile-time only (not for dynamic user content)

### Syntax Highlighting Sub-Decision

For code blocks within rendered markdown and in the editor:

| Library | Bundle | Speed | Quality | Notes |
|---------|--------|-------|---------|-------|
| **Shiki** | ~250 KB + WASM | Slower (7x vs Prism) | VS Code quality (TextMate grammars) | Best for pre-rendered/SSR; `@shikijs/transformers` for line highlighting |
| **highlight.js** | ~50-100 KB (with languages) | Fast (44x faster than Shiki) | Good, 185+ languages | `svelte-highlight` v7.9.0 supports Svelte 5 |
| **Prism** | ~20 KB core | Fastest | Good | Older, less active |

**Recommendation for syntax highlighting:** Use **Shiki** for pre-rendered/static code blocks (governance artifacts viewed in document panels). Use **highlight.js** via `svelte-highlight` for streaming/dynamic code blocks in conversation messages where runtime performance matters.

### Q1 Recommendation

**Two-component strategy:**

1. **Read-only rendering (conversation messages, document preview):** Use `@humanspeak/svelte-markdown` for fast, cached rendering of markdown in conversation messages. Supplement with `svelte-highlight` for code block syntax highlighting. For document panels where artifacts are displayed read-only, the same renderer works. If more control is needed over frontmatter extraction, use the unified/remark pipeline directly.

2. **Editing (governance artifacts):** Use **CodeMirror 6** via `svelte-codemirror-editor` (or direct integration). CodeMirror provides:
   - Excellent large-document performance (virtual scrolling)
   - Markdown language mode with syntax highlighting
   - YAML mode for frontmatter sections
   - Extension ecosystem for search, folding, vim bindings
   - Hybrid preview possible via community extensions (Ixora-style live preview)
   - Split-pane editing with the read-only renderer showing live preview

**Why not Carta?** Carta is appealing as a Svelte-native all-in-one, but CodeMirror 6's virtual scrolling, extension depth, and battle-tested nature make it the safer choice for governance artifacts that can be large. Carta could be reconsidered if CM6 integration proves overly complex.

**Why not Milkdown/TipTap?** WYSIWYG markdown editors fight with markdown's semantics. Governance artifacts are markdown files committed to git — source-level editing (CodeMirror) preserves formatting exactly. WYSIWYG editors can mangle whitespace, frontmatter, and unusual markdown constructs during round-tripping.

**Decision:** CodeMirror 6 for editing + @humanspeak/svelte-markdown for rendering. Revisit Carta if a simpler all-in-one is needed.

## Q2: Conversation UI Component

**Question:** How should the conversation/chat UI be built?

**Requirements:**
- Streaming token display (characters appear as they arrive)
- Tool call rendering (collapsible cards with input/output, approval buttons)
- Agent delegation indicators (which agent is running, what it's doing)
- Message search across session history
- Session history navigation (sidebar or tabs)
- Code blocks with syntax highlighting and copy buttons
- Markdown rendering in messages
- User message editing and resending

### Existing Libraries Evaluated

#### Vercel AI SDK (`@ai-sdk/svelte`)

- **Package:** `@ai-sdk/svelte` v4.0.78 (published daily, extremely active)
- **Parent:** AI SDK 5 — Svelte now has full feature parity with React
- **Key class:** `Chat` (Svelte equivalent of React's `useChat` hook — uses classes, not hooks)
- **Features:** Streaming via SSE, typed tool invocations, message metadata, custom message types, data parts for streaming arbitrary typed data

**Assessment for OrqaStudio:** The AI SDK is designed for SvelteKit server routes proxying to OpenAI/Anthropic. OrqaStudio's architecture is different: Rust backend handles the Claude API, streams via Tauri `Channel<T>`, and the frontend receives events through Tauri's IPC. The AI SDK's transport layer (SSE to SvelteKit server) does not fit OrqaStudio. However, the SDK's **UI patterns** (message types, streaming text accumulation, tool call rendering) are worth studying.

**Verdict:** Do not use as a dependency. Study its patterns for the custom implementation.

#### svelte-ai-chat (YusufCeng1z)

- **Style:** Copy-paste component (shadcn-style, not an npm package)
- **Features:** Glassmorphism UI, mobile-first, Svelte 5 runes, AI SDK integration
- **Dependencies:** marked, zod, lucide-svelte

**Assessment:** Lightweight starting point for a chat widget, but far too simple for OrqaStudio's needs (no tool call cards, no approval flows, no agent delegation, no search).

**Verdict:** Reference for styling patterns only.

#### shadcn-svelte-chat

- **Package:** `@shadcn-svelte-chat/cli` — CLI tool that scaffolds chat components into your project
- **Dependencies:** Requires shadcn-svelte (button, avatar, textarea)
- **Style:** shadcn copy-paste philosophy

**Assessment:** Provides basic chat UI building blocks (message bubbles, input areas, avatars). Useful starting point but would need heavy extension for OrqaStudio's tool call rendering, streaming, and approval flows.

**Verdict:** Evaluate as a starting scaffold, extend significantly.

#### shadcn-svelte-extras Chat Component

- **Package:** Available at shadcn-svelte-extras.com
- **Style:** Additional shadcn-svelte components including chat interfaces

**Assessment:** Similar to shadcn-svelte-chat — basic building blocks.

### Streaming Text Pattern in Svelte 5

The core pattern for streaming token display with Svelte 5 runes:

```svelte
<script>
  // Message content grows as tokens arrive from Tauri Channel<T>
  let content = $state('');
  let isStreaming = $state(false);

  // Tauri channel listener (from Rust SSE → Channel<T>)
  function onToken(token: string) {
    content += token; // $state mutation triggers fine-grained re-render
  }

  // Only the text node containing `content` re-renders — not the whole message list
  // Svelte 5's fine-grained reactivity is ideal for streaming
</script>

<div class="message">
  {@html renderMarkdown(content)}
  {#if isStreaming}
    <span class="cursor" />
  {/if}
</div>
```

Key Svelte 5 reactivity advantages for streaming:
- `$state` mutations trigger only the DOM nodes that depend on the changed value
- No virtual DOM diffing — direct DOM updates
- `$derived` for computed values (e.g., rendered markdown from raw content)
- `$effect` for side effects (e.g., scroll-to-bottom on new content)

### Patterns from Production AI Chat UIs

**Claude.ai patterns:**
- Streaming text with blinking cursor indicator
- Tool use shown as collapsible cards with tool name header
- Artifact panel (separate side panel for generated content)
- Message search via Ctrl+F
- Conversation history in left sidebar

**ChatGPT patterns:**
- Streaming with typing indicator
- Code blocks with copy button and language label
- Canvas mode (side panel for editing generated content)
- Suggested follow-up prompts

**Cursor patterns:**
- Inline diff rendering (green/red for additions/removals)
- File tree integration (show which files are being modified)
- Tool call progress indicators (spinner + file name being edited)
- Accept/reject buttons for each file change

### Q2 Recommendation

**Build custom from shadcn-svelte primitives.** No existing library matches OrqaStudio's requirements.

**Architecture:**

1. **Message list component** — Scrollable, virtualized for long sessions. Each message is a `MessageBubble` component.
2. **MessageBubble component** — Routes to sub-renderers based on content type:
   - Text content: `@humanspeak/svelte-markdown` for rendering, with `svelte-highlight` for code blocks
   - Tool call: Custom `ToolCallCard` component (collapsible, shows name/input/output, approve/deny buttons)
   - Agent delegation: Custom `AgentIndicator` component (agent name, status badge)
3. **Streaming handler** — Svelte 5 `$state` for content accumulation from `Channel<T>`. `$effect` for auto-scroll. `$derived` for markdown rendering of partial content.
4. **Input component** — shadcn-svelte `Textarea` + `Button`. Support for Shift+Enter multiline, Ctrl+Enter send.
5. **Session sidebar** — shadcn-svelte `ScrollArea` + custom list items. Search via FTS5 query to SQLite backend.
6. **Code blocks** — Custom component wrapping `svelte-highlight` with copy button (shadcn-svelte `Button` + clipboard API), language label.

**Key dependencies:**
- `shadcn-svelte` — Base UI primitives (Button, ScrollArea, Collapsible, Textarea, Badge, Avatar, Tooltip)
- `@humanspeak/svelte-markdown` — Markdown rendering in messages
- `svelte-highlight` v7.9.0 — Syntax highlighting in code blocks
- `lucide-svelte` — Icons (copy, check, chevron, etc.)

**Why not use the Vercel AI SDK?** OrqaStudio does not use SvelteKit server routes for AI. The Rust backend manages the Claude API connection. The AI SDK's transport layer (SSE to Node.js server) is incompatible with OrqaStudio's Tauri Channel<T> architecture. Building custom ensures the conversation UI integrates tightly with Tauri's IPC.

**Decision:** Custom build on shadcn-svelte primitives + @humanspeak/svelte-markdown + svelte-highlight.

## Q3: Panel Layout System

**Question:** How should the multi-panel layout work?

**Requirements:**
- Conversation panel (always visible, primary)
- Side panels: docs, file tree, scanners, metrics, artifacts
- Resizable, collapsible panels
- Tabbed panels (multiple docs open)
- Drag to reorder or dock
- Responsive — works on smaller screens (collapse panels to tabs)
- Persist layout state across sessions

### Evaluated Options

#### Option A: PaneForge

- **Package:** `paneforge` v1.0.2 (published ~Sep 2025)
- **Repository:** svecosystem/paneforge (same ecosystem as shadcn-svelte's underlying bits-ui)
- **Svelte 5:** Yes — v1.0.0 explicitly added Svelte 5 support with reactive minSize prop
- **Inspiration:** Port of react-resizable-panels (Bryan Vaughn) — proven API design
- **Components:** `PaneGroup`, `Pane`, `PaneResizer`
- **Features:**
  - Draggable resizers between panes
  - Nested pane groups for complex layouts
  - LocalStorage/cookie persistence for layout state
  - CSS customization via props
  - Accessibility compliant
  - Collapsible panes
  - Shadow DOM support

**Strengths for OrqaStudio:**
- Same ecosystem as shadcn-svelte (svecosystem) — guaranteed compatibility
- shadcn-svelte's `Resizable` component IS PaneForge (it re-exports PaneForge)
- Proven API design from react-resizable-panels
- Built-in layout persistence (LocalStorage) — maps to OrqaStudio's requirement
- Simple, composable API: PaneGroup > Pane + PaneResizer
- Svelte 5 native

**Weaknesses:**
- No built-in tab system (must layer tabs on top)
- No drag-to-dock (fixed panel positions, resizable only)
- No built-in panel ordering/rearrangement

#### Option B: svelte-splitpanes

- **Package:** `svelte-splitpanes` v8.0.12 (published ~Mar 2026 — very actively maintained)
- **Repository:** orefalo/svelte-splitpanes
- **Origin:** Port of vue-splitpanes
- **Svelte 5:** Yes — SvelteKit and TypeScript friendly
- **Features:**
  - Horizontal and vertical splits
  - Min/max size boundaries
  - Snap behavior
  - Expand on double-click
  - Fixed-size panes
  - RTL support with auto-detection
  - Splitter pane pushing
  - First splitter toggle
  - Pane toggle (collapse/expand)
  - Edge snapping
  - Programmatic resizing and two-way size binding
  - Programmatic pane add/remove

**Strengths for OrqaStudio:**
- More features out of the box than PaneForge (snap, push, edge snap, double-click expand)
- Very actively maintained (release within days)
- Comprehensive demo site with examples
- Two-way size binding useful for programmatic layout control

**Weaknesses:**
- Not in the svecosystem (separate from shadcn-svelte)
- No built-in tab system
- No drag-to-dock
- No built-in persistence (must implement manually)
- Ported from Vue — not Svelte-native in origin

#### Option C: Custom CSS Grid

- **No package** — built with CSS Grid/Flexbox + Svelte 5 reactivity
- **Approach:** CSS Grid for overall layout, custom resize handlers, `$state` for panel sizes

**Strengths:**
- Zero dependency
- Full control over every interaction
- Can match any design exactly

**Weaknesses:**
- Significant implementation effort for resize handles, accessibility, keyboard nav
- Must implement persistence manually
- Must handle edge cases (min/max sizes, snap, touch/mouse events)
- Reinventing well-solved problems

#### Option D: Hybrid (PaneForge + Custom Tabs)

- Use PaneForge for the resizable panel structure
- Layer shadcn-svelte `Tabs` component inside panes for tabbed content
- Custom panel header with collapse/expand button
- Persist layout state via Tauri's `tauri-plugin-store` (better than LocalStorage for desktop apps)

### Q3 Recommendation

**PaneForge + shadcn-svelte Tabs + custom panel headers.**

Rationale:
1. **PaneForge is shadcn-svelte's own resizable component** — using anything else means maintaining a parallel dependency for the same functionality.
2. PaneForge's API (PaneGroup/Pane/PaneResizer) maps cleanly to OrqaStudio's layout: `PaneGroup(horizontal) > [Sidebar Pane, PaneResizer, Main PaneGroup(horizontal) > [Conversation Pane, PaneResizer, Detail Pane]]`
3. Built-in persistence via LocalStorage covers the requirement. Can be upgraded to `tauri-plugin-store` for a more desktop-native persistence layer.
4. Tabs are a separate concern — shadcn-svelte's `Tabs` component handles multiple documents open in the same panel.
5. Collapse/expand can be done via PaneForge's collapsible pane support + a custom toggle button.

**Layout structure:**

```
+------------------+------------------------+-------------------+
|                  |                        |                   |
|  Sidebar         |  Conversation          |  Detail Panel     |
|  (collapsible)   |  (primary, min 40%)    |  (collapsible)    |
|                  |                        |                   |
|  - Session list  |  - Message list        |  - Tabs:          |
|  - File tree     |  - Input area          |    - Doc viewer    |
|  - Navigation    |                        |    - Artifact edit |
|                  |                        |    - Scanner       |
+------------------+------------------------+-------------------+
```

This is a standard three-pane layout (similar to VS Code, Slack, email clients). The left sidebar and right detail panel are collapsible. PaneForge handles the resize. Tabs handle the multi-document view in the right panel. No drag-to-dock is needed for MVP — this can be revisited in later phases if users request it.

**Why not svelte-splitpanes?** It has more features, but PaneForge is the canonical choice in the shadcn-svelte ecosystem. Using PaneForge means the resizable panels look and feel consistent with the rest of the shadcn-svelte design system. svelte-splitpanes is a fine library but introduces a separate styling/API paradigm.

**Decision:** PaneForge (via shadcn-svelte Resizable) + shadcn-svelte Tabs. Custom panel headers with collapse toggle.

## Q4: Chart/Visualization Library

**Question:** What library should render metrics and scanner dashboards?

**Requirements:**
- Time series charts (scanner pass/fail over time, metric trends)
- Bar charts (coverage by module)
- Status indicators (pass/fail badges, trend arrows)
- Lightweight (not a full BI tool)
- Works well with Svelte reactivity

### Evaluated Options

#### Option A: LayerChart (built on LayerCake)

- **Package:** `layerchart` — v2.0 in pre-release (`2.0.0-next.#`), stable v1.x available
- **Built on:** LayerCake (graphics framework for Svelte)
- **Svelte 5:** v2.0 is migrating to full Svelte 5 (not just compat mode). Removing LayerCake dependency internally for better typing.
- **shadcn-svelte integration:** shadcn-svelte's `Chart` component IS LayerChart. Example project exists: `techniq/layerchart-shadcn-svelte`.
- **Features:** Area, Bar, Line, Pie, Scatter, Heatmap, Tree, Sankey, and more. Composable Svelte components.
- **CSS:** Supports css-only usage without Tailwind, with first-class Tailwind support.

**Strengths for OrqaStudio:**
- shadcn-svelte's official charting solution — guaranteed design consistency
- Composable Svelte components (not a canvas/WebGL black box)
- Time series, bar charts, and line charts all supported
- Svelte-native reactivity — charts update reactively with `$state`
- Excellent documentation with examples

**Weaknesses:**
- v2.0 is pre-release (potential breaking changes)
- More assembly required than Chart.js (composable means more code per chart)
- Smaller community than Chart.js/ECharts

#### Option B: LayerCake (raw)

- **Package:** `layercake` v10.0.2 (published ~Nov 2025)
- **GitHub stars:** ~1,700
- **Svelte 5:** v8.4.0+ supports Svelte 5. v10.0.0 released Aug 2025. All examples now use runes syntax.
- **Bundle size:** Very small — LayerCake is a layout/scale framework, you draw with standard SVG/Canvas/HTML

**Strengths:**
- Maximum flexibility — you compose charts from SVG/Canvas/HTML
- Tiny core (layout + scales only)
- D3 scales integration

**Weaknesses:**
- Very low-level — you write the SVG paths, axes, legends yourself
- Significant code per chart type
- LayerChart exists precisely to solve this problem

#### Option C: Chart.js (via svelte5-chartjs)

- **Package:** `svelte5-chartjs` (fork of stale `svelte-chartjs`)
- **Repository:** LupusAI/svelte5-chartjs
- **Chart.js version:** Chart.js v4+
- **Svelte 5:** Yes — specifically built for Svelte 5
- **Bundle size:** Chart.js core is ~200 KB minified, tree-shakeable in v3+
- **Features:** Line, Bar, Pie, Doughnut, Radar, Polar, Bubble, Scatter charts

**Strengths:**
- Familiar Chart.js API — enormous community and documentation
- Tree-shakeable — only include chart types you use
- Quick to implement — declarative component API
- Animations built in

**Weaknesses:**
- Canvas-based rendering — not SSR-friendly, less accessible
- Not Svelte-native — wrapper around a vanilla JS library
- `svelte5-chartjs` is a community fork, not officially maintained
- Chart.js reactivity is its own system, not Svelte's `$state`

#### Option D: uPlot (via uplot-svelte)

- **Package:** `uplot-svelte` (from `uplot-wrappers` by skalinichev)
- **uPlot core:** ~45 KB minified
- **Performance:** Creates interactive chart with 150,000 data points in ~90ms
- **Rendering:** Canvas 2D (no WebGL/WASM)
- **Chart types:** Time series, lines, areas, OHLC, bars

**Strengths:**
- Extremely fast — best performance of any option
- Tiny bundle size (~45 KB)
- Good for large datasets

**Weaknesses:**
- Limited chart types (primarily time series / line / bar)
- Svelte wrapper is community-maintained, unclear Svelte 5 status
- Minimal styling/theming out of the box
- Less polished defaults — requires more configuration

#### Option E: Apache ECharts (via svelte-echarts)

- **Package:** `svelte-echarts` (bherbruck/svelte-echarts)
- **ECharts core:** Very large (~1 MB, but tree-shakeable)
- **Svelte 5:** No native Svelte wrapper, but integrates via Svelte `use:action` or `onMount`. The svelte-echarts wrapper has recent dependency updates (Apr 2025).
- **Features:** Virtually every chart type imaginable (100+ chart types)

**Strengths:**
- Most comprehensive charting library available
- Beautiful defaults with themes
- Interactive tooltips, zoom, brush selection
- Heatmaps, treemaps, 3D charts if ever needed

**Weaknesses:**
- Massive bundle size (needs aggressive tree-shaking)
- Overkill for OrqaStudio's dashboard needs
- Canvas/WebGL rendering — not Svelte-native
- Svelte wrapper is community-maintained, thin

#### Option F: Unovis (@unovis/svelte)

- **Package:** `@unovis/svelte`
- **Features:** Modular dashboard components, dark mode via CSS variables
- **Svelte 5:** Compatible via wrapper

**Strengths:**
- Designed for SaaS dashboards (matches OrqaStudio's use case)
- Separates data logic from visual rendering
- Dark mode built in

**Weaknesses:**
- Smaller community
- Less documentation than LayerChart or Chart.js

### Q4 Recommendation

**LayerChart (via shadcn-svelte Chart component).**

Rationale:
1. **LayerChart is shadcn-svelte's official charting solution** — it is literally the `Chart` component in shadcn-svelte. Using anything else means fighting the design system.
2. Composable Svelte components mean charts integrate naturally with Svelte 5 reactivity (`$state`, `$derived`).
3. Covers all of OrqaStudio's needs: time series (line/area), bar charts, and can render status indicators via custom Svelte components.
4. v2.0 pre-release is a risk, but v1.x is stable and the shadcn-svelte integration works today.
5. If LayerChart proves insufficient for a specific chart type later, ECharts can be added as a targeted supplement — but OrqaStudio's Phase 3 needs (pass/fail trends, coverage bars, status badges) are well within LayerChart's capability.

**For status indicators (badges, trend arrows):** These are not charts — they are simple UI components. Use shadcn-svelte `Badge` + `lucide-svelte` icons (TrendingUp, TrendingDown, CheckCircle, XCircle). No charting library needed.

**Decision:** LayerChart via shadcn-svelte Chart component. Status indicators via shadcn-svelte Badge + lucide-svelte icons.

| **Status indicators** | shadcn-svelte Badge + icons | — | `shadcn-svelte`, `lucide-svelte` |

## Key Theme: shadcn-svelte as the Unifying Layer

The strongest pattern across all four decisions is that **shadcn-svelte has already made most of these choices**:
- Its `Resizable` component is PaneForge
- Its `Chart` component is LayerChart
- Its component library (Button, Tabs, Badge, Collapsible, ScrollArea, etc.) provides the building blocks for the conversation UI
- Its design system (Tailwind + CSS variables) provides the styling foundation

By aligning with shadcn-svelte's choices, OrqaStudio gets a coherent, well-maintained design system where all components work together. The only choices outside the shadcn-svelte ecosystem are CodeMirror 6 (for editing — no shadcn-svelte editor component exists) and the markdown renderer (which shadcn-svelte does not provide).
