## Current State

The ArtifactNav component has a text-only `SearchInput` (`h-7`, `size="xs"`) in a `border-b border-border p-2` container at the top. Below it, a `ScrollArea` renders either a tree view (collapsible folders with leaf items) or a flat list of `ArtifactListItem` components.

The NavSubPanel header is `h-10` with `text-xs font-semibold uppercase tracking-wide text-muted-foreground`. The Breadcrumb bar in the ArtifactViewer is also `h-10`. These establish the visual rhythm вҖ” `h-10` (40px) is the standard fixed-height bar in the sidebar.

The current `SearchInput` is `h-7` (28px) inside `p-2` padding, making the total bar ~44px вҖ” close to `h-10` but inconsistent.

## Design Decision: Remove Tree Mode

The ArtifactNav currently has two rendering modes: tree (for Documentation with subdirectories) and flat list. Tree mode is no longer needed.

**Rationale:** The existing three-layer navigation вҖ” ActivityBar вҶ’ GroupSubPanel вҶ’ ArtifactNav вҖ” already provides hierarchical navigation. Documentation subdirectories (architecture, product, development, etc.) are registered as separate artifact types in `project.json`, each with their own entry in GroupSubPanel. With collapsible grouping support in the flat list (group-by status, priority, or any enum field), the artifact panel gains a powerful secondary navigation structure that works uniformly across ALL artifact types вҖ” not just docs.

Tree mode was a Documentation-specific accommodation. Grouping is a universal capability that replaces the need for it.

**Impact:** Remove the `isTree`, `treeSection` snippet, and `Collapsible` tree rendering from ArtifactNav. All artifact types render as flat lists with optional collapsible group headers.

## Design Decision: Toolbar Replaces Search

The text filter input is removed from the artifact panel. In its place: a compact icon-based toolbar at exactly `h-10` to match the NavSubPanel header and breadcrumb bar heights.

Search becomes a global AI-driven feature in the ActivityBar вҖ” it leaves the artifact panel entirely.

## Filter Dropdown Design

The filter dropdown is the most UX-critical element. It must feel structured and scannable, not like a generic checkbox list.

**Component choice: Popover, not DropdownMenu.** The existing `SessionDropdown.svelte` establishes a pattern of using `Popover` for rich interactive panels (with search inputs, scroll areas, custom list rendering). `DropdownMenu` is better for simple action menus. The filter panel is a rich panel вҖ” it uses `Popover`.

### Design Details

**Container:**
- `Popover.Content` with `w-52` width (208px) вҖ” compact for a sidebar panel
- `align="start"` вҖ” aligns to the left edge of the filter button
- `sideOffset={4}` вҖ” small gap from toolbar
- Scrollable if content exceeds viewport: `max-h-[calc(100vh-200px)] overflow-y-auto`
- Uses the same Popover pattern as `SessionDropdown.svelte` вҖ” proven in the codebase for rich interactive panels

**Section Headings:**
- Custom `<p>` element: `text-[10px] font-semibold uppercase tracking-widest text-muted-foreground/60 px-2 pt-3 pb-1`
- Designed to be smaller and more recessive than list items вҖ” they are category labels, not items themselves

**Checkbox Rows:**
- Custom checkbox rows (not DropdownMenu.CheckboxItem вҖ” we're in a Popover): `h-7 flex items-center gap-2 px-2 text-xs rounded hover:bg-accent/50 cursor-pointer`
- Use shadcn `Checkbox` component (`h-3.5 w-3.5`) on the left
- Label text next to checkbox
- For `status` fields: include a status dot (`h-2 w-2 rounded-full`) before the label, using existing `StatusIndicator` colour system
- Clicking the row toggles the checkbox (entire row is the click target, not just the checkbox)

**Status Dot Colours** (reuse existing `StatusIndicator` `GROUP_DOT_CLASSES`):
- Draft/Todo/Captured/Planning/Proposed вҶ’ `bg-zinc-400` (gray)
- In-progress/Exploring/Active/Ready вҶ’ `bg-blue-500`
- Done/Complete/Accepted вҶ’ `bg-emerald-500`
- Review/Recurring вҶ’ `bg-amber-500`
- Promoted вҶ’ `bg-purple-500`
- Superseded/Deprecated/Archived/Surpassed вҶ’ `bg-red-400`

**Priority Colours:**
- P1 вҶ’ `text-red-500 font-medium`
- P2 вҶ’ `text-amber-500`
- P3 вҶ’ `text-muted-foreground`

**Separators:**
- `<div class="h-px bg-border my-1" />` between sections

**Clear Action:**
- Only visible when any filter is active
- `text-xs text-primary hover:underline cursor-pointer px-2 py-1.5`
- Positioned at the bottom, after the last section separator

### Schema-Driven Dynamic Sections

The filter panel is **not hardcoded** to specific fields. It dynamically generates sections by reading the `schema.json` for the current artifact type. The algorithm:

1. Load the schema for the artifact type being browsed (the scanner already knows which directory вҶ’ which schema)
2. Walk the schema's `properties` object
3. For each property that has an `enum` array, generate a filter section with those values as checkbox options
4. Skip non-filterable field types (strings without enum, arrays, objects, patterns)
5. Humanize field names for section headings (e.g. `status` вҶ’ "STATUS", `evolves-into` вҶ’ skip because it's not an enum)

**Supported field types for filtering:**

| Schema Pattern | Filter Behaviour |
|---------------|-----------------|
| `{ "enum": [...] }` | Checkbox section with one item per enum value |
| `{ "type": "string", "enum": [...] }` | Same вҖ” checkbox section |
| Everything else | Not filterable вҖ” skip |

This means:
- `status` generates a section automatically for ALL types (they all have a status enum)
- `priority` generates a section only for Epics (the only type with a priority enum)
- `layer` generates a section for Rules, Skills, Agents, Decisions
- `scope` generates a section for Rules, Skills, Agents
- If a new enum field is added to any schema in the future, it automatically appears in the filter panel вҖ” no frontend code change needed

**Visual enhancements by field name:**
The system is generic, but specific field names get visual treatment:
- `status` вҶ’ items get a status dot (colour from `StatusIndicator`)
- `priority` вҶ’ items get priority colours (P1 red, P2 amber, P3 muted)
- All other enum fields вҶ’ plain text labels

These are optional decorators, not hardcoded sections. The filter panel works without them вҖ” they just make known fields prettier.

**Schema delivery to frontend:**
The Rust scanner includes the `schema.json` enum fields in the NavType response (not the full schema вҖ” just the filterable field names and their enum values). This avoids the frontend needing to fetch schemas separately.

- `DropdownMenu.RadioGroup` with `DropdownMenu.RadioItem` for each option
- Active sort shown with `CheckIcon` on the left (built into RadioItem)
- Items: `h-8 px-2 text-xs`
- Sort options are dynamically generated from the schema:
  - `title` (string, always present) вҶ’ "Title (A вҶ’ Z)" / "Title (Z вҶ’ A)"
  - `created` (date, always present) вҶ’ "Created (newest)" / "Created (oldest)"
  - `updated` (date, always present) вҶ’ "Updated (newest)" / "Updated (oldest)"
  - Any other `date` or `string` field вҶ’ humanized label with direction
- The universal fields (title, created, updated) appear for every type. Type-specific sortable fields appear dynamically.

**Group Section:**
- `DropdownMenu.RadioGroup` for group-by (only one active grouping)
- Options generated from the same schema enum detection as the filter panel:
  - "None" (always first)
  - One option per enum field in the schema (status, priority, layer, etc.)
- When a group-by is active, the artifact list renders with **collapsible section headers** instead of a flat list. Each group header uses the same style as the old tree folder headers: `text-xs font-semibold uppercase tracking-wide text-muted-foreground` with a chevron toggle.
- Groups are collapsible вҖ” this replaces the tree rendering capability with something more universally useful across all artifact types
- **Group ordering**: By default, groups are ordered according to the enum array order in `schema.json` (which is intentional вҖ” e.g., status enums are ordered by lifecycle: `draft вҶ’ ready вҶ’ in-progress вҶ’ review вҶ’ done`). `_navigation.json` can override this with a `group_order` map. If a group-by field has no configured order and no schema enum order, groups fall back to alphabetical.

**Reset Action:**
- Same pattern as "Clear all" in filter dropdown
- Only visible when current sort/group differs from `_navigation.json` defaults

## Sorting & Filtering Logic (Client-Side)

All sorting and filtering happens client-side on the DocNode array. No new backend commands needed.

### Filtering

```typescript
function applyFilters(nodes: DocNode[], filters: Record<string, string[]>): DocNode[] {
  return nodes.filter(node => {
    for (const [field, values] of Object.entries(filters)) {
      if (values.length === 0) continue;
      const nodeValue = getNodeFieldValue(node, field);
      if (nodeValue && !values.includes(nodeValue)) return false;
    }
    return true;
  });
}
```

`getNodeFieldValue` extracts a field value from the DocNode. Currently DocNode only exposes `status`. To support filtering by `priority`, `layer`, etc., DocNode needs a `frontmatter` field вҖ” a `Record<string, unknown>` carrying the parsed YAML fields.

**Key change**: The Rust scanner must include frontmatter fields in DocNode for leaf nodes. This is a small backend change вҖ” the scanner already parses frontmatter for `status` and `description`; it just needs to include additional fields.

### Sorting

```typescript
function applySort(nodes: DocNode[], sort: SortConfig): DocNode[] {
  return [...nodes].sort((a, b) => {
    const aVal = getNodeFieldValue(a, sort.field);
    const bVal = getNodeFieldValue(b, sort.field);
    // String comparison for title/status; date comparison for created/updated
    const cmp = compareValues(aVal, bVal, sort.field);
    return sort.direction === "asc" ? cmp : -cmp;
  });
}
```

### Grouping

When `group` is set (e.g. "status"), the sorted list is partitioned into sections:

```typescript
function applyGrouping(nodes: DocNode[], groupField: string): Map<string, DocNode[]> {
  const groups = new Map<string, DocNode[]>();
  for (const node of nodes) {
    const key = getNodeFieldValue(node, groupField) ?? "Other";
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key)!.push(node);
  }
  return groups;
}
```

Group headers are ordered using the following priority:

1. **`_navigation.json` `group_order`** вҖ” if the grouped field has an explicit order in config, use it
2. **Schema enum order** вҖ” if the field is an enum in `schema.json`, use the enum array order (which is intentional lifecycle ordering)
3. **Alphabetical** вҖ” fallback for fields with no configured or schema-defined order

```typescript
function getGroupOrder(
  groupField: string,
  navConfig: NavigationConfig | null,
  filterableFields: FilterableField[]
): string[] | null {
  // 1. Check _navigation.json group_order override
  const configOrder = navConfig?.defaults?.group_order?.[groupField];
  if (configOrder) return configOrder;

  // 2. Fall back to schema enum order
  const schemaField = filterableFields.find(f => f.name === groupField);
  if (schemaField) return schemaField.values;

  // 3. null = alphabetical
  return null;
}
```

When rendering, groups whose keys appear in the order array are displayed in that order. Any groups with keys NOT in the order array are appended at the end alphabetically (handles unexpected/new values gracefully).

Grouped output renders with section headers (same style as tree folder headers: `text-xs font-semibold uppercase tracking-wide text-muted-foreground`).

## DocNode Frontmatter Extension

The DocNode type needs a frontmatter bag:

```typescript
export interface DocNode {
  label: string;
  path: string | null;
  children: DocNode[] | null;
  status?: string | null;
  description?: string | null;
  icon?: string | null;
  /** Additional frontmatter fields for sorting/filtering */
  frontmatter?: Record<string, unknown>;
}
```

The Rust backend populates this with ALL scalar frontmatter fields from the YAML parse (strings, numbers, dates, booleans). Array and object fields are excluded вҖ” they're not useful for sort/filter. The frontend reads whatever is provided, and the schema-driven filter/sort system picks out the fields it can use (enums for filtering, strings/dates for sorting).

## Implementation Phases

## Decisions (resolved during design review)

1. **Tree mode removed.** The three-layer navigation (ActivityBar вҶ’ GroupSubPanel вҶ’ ArtifactNav) plus collapsible grouping replaces tree rendering. Grouping is universally applicable; tree mode was Documentation-specific. Documentation already decomposes into sub-groups via config.

2. **Filter/sort is fully schema-driven.** Sections are not hardcoded to "status, priority, layer". The system reads `schema.json` for the artifact type and generates filter sections for enum fields and sort options for date/string fields. Visual decorators (status dots, priority colours) are optional enhancements keyed by field name вҖ” the system works without them.

3. **No count badges.** The filter panel is for navigation, not analytics. Checkbox items show the value label only вҖ” no counts next to each option.

4. **DocNode carries all scalar frontmatter.** Not a curated subset вҖ” all scalar fields from the YAML parse. The schema-driven frontend picks what it can use. This avoids the backend needing to know which fields are "useful" for filtering.
