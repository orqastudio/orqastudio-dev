## Decision

Context injection shifts from **content-based** (preload all potentially-relevant
docs, skills, and rules into every prompt) to **graph-based** (inject the artifact
relationship map with instructions on how to traverse it; agents read nodes on
demand).

The artifact graph is:
- **Nodes**: Every `.orqa/` markdown file with YAML frontmatter
- **Edges**: Relationship fields in frontmatter (`epic`, `depends-on`, `docs`,
  `skills`, `research-refs`, `docs-required`, `docs-produced`, `related-rules`,
  `scope`, `evolves-into`, `evolves-into`)

The orchestrator prompt becomes ~200 lines:
1. How to read the graph (follow edges to find context)
2. How to extend the graph (add edges when creating/modifying artifacts)
3. The process model (understand → plan → document → implement → review → learn)
4. Role definitions and ownership boundaries
5. NON-NEGOTIABLE safety constraints

Everything else — rules, skills, injection tables, project specifics — lives in
the graph and is read on demand.

### Key Schema Changes

- **`task.docs`** (new): Array of documentation paths to load during implementation
- **`task.skills`** (new): Array of skill names to load during implementation
- **`research.sources`** (new): Structured external references with URLs and tiers

These replace the hardcoded Tier 2 injection table in the orchestrator prompt.
The project wires skills to tasks via graph edges rather than path-matching tables.

### Self-Building Loop

1. Plugin reads the graph to determine what context to inject
2. Agent works with that context, creates/modifies artifacts
3. New artifacts extend the graph with proper relationships
4. Next session benefits from richer graph relationships
5. The system improves itself through normal use

### No Recursion Risk

The graph is a static data structure (nodes + edges), not executable. Instructions
say "read nodes when needed, add edges when you create/modify." The agent does not
re-process the graph or execute it — it navigates it.

## Rationale

1. **Context window efficiency**: Loading 40 rules + all skill injection tables
   consumes thousands of tokens on every session, most irrelevant to the task.
   Graph traversal loads only what's needed.

2. **Self-maintaining**: Adding a new skill with the right scope makes it
   automatically discoverable. No orchestrator prompt update needed.

3. **Domain-agnostic**: The graph structure works for any domain — software,
   marketing, research, legal. Only the project-layer nodes change.

4. **Plugin-app parity**: Both the companion plugin and the native app consume the
   same graph. Feature parity is structural, not duplicated implementation.

5. **Scalability**: As the governance system grows (more rules, skills, docs), the
   current approach degrades (bigger prompt). The graph approach stays constant
   (same ~200 line prompt, more nodes to traverse).

## Consequences

### Positive

- Orchestrator prompt drops from ~2000 lines to ~200 lines
- New skills/rules are auto-discoverable via graph edges
- Plugin and app achieve true feature parity
- System becomes self-building through normal use
- Works for non-software domains without code-specific assumptions

### Negative

- Agents must navigate the graph, adding read operations per task
- Graph integrity becomes critical — broken edges mean missing context
- Bootstrap problem: agents need to read the graph navigation instructions from
  somewhere (solved by keeping these in the orchestrator prompt)
- Backfill work needed to add `docs` and `skills` to existing tasks

### Risks

- **Cold start**: A new project with an empty graph has no context to inject.
  Mitigated by core firmware skills that are always loaded.
- **Graph drift**: Artifacts with missing or wrong edges degrade injection quality.
  Mitigated by [TASK-dbc452ad](TASK-dbc452ad) (plugin warns on missing edges) and [RULE-98682b5e](RULE-98682b5e) (core
  protection).
