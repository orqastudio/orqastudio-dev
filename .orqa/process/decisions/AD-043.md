## Decision

Reorganize `.orqa/` from the current three-way split (governance/team/planning) into a new three-level structure:

```
.orqa/
  process/              # The machine — pipeline artifacts
    pillars/            #   Guiding principles
    lessons/            #   Observations & understanding
    decisions/          #   Architectural principles
    rules/              #   Enforcement
    skills/             #   Practice
    agents/             #   Roles

  delivery/             # The work — idea to verified outcome
    ideas/              #   IDEA-NNN
    research/           #   RES-NNN
    milestones/         #   MS-NNN
    epics/              #   EPIC-NNN
    tasks/              #   TASK-NNN
    verification/       #   VER-NNN

  documentation/        # The human layer — guides, product docs
    guide/              #   What it all means, how to use the app
    product/            #   Vision, roadmap, product narrative
    architecture/       #   Contributor-facing architecture overview
    development/        #   Dev setup, commands, coding standards ref
```

### The First-Class Artifact Principle

If an artifact is needed by the learning loop, enforcement mechanism, or delivery cycle, it must be a first-class artifact — schema-validated, relationship-tracked, integrity-enforced.

Documentation is a human-readable guide maintained by AI. It explains what things mean, how to use the app, and documents the product being built. It references process artifacts but never defines them. The process artifacts define themselves.

## Rationale

**Current structure** groups by organizational concern:
- `governance/` = rules + decisions + lessons (enforcement)
- `team/` = skills + agents (people/roles)
- `planning/` = work items + pillars (delivery)

This doesn't reflect how the knowledge maturity pipeline actually flows. The pipeline is: Observation (lesson) → Understanding (lesson) → Principle (decision) → Practice (skill) → Enforcement (rule) → Verification. All of these are **process** artifacts — they form the learning machine. Scattering them across `governance/` and `team/` obscures this.

**New structure** groups by function:
- `process/` = the learning machine (all pipeline artifacts)
- `delivery/` = the work pipeline (ideas → research → milestones → epics → tasks → verification)
- `documentation/` = human-readable guides (not process definitions)

## Consequences

**Positive:**
- Pipeline artifacts are co-located, making the learning loop visible
- Delivery artifacts are co-located, making the work pipeline clear
- Documentation is scoped to human-consumption content, removing confusion about what defines process
- Pillars move to `process/` where they belong (they're guiding principles, not delivery items)

**Negative:**
- Every path reference in the codebase must be updated
- `.claude/` symlinks must be recreated
- `project.json` artifacts array must be rewritten
- Pre-commit hook paths must change
- Temporary disruption during migration
