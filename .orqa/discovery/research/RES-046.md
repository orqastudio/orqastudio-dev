## Research Questions

1. How should OrqaStudio support multiple AI providers?
2. Should each provider be its own sidecar, or adapters in one sidecar?
3. How do local models fit in?
4. What should be built-in vs plugin?
5. Should GitHub integration be built-in?

---

## Findings

### Current Architecture (Already Well-Positioned)

The existing sidecar architecture is **already provider-agnostic**:

- **Provider interface** (`provider-interface.ts`) defines the contract: `healthCheck()`, `streamMessage()`, `cancelStream()`, `generateSummary()`, `resolveToolResult()`, `resolveToolApproval()`
- **Provider factory** (`providers/index.ts`) with `createProvider(type?)` — currently returns only `ClaudeAgentProvider`
- **NDJSON protocol** between Rust and sidecar is provider-neutral per [AD-ff88ecea](AD-ff88ecea)
- The Rust backend **does not know or care** which AI provider the sidecar uses

Existing idea artifacts: [IDEA-0b05f1fd](IDEA-0b05f1fd) (local models), [IDEA-c31c1c6b](IDEA-c31c1c6b) (multi-provider routing), [IDEA-889dffd4](IDEA-889dffd4) (promoted to EPIC-0a7b21cf).

### Provider Architecture Options

#### Option A: Adapters Inside One Sidecar (Recommended)

Add providers to the existing sidecar. The `Provider` interface and factory are already in place. An `OpenAICompatibleProvider` covers OpenAI, Anthropic direct API, Ollama, LM Studio, and any OpenAI-compatible endpoint with one implementation.

- Single NDJSON protocol, single process to manage
- Rust needs zero changes
- Tool routing via MCP works identically for all providers
- Sidecar grows larger but stays cohesive

#### Option B: Multiple Sidecars

Each provider family gets its own sidecar binary. Rust selects which to spawn based on project config.

- Process isolation (one provider crash doesn't affect others)
- Each can use best runtime (Bun, Python, etc.)
- Multiple codebases to maintain
- NDJSON protocol must be implemented in each

#### Option C: Native Rust for Simple Providers

Skip sidecar for simple API providers (OpenAI, Ollama). Direct HTTP from Rust via `reqwest`. Keep sidecar only for Claude Agent SDK.

- Eliminates Node.js dependency for simple providers
- Two provider patterns to maintain (sidecar + native)
- Must implement SSE streaming in Rust

**Recommendation**: Start with Option A. The interface is ready. An `OpenAICompatibleProvider` is high-leverage — one adapter covers most providers. Extract to separate sidecars later if needed.

### Local Model Support

Local models (Ollama, LM Studio, llama.cpp) expose **OpenAI-compatible HTTP endpoints**. The same `OpenAICompatibleProvider` that handles cloud OpenAI also handles local Ollama with zero additional code. Differences are configuration only:

- Endpoint URL: `http://localhost:11434/v1` instead of `https://api.openai.com/v1`
- No API key needed (or dummy key)
- Model names are local identifiers (e.g., `llama3.2:7b`)
- No rate limits

**Key insight**: "Local model support" is not a separate feature — it's a configuration variant of the OpenAI-compatible provider.

### Built-in vs Plugin Decision Framework

| Criterion | Built-in | Plugin |
|-----------|---------|--------|
| Required by ALL users regardless of domain | Yes | No |
| Serves a core pillar directly | Yes | Maybe |
| Needs deep streaming pipeline integration | Yes | No |
| Provider-specific | No | Yes |
| Domain-specific (software only) | No | Yes |
| Requires external service accounts | No | Yes |
| Can evolve independently | No | Yes |

### Applying the Framework

| Feature | Verdict | Reason |
|---------|---------|--------|
| Provider abstraction (NDJSON, Provider interface) | **Built-in** | Core infrastructure |
| Claude Agent SDK provider | **Built-in** | Ships with app, reference implementation |
| OpenAI-compatible provider | **Built-in** | Universal — covers cloud + local |
| Specific cloud adapters (Azure, Vertex) | **Plugin** | Service-specific |
| Ollama/local model support | **Built-in** | Covered by OpenAI-compatible adapter |
| Model routing per agent role | **Built-in config** | Part of project settings |
| GitHub integration | **Plugin** | Not all users use GitHub; domain-specific |
| Issue tracker sync (Linear, Jira) | **Plugin** | Service-specific |
| Custom dashboards/views | **Plugin** | User-specific needs |
| Component library | **Built-in SDK** | Plugins need shared components to render views |

### GitHub Integration Scope

If built as a plugin:

| Capability | Complexity | Value |
|-----------|-----------|-------|
| Branch management from tasks | Low | High |
| PR creation with governance context | Low | High |
| Issue sync (bidirectional) | Medium | Medium |
| Status sync (PR merge updates artifacts) | Medium | High |
| Comment sync to lessons | Medium-High | Medium |

Linear's deep bidirectional sync is the gold standard. A GitHub plugin could start with PR creation + branch management (low complexity, high value) and grow from there.

### Plugin Architecture Requirements

For plugins to create dynamic views, the app needs:

1. **Component library SDK** — shared components (EmptyState, StatusIndicator, etc.) exposed as importable library
2. **Artifact Graph SDK** — already exists (`artifact-graph.svelte.ts`), needs documentation
3. **View registration API** — plugins register custom views for artifact types or dashboard panels
4. **Theme tokens** — plugins need access to the design system

This suggests a future epic for "Plugin View SDK" — extracting the component library and creating the registration mechanism.

---

## Recommendations

1. **Next provider**: Add `OpenAICompatibleProvider` to the existing sidecar — covers OpenAI + Ollama + LM Studio in one adapter
2. **GitHub integration**: Build as a plugin, start with PR creation + branch management
3. **Component library**: Extract as SDK for plugin authors (high-priority architectural work)
4. **Model routing**: Configuration concern, not architecture — add per-role model selection to project settings once multiple providers exist
