## Decision

Agent definitions represent **universal roles** — portable across software,
consulting, research, operations, or any other project type OrqaStudio
manages. Domain-specific knowledge lives in **skills** that are loaded
into roles based on the project context.

## Context

OrqaStudio's mission is domain-agnostic structured thinking. But the
initial agent set (16 agents) was designed around a software development
team: backend-engineer, frontend-engineer, debugger, security-engineer,
code-reviewer, etc. These roles are not portable — they make no sense
in a consulting engagement or a research project.

The insight: in any project, you still need someone who investigates,
plans, implements, reviews, and documents. The **roles** are universal.
The **knowledge** that makes them effective in a specific domain is not.

## The Seven Universal Roles

| Role | Purpose | Delegates TO this role |
|------|---------|----------------------|
| **Orchestrator** | Coordinates work, enforces process, manages governance | N/A — this IS the coordinator |
| **Researcher** | Investigates, gathers information, analyses findings | "Find out about X", "Evaluate options for Y" |
| **Planner** | Designs approaches, evaluates tradeoffs, maps structure | "Plan how to do X", "Assess the architecture of Y" |
| **Implementer** | Does the work — whatever "work" means in this domain | "Build X", "Write the code for Y", "Create the deliverable" |
| **Reviewer** | Checks quality, compliance, and correctness | "Review X against standards", "Audit Y for issues" |
| **Writer** | Creates documentation, communications, and records | "Document X", "Write the report for Y" |
| **Designer** | Designs experiences, interfaces, and structures | "Design the UX for X", "Create the layout for Y" |

### How Roles Map Across Domains

| Role | Software Project | Consulting Project | Research Project |
|------|-----------------|-------------------|-----------------|
| Orchestrator | Sprint coordination | Engagement management | Study coordination |
| Researcher | Tech spikes, API evaluation | Market analysis, interviews | Literature review |
| Planner | Systems architecture, solution design | Strategy design, engagement planning | Study design |
| Implementer | Writes code (Rust, Svelte, SQL) | Writes deliverables, runs workshops | Runs experiments |
| Reviewer | Code review, security audit, QA | Peer review, compliance check | Methodology review |
| Writer | Technical docs, ADRs, guides | Reports, proposals, presentations | Papers, findings |
| Designer | UI/UX, component design | Service design, process design | Framework design |

## Governance Concept Taxonomy

This decision also formalises the definitions of all governance concepts:

| Concept | Definition | Test |
|---------|-----------|------|
| **Agent** | A portable role you delegate work to. Has a distinct workflow and deliverable type. | "I need someone to do X" |
| **Skill** | Domain knowledge or methodology loaded into an agent's context. Shapes how work is done. | "The person doing X needs to know Y" |
| **Rule** | A constraint that must be followed. Binary: compliant or not. | "Anyone doing anything must follow Z" |
| **Hook** | An automated action triggered by an event. Mechanical enforcement. | "When event E happens, automatically do A" |
| **Lesson** | A learned pattern from experience. Promoted to rules/skills at recurrence threshold. | "We learned W the hard way" |

### Agent vs Skill Decision Framework

When deciding whether something should be an agent or a skill:

| Question | If Yes → | If No → |
|----------|----------|---------|
| Would you hire a different person for this? | Agent | Skill |
| Does it have a unique deliverable type? | Agent | Skill |
| Does it work across any project domain? | Agent | Skill |
| Is it a lens or methodology applied to existing work? | Skill | Agent |
| Can it be loaded by multiple different roles? | Skill | Agent |

## Consequences

1. **Agent count drops from 16 to 7** — simpler, more portable
2. **Skill count increases** — domain knowledge that was in agents moves to skills
3. **Project onboarding becomes skill-based** — a new project declares its domain skills, not its agent team
4. **The three-tier skill loading model [AD-53e80192](AD-53e80192) remains valid** — Tier 1 portable skills on role definitions, Tier 2 project skills injected by orchestrator
5. **Existing rules about agent delegation remain valid** — just with fewer, broader roles
6. **Claude Code agent types** — the mapping from universal roles to Claude Code's `subagent_type` parameter needs documenting (e.g., Implementer with backend skills → `Backend Engineer` subagent type)

## What Does NOT Change

- The orchestrator's coordination role
- The documentation-first workflow
- The learning loop (lessons → rules → skills)
- The three-tier skill loading model
- Rules and hooks — these are independent of agent identity
- The `.orqa/` directory structure

## Open Questions

1. Should Reviewer be split into "Quality Reviewer" and "Domain Reviewer"?
   Probably not — skills differentiate the lens.
2. Should Researcher and Planner be merged?
   They often work together but have distinct deliverables (findings vs plans).
3. How does this interact with Claude Code's fixed subagent types?
   The orchestrator maps universal roles + loaded skills to the closest
   Claude Code subagent_type at delegation time.
