## Open Questions

### Q1: Codebase Scanning Strategy

**Question:** When a user opens a new project in OrqaStudio, how does it understand the codebase structure?

**What needs to be detected:**
- Languages and frameworks in use
- Project structure (monorepo, single service, library)
- Test setup (test runner, coverage tool, test directory structure)
- Build system (cargo, npm, make, etc.)
- CI/CD configuration
- Existing governance artifacts (docs, rules, agents if already present)
- Branding and design system (color tokens, typography, theme configuration)

**Verdict: Three-tier hybrid approach (heuristics-first, Claude-augmented).**

#### Tier 1: Manifest-File Heuristics (Instant, Automatic)

The presence of specific files at the project root is an extraordinarily reliable signal. This covers 90%+ of real-world projects in <100ms.

**Key manifest detections:**

| File | Language | Package Manager | Framework Hints |
|------|----------|----------------|-----------------|
| `Cargo.toml` | Rust | cargo | Check `[dependencies]` for `actix-web`, `axum`, `tauri`, `rocket` |
| `Cargo.toml` + `[workspace]` | Rust | cargo workspace | Monorepo structure |
| `package.json` | JavaScript/TypeScript | npm/yarn/pnpm | Check `dependencies` for `react`, `svelte`, `vue`, `next` |
| `tsconfig.json` | TypeScript | ŌĆö | ŌĆö |
| `pyproject.toml` | Python | pip/poetry/uv | Check `[tool.poetry]`, `[project]` |
| `go.mod` | Go | go modules | Check for `gin`, `echo`, `fiber` |
| `Gemfile` | Ruby | bundler | Check for `rails`, `sinatra` |
| `*.sln` / `*.csproj` | C# | NuGet | Check for `Microsoft.AspNetCore` |
| `pom.xml` | Java | Maven | Check for `spring-boot` |
| `build.gradle` / `build.gradle.kts` | Java/Kotlin | Gradle | ŌĆö |
| `mix.exs` | Elixir | hex | Check for `phoenix` |
| `pubspec.yaml` | Dart | pub | Check for `flutter` |

**Project structure detection:**

| Signal | Meaning |
|--------|---------|
| `Cargo.toml` with `[workspace]` | Rust workspace (monorepo) |
| `package.json` with `"workspaces"` | JS/TS monorepo |
| `pnpm-workspace.yaml` | pnpm monorepo |
| `lerna.json` or `nx.json` or `turbo.json` | JS monorepo tooling |
| `backend/src-tauri/` directory | Tauri app (Rust + web frontend) |

**Test setup detection:**

| File/Dir | Test Runner |
|----------|------------|
| `vitest.config.*` | Vitest |
| `jest.config.*` | Jest |
| `playwright.config.*` | Playwright |
| `.pytest.ini` / `conftest.py` | pytest |

**CI/CD detection:**

| File/Dir | CI System |
|----------|-----------|
| `.github/workflows/*.yml` | GitHub Actions |
| `.gitlab-ci.yml` | GitLab CI |
| `Jenkinsfile` | Jenkins |
| `.circleci/config.yml` | CircleCI |

**Design system / branding detection:**

| File | What to Extract |
|------|----------------|
| `tailwind.config.*` | `theme.extend.colors`, `theme.extend.fontFamily`, custom spacing, dark mode strategy |
| `src/styles/tokens.*` or `src/theme.*` | CSS custom properties (`--color-*`, `--font-*`), design token definitions |
| `:root` in any `.css` file | CSS custom properties used as design tokens |
| `package.json` dependencies | Presence of `@radix-ui/*`, `shadcn`, `chakra-ui`, `mantine` etc. indicates component library conventions |
| `.storybook/` | Storybook config may reference theme providers |
| `figma.config.*` or `.figmarc` | Figma integration tokens |
| Brand assets directory (`public/`, `assets/`, `static/`) | Logo files, favicon, brand color usage in SVGs |

OrqaStudio uses extracted design tokens to theme its own UI when a project is open, making OrqaStudio feel like a native companion to the project. Extraction is best-effort ŌĆö missing tokens fall back to OrqaStudio's default theme.

**Governance artifact detection:**

| File/Dir | Meaning |
|----------|---------|
| `.claude/` | Claude Code governance (OrqaStudio-compatible) |
| `.claude/CLAUDE.md` | Main orchestrator instructions |
| `.orqa/agents/` | Agent definitions |
| `.orqa/rules/` | Governance rules |
| `.orqa/skills/` | Skill definitions |
| `.orqa/hooks/` | Hook scripts |

**Implementation in Rust:** A handful of `Path::exists()` calls and a few file reads, completing in single-digit milliseconds. Parse manifest dependency lists (with `toml`, `serde_json`, `serde_yaml`) for framework detection ŌĆö more reliable than tree-sitter because frameworks are declared, not inferred.

#### Tier 2: Language Detection with Hyperpolyglot (Fast, Automatic)

The [`hyperpolyglot`](https://crates.io/crates/hyperpolyglot) crate (pure Rust, based on GitHub Linguist heuristics) provides language breakdown across the full source tree. ~38x faster than Linguist. Returns a `HashMap<&str, Vec<(Detection, PathBuf)>>` for directory breakdown.

**When it adds value:** Projects with many languages or unusual file types. For most projects, manifest-based heuristics are sufficient. Hyperpolyglot catches languages not represented by manifest files (shell scripts, SQL files, config languages).

**Limitation:** Detects languages, not frameworks. "TypeScript" not "React" or "Svelte".

Triggered automatically after Tier 1. Results update the UI asynchronously (~1-2 seconds).

#### Tier 3: Claude Analysis (On-Demand)

Send file tree + manifests + README to Claude for deep analysis. Triggered by user action ("Analyze this project") or as part of the onboarding wizard.

**What Claude adds over heuristics:**
- Understands what the project does (not just what language it uses)
- Detects architectural patterns (hexagonal, event sourcing, CQRS)
- Identifies coding conventions and style
- Understands relationships between components in a monorepo
- Suggests appropriate governance artifacts based on project type

**Token cost:** Under $0.10 for most projects. A file tree + 5 sample files Ōēł 10K-20K tokens.

**What other tools do:**

| Tool | Approach |
|------|----------|
| VS Code | Extension-based per-language detection |
| IntelliJ | Scans for build files, deep framework support |
| Cursor | Full codebase embedding index via Merkle trees |
| GitHub Linguist | Filename, extension, heuristics, Bayesian classifier |

**Decision:** Three-tier hybrid. Tier 1 (manifest heuristics) runs on project open in <100ms. Tier 2 (hyperpolyglot) runs async in ~1-2s. Tier 3 (Claude) on user request. ŌåÆ AD needed.

**Crates needed:** `toml`, `serde_json`, `serde_yaml`, `hyperpolyglot`, `globset`, `ignore`

### Q2: Governance Framework Format

**Question:** Should OrqaStudio adopt the exact `.claude/` format (compatible with Claude Code CLI) or define its own more structured format?

**Verdict: Option C ŌĆö `.claude/` files on disk + structured metadata in SQLite.**

#### Option A: Exact `.claude/` format only

Parse `.claude/` files as-is. YAML frontmatter + markdown body for agents/skills. Pure markdown for rules. JSON for settings.

**Parsing in Rust:**
- [`yaml-front-matter`](https://crates.io/crates/yaml-front-matter) ŌĆö Splits `---` delimited frontmatter, deserializes into a serde struct. Returns `Document<T>` with `.metadata` and `.content`.
- [`gray_matter`](https://crates.io/crates/gray_matter) ŌĆö Rust port of JavaScript `gray-matter`. Supports YAML, TOML, JSON frontmatter.
- [`comrak`](https://github.com/kivikakk/comrak) ŌĆö Full CommonMark + GFM parser (used by crates.io, docs.rs, GitLab). Has `front_matter_delimiter` option. Full AST for extracting structured sections.

**Pros:** Full CLI compatibility. Zero migration friction. Git-friendly.
**Cons:** Cannot store OrqaStudio-specific metadata without modifying files.

#### Option B: OrqaStudio-specific structured format

JSON or YAML schema for each artifact type.

**Verdict:** Not recommended. Breaks CLI compatibility completely. Markdown body contains nuanced instructions that don't reduce to structured fields. Migration cost is high.

#### Option C: `.claude/` on disk + SQLite metadata (RECOMMENDED)

Preserves full compatibility while giving OrqaStudio structured data.

```
.claude/                    ŌåÉ Source of truth (git-committed, CLI-compatible)
  agents/backend-engineer.md
  rules/coding-standards.md
  settings.json

orqa.db (SQLite)           ŌåÉ Derived/enriched data (gitignored)
  agents table              ŌåÉ Parsed frontmatter + extracted metadata
  rules table               ŌåÉ Title, body, extracted constraints
  artifacts table           ŌåÉ All governance artifacts indexed
```

**Keeping DB in sync:** Use the [`notify`](https://crates.io/crates/notify) crate (cross-platform filesystem watcher) to watch `.claude/`. On file change: re-parse, update DB row, emit Tauri event to update UI.

**Conflict resolution:** Files are always authoritative. DB is a cache/index. SHA-256 hash comparison to detect actual content changes vs. metadata-only file touches.

**OrqaStudio-only metadata in SQLite (not in files):**

| Column | Purpose |
|--------|---------|
| `parsed_at` | When OrqaStudio last parsed the file |
| `file_hash` | SHA-256 for change detection |
| `extracted_tools` | Tools list from frontmatter |
| `compliance_status` | Does this artifact align with current codebase? |
| `last_used_at` | When last invoked in a session |
| `usage_count` | How many times invoked |

#### Option D: `.claude/` with conventions

Add comment markers for structured sections. **Verdict:** Not recommended. Fragile, clutters files, worse version of both A and C.

**What other tools use:**

| Tool | Format | Why |
|------|--------|-----|
| Claude Code CLI | Markdown + YAML frontmatter + JSON | Content IS instructions for an LLM ŌĆö markdown is natural |
| Cursor | `.cursor/rules/` (markdown) | Same pattern as Claude Code |
| Kubernetes | YAML + JSON Schema | Declarative infrastructure config (different domain) |

Tools that configure AI behavior use markdown because the content is natural language instructions. OrqaStudio's governance artifacts are AI instructions, so markdown is the right format.

**Decision:** Option C. Parse `.claude/` with `yaml-front-matter` + `comrak`. Store enriched data in SQLite. Watch with `notify` for live sync. Files are authoritative; DB is derived. `orqa.db` goes in `.gitignore`. ŌåÆ Extends persistence AD.

**Crates needed:** `yaml-front-matter`, `comrak`, `notify`, `sha2`

### Q3: Progressive Disclosure

**Question:** How should new users be introduced to OrqaStudio's governance features?

**Verdict: Conversation-first with organic feature introduction. Value in under 1 minute.**

#### Core Principle

OrqaStudio's target user is a PM/Tech Lead, not a developer in a code editor. The first experience should feel like opening a project management tool, not configuring an IDE. **Show the value before showing the machinery.**

#### Recommended First-Run Flow

**Step 0: API Key Setup (Gate)**

Single, focused screen. No sidebar, no panels, no navigation. Just:
- "Connect to Claude" headline
- API key input field
- "Verify & Continue" button
- Link to get an API key

This is the only mandatory "wizard" step.

**Step 1: Project Selection (Immediate)**

- "Open a Project" button (native file picker)
- Recent projects list (empty on first run, populated after)

No empty dashboard. No 15 empty panels. Just an action.

**Step 2: Project Scan (Automatic, <3 seconds)**

When the user opens a directory, Tier 1 + Tier 2 heuristic scan runs automatically:

```
Scanning project...
  Found: Rust (Cargo.toml), TypeScript (package.json)
  Structure: Tauri desktop app
  Build: cargo + npm
  Tests: cargo test + Vitest
  CI: GitHub Actions
  Governance: .claude/ found (15 agents, 18 rules, 1 skill)
```

**Step 3: The Conversation (Primary Interface)**

User lands on the conversation view. Pre-populated system message summarizes the scan and suggests what the user can do. The user types. OrqaStudio responds. Governance features appear as they become relevant.

**Step 4: Feature Introduction (Organic, Not Wizard)**

Features reveal themselves through conversation:

| User Action | Feature Revealed |
|-------------|-----------------|
| "Show me my agents" | Agent browser panel slides in |
| "Create a new rule" | Rule editor opens inline |
| "How compliant is the codebase?" | Scanner dashboard appears |
| "Start implementing the login feature" | Task pipeline view opens |
| Completes first implementation cycle | Retrospective prompt appears |
| Uses 3+ sessions | Session history becomes visible in sidebar |

#### Progressive Navigation Design

Navigation grows as the user engages:

**Day 1:** `[Conversation]` only
**After first governance interaction:** + `[Artifacts]`
**After first implementation cycle:** + `[Sessions]`
**Power user:** `[Conversation] [Artifacts (Agents, Rules, Skills, Hooks)] [Sessions] [Scanners] [Metrics] [Settings]`

#### Feature Gates Implementation

```sql
CREATE TABLE feature_gates (
    feature_id TEXT PRIMARY KEY,
    first_used_at TEXT,          -- NULL if never used
    use_count INTEGER DEFAULT 0,
    visible INTEGER DEFAULT 0    -- 0 = hidden, 1 = visible in nav
);
```

When a feature is first triggered, set `visible = 1`. Frontend conditionally renders nav items based on gate state.

#### Empty State Design

When a panel is first opened but has no content, the empty state should be **actionable**, not decorative:

- Explain what the feature IS (one sentence)
- Explain what it DOES (one sentence)
- Primary action button
- Secondary "easy start" option (template, import, auto-generate)

#### PM/Tech Lead Persona Considerations

Use the technical terms (agent, rule, skill, hook) for `.claude/` compatibility, but always with brief contextual explanations the first time they appear. The PM should be able to ask governance questions in natural language and get meaningful answers backed by parsed artifacts.

#### Minimum Viable First-Run Experience

1. Enter API key (30 seconds)
2. Select project directory (10 seconds)
3. See project scan results (2 seconds, automatic)
4. Land in conversation with Claude (immediate)
5. Ask "What is the state of governance in this project?" ŌåÆ meaningful answer

**Total time to value: under 1 minute.**

#### Anti-Patterns to Avoid

1. **Feature tour on first launch** ŌĆö Users skip it and learn nothing
2. **Dashboard with empty cards** ŌĆö "0 agents, 0 rules, 0 scans" is demoralizing
3. **Mandatory wizard** ŌĆö "Step 1 of 5: Configure your governance framework" is premature
4. **Jargon without context** ŌĆö "Create an Agent TOML" means nothing to a PM on day one
5. **Everything visible at once** ŌĆö 15 nav items on first load creates cognitive overload

**What other tools do well:**

| Tool | Pattern | Lesson for OrqaStudio |
|------|---------|-----------------|
| Linear | Minimal setup, then immediately into the issue board | Speed to value |
| Notion | Empty workspace with a single page. Templates available but not pushed. | Blank canvas approach |
| Obsidian | "Start here" note. Plugins discoverable but not required. | Explain-by-doing |
| VS Code | Welcome tab, extensions suggested based on detected languages | Context-aware suggestions |

**Decision:** Conversation-first progressive disclosure. No wizard. Features appear as they become relevant. Feature gate table in SQLite controls navigation visibility. ŌåÆ Informs Phase 0d UX design.
