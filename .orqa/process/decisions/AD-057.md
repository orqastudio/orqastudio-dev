# AD-05756de7: Artifact IDs — Type-Prefixed 8-Char Hex

## Decision

All artifact IDs use the format `TYPE-XXXXXXXX` where:
- `TYPE` is the artifact type prefix (SKILL, TASK, EPIC, RULE, etc.)
- `XXXXXXXX` is 8 lowercase hexadecimal characters, randomly generated

Example: `SKILL-a7f3b2c1`, `TASK-9e4d1f08`, `EPIC-c3a82b7e`

## Context

The previous ID scheme used sequential numbers (`SKILL-1c708b68`, `TASK-367f0026`) with optional location prefixes (`SKILL-SVE-89d35141`, `SKILL-CORE-005`). This caused two problems:

1. **Location coupling** — IDs encoded where an artifact lived (SVE = svelte plugin, CORE = platform). Moving an artifact between core, plugins, or projects required renaming the ID and updating every reference across the entire graph.

2. **Collision risk in plugins** — without location prefixes, two independently developed plugins could both create `SKILL-30a419dd`. With location prefixes, the collision problem was solved but the rename cascade problem was introduced.

## Why 8-Char Hex

- **4.3 billion possible values per type prefix** — collision probability is 0.001% at 10,000 artifacts of the same type, effectively zero for any realistic project
- **Follows git's precedent** — git short hashes use 7 hex chars and handle millions of objects; 8 chars gives additional margin
- **Human-typeable** — `SKILL-a7f3b2c1` is compact enough for conversation and prose references
- **Type prefix preserved** — you still know it's a SKILL, TASK, or EPIC at a glance
- **Location-independent** — the ID doesn't change when a skill moves from core to a plugin or between projects
- **No coordination required** — plugins can generate IDs independently without a central registry

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| Sequential (SKILL-1c708b68) | Human-readable, orderable | Collisions across plugins, no location independence |
| Location-prefixed (SKILL-SVE-030) | No collisions | Rename cascades when moving, encodes location in identity |
| Full UUIDs (550e8400-e29b-...) | Zero collisions | Unreadable, untypeable, clutters prose |
| 6-char hex (SKILL-a7f3b2) | Compact | 3% collision probability at 1,000 artifacts |
| **8-char hex (SKILL-a7f3b2c1)** | **Compact, safe, git-like** | **None significant** |

## ID Generation

IDs are generated by:
1. Taking the type prefix from the artifact type (SKILL, TASK, EPIC, etc.)
2. Generating 8 random hex characters
3. Combining as `TYPE-XXXXXXXX`

In Rust: `format!("{}-{:08x}", prefix, rand::random::<u32>())`
In TypeScript: `` `${prefix}-${crypto.randomBytes(4).toString('hex')}` ``
In shell: `` `${PREFIX}-$(openssl rand -hex 4)` ``

## Migration

Existing sequential IDs remain valid indefinitely. New artifacts use the hex format. A bulk migration is not required — IDs can be migrated opportunistically or via a one-time script when ready. The artifact scanner and graph builder accept both formats.

## Constraints

- The type prefix MUST match the artifact's actual type (enforced by integrity checks)
- IDs are case-insensitive for matching but stored lowercase
- The hex portion MUST be exactly 8 characters
- Existing sequential IDs (SKILL-1c708b68, TASK-367f0026) remain valid — no breaking change
