## Key Findings

### Three independent implementations, no single source of truth

| Location | Checks | Runs when |
|----------|--------|-----------|
| Rust scanner (artifact_graph.rs) | 10 graph-level checks | App runtime, on-demand |
| CLI tools (verify-links.mjs, verify-pipeline-integrity.mjs) | ~15 checks | `make verify`, pre-commit |
| Pre-commit validators (.githooks/validate-*.sh/mjs) | ~12 checks | Every commit |

### Duplicated checks (different implementations)
- Broken links: Rust + verify-links.mjs (different field coverage)
- Missing inverses: Rust + verify-links.mjs (same logic, different code)
- Task dependencies: Rust + validate-task-deps.sh

### Checks only in Rust (no CLI equivalent)
- CircularDependency, SupersessionSymmetry, MilestoneGate, IdeaPromotionValidity, IdeaDeliveryTracking, PlanningPlacement, ResearchGap

### Checks only in CLI (no Rust equivalent)
- Status transition validation, pillar alignment, stub scanning, lint suppression audit, historical preservation, epic readiness gate, schema validation, deprecated field detection

### Resolution: standalone library
Extract all checks into a single Node/TypeScript library that:
- Takes project config (.orqa/project.json) and paths as input
- Runs all checks against the artifact graph
- Returns structured results (category, severity, artifact, message, auto-fixable)
- Is consumed by: pre-commit hook, `make verify`, CLI tools, plugins, and eventually the app
- Replaces both verify-links.mjs and verify-pipeline-integrity.mjs
- The Rust scanner delegates to or mirrors this library's check list
