## Problem

The artifact system currently has no unified way to query relationships between artifacts. The frontend accesses artifact data through multiple ad-hoc patterns:

1. **NavTree** — built by `artifact_scan_tree`, provides label/path/status/description for sidebar rendering
2. **File reading** — `read_artifact` returns raw markdown content for the viewer
3. **ArtifactLink** — resolves IDs via a hardcoded prefix map in the frontend (fragile, incomplete)
4. **No backreferences** — "what links to [EPIC-e045ab6d](EPIC-e045ab6d)?" requires scanning every artifact file

This fragmentation means every new feature (plugin, visualization, search) must reinvent artifact access. Plugins can't discover relationships. The sidebar, viewer, and linking system all use different data paths.

## Vision

A single artifact graph that:

1. **Knows every artifact** — ID, path, type, frontmatter metadata
2. **Knows every relationship** — forward refs (this → that) and backrefs (that → this), computed bidirectionally
3. **Is queryable from the frontend** via a typed SDK with a clean API
4. **Is the single source of truth** for all artifact rendering — sidebar, viewer, links, and future plugins
5. **Stays live** via file watcher — no manual rescan needed

## Design

### Backend: Artifact Node Graph

```rust
pub struct ArtifactGraph {
    nodes: HashMap<String, ArtifactNode>,    // keyed by artifact ID
    path_index: HashMap<String, String>,     // path → ID (reverse lookup)
}

pub struct ArtifactNode {
    pub id: String,                          // "EPIC-e045ab6d"
    pub path: String,                        // ".orqa/delivery/epics/EPIC-e045ab6d.md"
    pub artifact_type: String,               // "epic"
    pub title: String,                       // from frontmatter
    pub description: Option<String>,         // from frontmatter
    pub status: Option<String>,              // from frontmatter
    pub frontmatter: serde_json::Value,      // full frontmatter as JSON (for plugins)
    pub references_out: Vec<ArtifactRef>,    // what this node links to
    pub references_in: Vec<ArtifactRef>,     // what links to this node (computed)
}

pub struct ArtifactRef {
    pub target_id: String,                   // "MS-654badde"
    pub field: String,                       // "milestone" (which frontmatter field)
    pub source_id: String,                   // "EPIC-e045ab6d" (who references)
}
```

**Graph construction** happens during `artifact_scan_tree`:

1. First pass: scan all `.md` files, extract frontmatter, create `ArtifactNode` with `references_out`
2. Second pass: invert all `references_out` to populate `references_in` on target nodes
3. Store graph in `AppState` alongside the existing NavTree

**Reference extraction** parses known link fields from frontmatter:
- Single ID fields: `milestone`, `epic`, `evolves-into`, `evolves-into`, `surpassed-by`, `evolves-into`, `evolves-from`
- Array ID fields: `depends-on`, `blocks`, `pillars`, `research-refs`
- Path fields: `docs-required`, `docs-produced` (resolved via `path_index`)

Files without an `id` frontmatter field (documentation pages, READMEs) are indexed by path only — they appear in `path_index` but not in `nodes`. They can still be targets of `docs-required`/`docs-produced` references.

### Tauri Commands

```rust
// Core resolution
resolve_artifact(id: String) -> Option<ArtifactNode>
resolve_path(path: String) -> Option<ArtifactNode>

// Relationship queries
get_references_from(id: String) -> Vec<ArtifactRef>    // what does this artifact link to?
get_references_to(id: String) -> Vec<ArtifactRef>      // what links to this artifact?

// Bulk queries
get_artifacts_by_type(artifact_type: String) -> Vec<ArtifactNode>
get_artifact_children(id: String) -> Vec<ArtifactNode>  // e.g., tasks within an epic

// Content
read_artifact_content(path: String) -> String           // raw markdown body (below frontmatter)

// Graph metadata
get_graph_stats() -> GraphStats                         // node count, edge count, orphans, broken refs
```

### Frontend SDK: `artifactGraph`

A typed Svelte 5 rune store that wraps the Tauri commands and provides reactive access:

```typescript
// ui/src/lib/sdk/artifact-graph.svelte.ts

class ArtifactGraphSDK {
    // -- Reactive state --
    graph = $state<Map<string, ArtifactNode>>(new Map());
    loading = $state(false);
    lastRefresh = $state<Date | null>(null);

    // -- Resolution --
    resolve(id: string): ArtifactNode | undefined
    resolveByPath(path: string): ArtifactNode | undefined

    // -- Relationships --
    referencesFrom(id: string): ArtifactRef[]
    referencesTo(id: string): ArtifactRef[]
    children(id: string, type?: string): ArtifactNode[]    // tasks of an epic, epics of a milestone

    // -- Bulk queries --
    byType(type: string): ArtifactNode[]
    byStatus(status: string): ArtifactNode[]

    // -- Content --
    async readContent(path: string): string                // markdown body

    // -- Graph health --
    brokenRefs(): ArtifactRef[]                            // refs where target doesn't exist
    orphans(): ArtifactNode[]                              // nodes with no incoming refs

    // -- Lifecycle --
    async refresh(): void                                  // manual refresh
    // Auto-refreshes via file watcher events from backend

    // -- Subscriptions (plugin API) --
    subscribe(id: string, callback: (node: ArtifactNode | undefined) => void): () => void
    subscribeType(type: string, callback: (nodes: ArtifactNode[]) => void): () => void
    // Returns unsubscribe function. Callbacks fire on graph refresh when the watched node/type changes.
}

export const artifactGraph = new ArtifactGraphSDK();
```

**Key design decisions:**

1. **Eagerly loaded** — the full graph is small (hundreds of nodes, not millions). Load it all on app start, keep it in memory, refresh on watcher events.
2. **Synchronous reads** — `resolve()`, `referencesTo()`, etc. are synchronous lookups into the in-memory graph. No IPC round-trip for navigation.
3. **Async content** — `readContent()` is the only async method, since it reads file content from disk on demand.
4. **File watcher events** — backend emits a Tauri event when the graph is rebuilt. The SDK listens and refreshes its local copy.

### Migration Path: What the SDK Replaces

| Current Pattern | Replaced By |
|----------------|-------------|
| `ARTIFACT_PREFIX_MAP` in NavigationStore | `artifactGraph.resolve(id)` |
| `pendingArtifactId` + `label.startsWith()` | `artifactGraph.resolve(id).path` → `navigateToPath()` |
| `ArtifactLink` hardcoded prefix routing | `artifactGraph.resolve(id)` for navigation |
| `read_artifact` Tauri command (raw file read) | `artifactGraph.readContent(path)` for body, `artifactGraph.resolve(id)` for metadata |
| `artifact_scan_tree` as the only data source | NavTree for sidebar shape, `artifactGraph` for everything else |
| Ad-hoc frontmatter parsing in components | `node.frontmatter` from the graph (already parsed) |

### NavTree Relationship

The existing `NavTree` structure (groups → types → nodes) serves a different purpose: it defines the **sidebar shape** — how artifacts are grouped and ordered for navigation. The artifact graph does NOT replace the NavTree. Instead:

- **NavTree** = UI structure (groups, ordering, icons, labels for the sidebar)
- **ArtifactGraph** = data structure (nodes, edges, metadata, relationships)
- `DocNode.id` field (added in this epic) connects the two: sidebar items link to graph nodes

The sidebar continues to use NavTree for rendering. When a user clicks an item, the component looks up `artifactGraph.resolve(node.id)` for full metadata and relationships.

### Plugin Foundation

The SDK is designed to be the primary artifact API for plugins:

```typescript
// Plugin accessing the artifact graph
import { artifactGraph } from '@orqa/sdk';

// Find all lessons that reference a specific rule
const ruleRefs = artifactGraph.referencesTo('RULE-5e03e67b');
const lessons = ruleRefs
    .filter(ref => ref.source_id.startsWith('IMPL-'))
    .map(ref => artifactGraph.resolve(ref.source_id));

// Find orphaned epics (no milestone reference)
const orphanedEpics = artifactGraph.byType('epic')
    .filter(e => !e.frontmatter.milestone);

// Get the full dependency chain for a task
function getDependencyChain(taskId: string): ArtifactNode[] {
    const node = artifactGraph.resolve(taskId);
    const deps = (node?.frontmatter['depends-on'] || []) as string[];
    return deps.flatMap(id => [artifactGraph.resolve(id)!, ...getDependencyChain(id)]);
}
```

Plugins get the same API the core app uses. No special plugin SDK — just the artifact graph.

### File Watcher Integration

```
.orqa/ file change detected (Tauri fs watcher)
  → Debounce (500ms)
  → Rebuild ArtifactGraph in backend
  → Emit "artifact-graph-updated" Tauri event
  → Frontend SDK receives event
  → SDK refreshes local graph copy
  → All $derived values in components reactively update
```

The watcher covers create, modify, delete, and rename events within `.orqa/`. Changes outside `.orqa/` are ignored.

### Sidebar and Viewer Migration

**Sidebar (`ArtifactNav`):**
- Currently renders `DocNode[]` from NavTree
- After: still renders NavTree for shape, but enriches with graph data where needed (e.g., backreference count badges)
- `DocNode.id` links to `artifactGraph.resolve(id)` for metadata

**Viewer (`ArtifactViewer`):**
- Currently reads raw markdown via `read_artifact` and parses frontmatter in the frontend
- After: uses `artifactGraph.resolve(id)` for metadata (already parsed), `artifactGraph.readContent(path)` for the markdown body
- Relationship sidebar: "Referenced by" and "References" sections powered by `referencesTo()` and `referencesFrom()`

**FrontmatterHeader:**
- Currently renders fields with hardcoded field classification
- After: `ArtifactLink` components use `artifactGraph.resolve(id)` for navigation
- Broken links detected by checking `artifactGraph.resolve(id) === undefined`

## Resolved Questions

- **Non-.orqa files**: No — graph covers `.orqa/` artifacts only. Source code files referenced in task `scope` fields are informational, not navigable nodes. Future enhancement if needed.
- **Content caching**: Always read from disk. Simple, always fresh, no cache invalidation complexity. The file watcher handles freshness of metadata in the graph; content is read on demand.
- **Watcher event payload**: Full graph snapshot. The graph is small (hundreds of nodes) and full replacement is simpler than incremental diffing. Note: when the graph expands to include the full codebase in the future, this will need to change to incremental updates.
- **Plugin subscriptions**: Yes — the SDK exposes a `subscribe(id, callback)` pattern for plugins that want to watch specific artifacts. This complements Svelte reactivity for non-Svelte plugin contexts.
