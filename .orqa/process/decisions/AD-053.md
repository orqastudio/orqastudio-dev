## Decision

### Canonical Statuses (App-Level)

These represent universal stages of thinking. They exist in every project. The underlying concept is fixed; the display label is aliasable per project.

| Status | Meaning |
|---|---|
| `captured` | Recorded, not yet explored |
| `exploring` | Under investigation |
| `ready` | Explored, awaiting prioritisation |
| `prioritised` | Chosen for action |
| `active` | Work in progress |
| `hold` | Paused deliberately |
| `blocked` | Can't proceed (inferred from dependencies, or set manually for external blockers) |
| `review` | Done, awaiting validation |
| `completed` | Finished |
| `surpassed` | Superseded by newer thinking — something better exists |
| `archived` | Explored but didn't lead anywhere — preserved but inactive |
| `recurring` | Completed but expected to repeat |

### `surpassed` vs `archived`

Different meanings, different semantics:
- **`surpassed`**: Thinking evolved. A merged idea is surpassed — the `merged-into` relationship tells you what replaced it. Outdated research is surpassed — something newer exists.
- **`archived`**: Thinking was explored but didn't go anywhere. No successor exists. The idea was valid but the team decided not to pursue it.

### Status Subsets Per Type

Each type declares which statuses it allows. The app enforces the restriction — you can't set a status that isn't in the type's allowed subset.

A task doesn't need `exploring`. A milestone doesn't need `recurring`. Core types get sensible defaults. Project types declare their own subsets.

### Display Aliases

Projects configure how statuses appear in the UI. The canonical key is always stored; the alias is a rendering concern.

A research project might display `exploring` as "Investigating" and `active` as "In Lab." The app stores `exploring` and `active`. The UI renders the aliases.

### Project Statuses (Project-Level)

If delivery or discovery types need statuses beyond the canonical vocabulary, they define them in `project.json`. These are scoped to the types that use them and don't pollute the canonical vocabulary.

### Transitions and Auto-Rules

Auto-rules are graph queries that can reference both canonical and project relationships:

```json
{
  "condition": "all-related-in-status",
  "relationship": "delivers",
  "status": "completed",
  "target": "review"
}
```

A project might define: "when all `depends-on` targets are `completed` → auto-unblock this artifact."

## Rationale

One canonical vocabulary prevents fragmentation — every project speaks the same language about thinking stages. Subsets prevent nonsensical states. Aliases give projects domain-appropriate language without losing canonical semantics.

Separating `surpassed` from `archived` preserves provenance. `surpassed` means the graph has the successor. `archived` means the thinking stopped.

## Consequences

- 12 canonical statuses (added `archived` to the vocabulary)
- Type definitions include `allowed_statuses` arrays
- `project.json` supports `status_aliases` for display configuration
- App rejects status changes that violate type subset restrictions
- Project types can define additional statuses beyond canonical
- Auto-rules reference both canonical and project relationships
