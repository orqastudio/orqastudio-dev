**Date:** 2026-03-02 | **Status:** Complete

Research into how OrqaStudio stores, maps, and applies design tokens at runtime — both OrqaStudio's own baseline theme and per-project themes extracted from opened projects.

---


## Context

OrqaStudio needs two design token capabilities:

1. **OrqaStudio's own design system** -- A set of default design tokens (colors, typography, spacing, dark/light mode) that define OrqaStudio's visual identity. This is the baseline theme that ships with the application.

2. **Per-project design token extraction and application** -- When a user opens a project in OrqaStudio, OrqaStudio scans for design tokens (tailwind.config.*, CSS custom properties in `:root`, component library conventions) and adapts its own UI to match the project's color palette. Each opened project should make OrqaStudio feel like a native companion to that project.

The scanning/detection side is covered in `docs/research/onboarding.md` (Tier 1 manifest heuristics -- specifically the "Design system / branding detection" table). This research focuses on what happens AFTER tokens are detected: how they are stored, mapped to OrqaStudio's internal schema, and applied to the UI at runtime.

### Constraints

- **Tauri v2 desktop app** -- No server. All processing happens locally.
- **Svelte 5 + shadcn-svelte** -- The component library dictates the CSS variable naming convention.
- **SQLite for persistence** -- `orqa.db` stores project metadata and derived data.
- **`notify` file watcher** -- Already planned for `.claude/` file watching; can also watch design token source files.
- **OKLCH color space** -- shadcn-svelte's current version uses OKLCH for all CSS variables.

## Q1: Design Token Format & Storage

**Question:** What format should OrqaStudio store design tokens internally? How do popular standards work? What is the mapping layer between extracted project tokens and OrqaStudio's internal token schema?

### OrqaStudio's Internal Token Schema: shadcn-svelte's CSS Variable Convention

### External Standards Evaluated

#### W3C Design Tokens (DTCG) Specification

The Design Tokens Community Group released the first stable specification (2025.10) in October 2025. The format is JSON with `$`-prefixed properties:

```json
{
  "colors": {
    "$type": "color",
    "primary": {
      "$value": "#1a1a2e",
      "$description": "Main brand color"
    },
    "accent": {
      "$value": "{colors.primary}",
      "$description": "Alias to primary"
    }
  }
}
```

Key features: `$type` for categorization, `$value` for the token value, `$description` for documentation, alias syntax `{group.token}` for references, `$extensions` for vendor metadata. File extension: `.tokens` or `.tokens.json`. MIME type: `application/design-tokens+json`.

**Relevance to OrqaStudio:** The DTCG spec is a vendor-neutral interchange format. OrqaStudio does not need to adopt it as its internal representation, but understanding it helps if a project ships `.tokens` files. This is a future detection target for Tier 1 heuristics.

#### Style Dictionary

Style Dictionary is a build system that transforms design tokens (stored as JSON/YAML) into platform-specific outputs (CSS custom properties, iOS, Android, etc.). It is forward-compatible with the DTCG spec. Style Dictionary would be overkill for OrqaStudio's use case -- OrqaStudio does not need to output tokens to multiple platforms. OrqaStudio only needs to consume tokens and map them to its CSS variables.

#### Tailwind Theme Format

Tailwind v3 uses a JavaScript config object (`tailwind.config.js`) with `theme.colors`, `theme.extend.colors`, `theme.fontFamily`, etc. Tailwind v4 moves to CSS-first configuration with the `@theme` directive, exposing all design tokens as CSS custom properties on `:root` automatically (e.g., `--color-blue-500`, `--font-sans`).

**Relevance to OrqaStudio:** Many projects OrqaStudio will open use Tailwind. Extracting colors from `tailwind.config.*` (v3) or from the generated CSS output (v4) is a primary extraction target.

### Internal Storage Format

**Verdict: Store as a JSON object in SQLite, keyed by shadcn-svelte variable names, with values in OKLCH format.**

The internal storage format maps directly to the shadcn-svelte variable names. A project theme is a JSON object where each key is a CSS variable name (without `--` prefix) and each value is an OKLCH string:

```json
{
  "source": "tailwind.config.ts",
  "extracted_at": "2026-03-02T14:30:00.000Z",
  "source_hash": "a1b2c3d4...",
  "tokens": {
    "primary": "oklch(0.37 0.15 265)",
    "primary-foreground": "oklch(0.98 0.01 265)",
    "secondary": "oklch(0.92 0.03 265)",
    "secondary-foreground": "oklch(0.20 0.02 265)",
    "accent": "oklch(0.65 0.20 145)",
    "accent-foreground": "oklch(0.98 0.01 145)",
    "background": "oklch(0.99 0.005 265)",
    "foreground": "oklch(0.15 0.02 265)",
    "muted": "oklch(0.95 0.01 265)",
    "muted-foreground": "oklch(0.55 0.02 265)",
    "border": "oklch(0.90 0.01 265)",
    "input": "oklch(0.90 0.01 265)",
    "ring": "oklch(0.37 0.15 265)",
    "destructive": "oklch(0.577 0.245 27.325)"
  },
  "tokens_dark": {
    "primary": "oklch(0.75 0.15 265)",
    "primary-foreground": "oklch(0.15 0.02 265)",
    "background": "oklch(0.15 0.02 265)",
    "foreground": "oklch(0.98 0.01 265)"
  },
  "unmapped": {
    "tailwind.blue.500": "#3b82f6",
    "tailwind.blue.600": "#2563eb"
  },
  "user_overrides": {}
}
```

**Why this format:**

1. **Direct application** -- Keys map 1:1 to CSS variable names. Applying the theme is iterating the object and calling `document.documentElement.style.setProperty('--' + key, value)`.
2. **Partial themes** -- Not every project will provide every token. Missing keys fall back to OrqaStudio's defaults.
3. **OKLCH normalization** -- All extracted colors are converted to OKLCH before storage, matching shadcn-svelte's format. This avoids runtime conversion.
4. **Provenance tracking** -- `source` and `source_hash` enable cache invalidation when the source file changes.
5. **User overrides** -- A separate `user_overrides` object lets users tweak auto-extracted values without losing the original extraction.
6. **Unmapped values** -- Raw extracted values that could not be automatically mapped to a shadcn-svelte variable are preserved for future use or manual mapping.

### Fallback Chain

When OrqaStudio resolves a CSS variable value for a given project, it follows this chain:

```
User Override  >  Extracted Project Token  >  OrqaStudio Default Theme
```

In code, when applying a theme:

```typescript
function resolveToken(name: string, projectTheme: ProjectTheme | null): string {
  // 1. User override wins
  if (projectTheme?.user_overrides?.[name]) return projectTheme.user_overrides[name];
  // 2. Extracted token
  if (projectTheme?.tokens?.[name]) return projectTheme.tokens[name];
  // 3. OrqaStudio default (already in CSS from the stylesheet -- no action needed)
  return ''; // empty means "use the CSS default"
}
```

In practice, the fallback is handled by CSS itself: OrqaStudio's default theme is defined in the global stylesheet as `:root` variables. Project-specific overrides are applied via `document.documentElement.style.setProperty()`, which takes precedence over stylesheet rules. Removing a property (`.removeProperty()`) restores the default.

## Q2: Runtime Theme Application in Svelte 5

**Question:** How do you dynamically apply CSS custom properties at runtime in a Svelte 5 / Tauri app? How does dark mode work? What are the performance considerations?

### How shadcn-svelte Theming Works

shadcn-svelte's theming is built on three layers:

1. **CSS custom properties in `:root`** -- Define all color/spacing tokens as CSS variables in the global stylesheet.
2. **`.dark` class on `<html>`** -- Dark mode redefines the same CSS variables under a `.dark` selector.
3. **Tailwind v4 `@theme inline`** -- Registers the CSS variables with Tailwind so utility classes like `bg-primary` reference them.

Components use Tailwind classes that resolve to these variables:

```svelte
<button class="bg-primary text-primary-foreground hover:bg-primary/90">
  Click me
</button>
```

The `bg-primary` class resolves to `background-color: var(--primary)`. This means changing `--primary` on `:root` instantly changes every primary-colored element in the UI.

### Runtime Theme Switching: The Mechanism

To apply a project theme at runtime, OrqaStudio sets CSS custom properties on `document.documentElement`:

```typescript
// theme-manager.ts -- runs in the Svelte frontend

import type { ProjectTheme } from '$lib/types';

const ORQA_DEFAULTS = {
  'primary': 'oklch(0.205 0 0)',
  'primary-foreground': 'oklch(0.985 0 0)',
  // ... all shadcn-svelte defaults
};

export function applyProjectTheme(theme: ProjectTheme | null): void {
  const root = document.documentElement;

  if (!theme) {
    // Reset to OrqaStudio defaults: remove all inline overrides
    for (const key of Object.keys(ORQA_DEFAULTS)) {
      root.style.removeProperty(`--${key}`);
    }
    return;
  }

  // Apply resolved tokens (user overrides > extracted > defaults)
  for (const [key, defaultValue] of Object.entries(ORQA_DEFAULTS)) {
    const resolved = theme.user_overrides?.[key]
      ?? theme.tokens?.[key]
      ?? null;

    if (resolved) {
      root.style.setProperty(`--${key}`, resolved);
    } else {
      root.style.removeProperty(`--${key}`);
    }
  }

  // Apply dark mode variants if present
  // (handled separately via dark mode class -- see below)
}

export function clearProjectTheme(): void {
  applyProjectTheme(null);
}
```

**When does this run?**

1. **Project open** -- After Tier 1 scanning detects design tokens, the Rust backend extracts them, stores the result in SQLite, and emits a Tauri event (`theme-extracted`). The frontend listens and calls `applyProjectTheme()`.
2. **Project switch** -- When the user switches to a different project, `clearProjectTheme()` is called first, then the new project's theme is applied.
3. **User override** -- When the user customizes a token in the settings panel, the override is saved to SQLite and `applyProjectTheme()` is called with the updated theme.

### Dark/Light Mode Implementation

**Verdict: Use `mode-watcher` (same library shadcn-svelte recommends) with per-project dark mode token overrides.**

shadcn-svelte recommends [`mode-watcher`](https://github.com/svecosystem/mode-watcher) for dark mode management. It handles:

- System preference detection (`prefers-color-scheme`)
- Manual light/dark/system toggle
- Persistent preference storage (localStorage)
- Applying/removing the `.dark` class on `<html>`

```svelte
<!-- ui/src/routes/+layout.svelte -->
<script lang="ts">
  import '../app.css';
  import { ModeWatcher } from 'mode-watcher';
  let { children } = $props();
</script>

<ModeWatcher />
{@render children?.()}
```

Toggle component:

```svelte
<script lang="ts">
  import { toggleMode, setMode, resetMode } from 'mode-watcher';
  import Sun from 'lucide-svelte/icons/sun';
  import Moon from 'lucide-svelte/icons/moon';
  import Monitor from 'lucide-svelte/icons/monitor';
</script>

<button onclick={toggleMode}>
  <Sun class="dark:hidden" />
  <Moon class="hidden dark:block" />
</button>

<!-- Or a dropdown for light/dark/system -->
<select onchange={(e) => setMode(e.target.value)}>
  <option value="light">Light</option>
  <option value="dark">Dark</option>
  <option value="system">System</option>
</select>
```

**Per-project dark mode overrides:** When a project theme provides `tokens_dark`, those values are injected into a `<style>` element scoped to `.dark`:

```typescript
function applyDarkOverrides(tokensDark: Record<string, string>): void {
  // Remove existing project dark override style
  let styleEl = document.getElementById('orqa-project-dark-theme');
  if (styleEl) styleEl.remove();

  if (!tokensDark || Object.keys(tokensDark).length === 0) return;

  // Build CSS rule
  const declarations = Object.entries(tokensDark)
    .map(([key, value]) => `  --${key}: ${value};`)
    .join('\n');

  styleEl = document.createElement('style');
  styleEl.id = 'orqa-project-dark-theme';
  styleEl.textContent = `.dark {\n${declarations}\n}`;
  document.head.appendChild(styleEl);
}
```

**Tauri-specific dark mode note:** There is a known issue on Linux where Tauri's WebView does not reliably propagate `prefers-color-scheme` from the OS to the web content. The `mode-watcher` approach avoids this problem entirely because it uses a class-based strategy (`.dark` on `<html>`) rather than relying on the `prefers-color-scheme` media query. This is the same reason shadcn-svelte recommends class-based dark mode over media queries.

### Performance Considerations

**Setting CSS custom properties on `:root` is extremely fast.** The browser does not need to re-parse the stylesheet -- it simply updates the computed value for every element that references the changed variable. This triggers a repaint, not a reflow, for color changes.

Benchmarks from real-world applications show that updating 20-30 CSS custom properties on `:root` takes under 1ms. The subsequent repaint for a typical UI is 2-5ms. Total perceived time: imperceptible.

**What to avoid:**

1. **Do not rebuild the entire stylesheet on theme change.** Injecting/replacing `<style>` elements with thousands of rules is slow. Setting individual CSS variables is fast.
2. **Do not use Svelte's `--css-prop={value}` syntax for global theming.** This syntax wraps each component in a `<svelte-css-wrapper>` element with `display: contents`, which adds DOM nodes and can affect CSS selectors using `>`. It is designed for per-component prop passing, not global theming.
3. **Do not convert colors at render time.** All color values should be pre-converted to OKLCH during extraction and stored in that format. Runtime conversion on every render would be wasteful.
4. **Batch property updates.** Set all CSS variables in a single synchronous block before the browser paints. This ensures a single repaint, not one per property.

## Q3: Token Extraction Pipeline

**Question:** How does OrqaStudio extract design tokens from project files and map them to its internal schema?

### Minimum Viable Token Set

OrqaStudio needs to map extracted project colors to these semantic roles (the shadcn-svelte variables):

| Token | Role | Extraction Priority |
|-------|------|-------------------|
| `primary` | Main brand/action color | Required |
| `primary-foreground` | Text on primary | Derived |
| `secondary` | Secondary UI surfaces | Optional |
| `secondary-foreground` | Text on secondary | Derived |
| `accent` | Highlights, active states | Optional |
| `accent-foreground` | Text on accent | Derived |
| `background` | Page/app background | Optional |
| `foreground` | Default text color | Optional |
| `muted` | Subdued backgrounds | Optional |
| `muted-foreground` | Subdued text | Optional |
| `destructive` | Error/danger color | Optional |
| `border` | Border color | Derived |
| `input` | Input border/background | Derived |
| `ring` | Focus ring | Derived |

**"Required" means:** if the extraction finds nothing for this role, the project theme is considered empty and OrqaStudio falls back entirely to its default theme. A project theme with only a primary color is still useful.

**"Derived" means:** these can be computed from the required/optional values. For example, `primary-foreground` is computed to be white or black depending on the lightness of `primary` (contrast ratio check). `border` and `input` are typically derived from `background` with reduced lightness.

### Extraction Sources (in priority order)

#### Source 1: CSS Custom Properties in `:root`

The highest-fidelity source. If a project defines `:root { --color-primary: #1a1a2e; }`, this is an explicit design decision.

**Parsing approach (Rust):** Use the `csscolorparser` crate for color value parsing. For CSS file parsing, use a regex-based approach rather than a full CSS parser -- OrqaStudio only needs to extract `:root` (and `.dark`) variable declarations, not understand the full CSS AST.

```rust
use csscolorparser::Color;
use regex::Regex;
use std::collections::HashMap;

/// Extract CSS custom properties from :root and .dark selectors
pub fn extract_css_tokens(css_content: &str) -> ExtractedTokens {
    let root_re = Regex::new(r":root\s*\{([^}]+)\}").unwrap();
    let dark_re = Regex::new(r"\.dark\s*\{([^}]+)\}").unwrap();
    let var_re = Regex::new(r"--([\w-]+)\s*:\s*([^;]+);").unwrap();

    let mut light_tokens = HashMap::new();
    let mut dark_tokens = HashMap::new();

    // Extract :root variables
    for cap in root_re.captures_iter(css_content) {
        let block = &cap[1];
        for var_cap in var_re.captures_iter(block) {
            let name = var_cap[1].to_string();
            let value = var_cap[2].trim().to_string();
            light_tokens.insert(name, value);
        }
    }

    // Extract .dark variables
    for cap in dark_re.captures_iter(css_content) {
        let block = &cap[1];
        for var_cap in var_re.captures_iter(block) {
            let name = var_cap[1].to_string();
            let value = var_cap[2].trim().to_string();
            dark_tokens.insert(name, value);
        }
    }

    ExtractedTokens { light: light_tokens, dark: dark_tokens }
}
```

**File discovery:** Scan for CSS files in common locations:
- `src/app.css`, `src/global.css`, `src/index.css`
- `src/styles/*.css`
- `app/globals.css` (Next.js convention)
- `ui/src/lib/styles/*.css` (SvelteKit convention)

If the project already uses shadcn conventions (variables named `--primary`, `--secondary`, etc.), the mapping is trivial -- the names match directly.

#### Source 2: Tailwind Config (v3)

For projects using Tailwind v3 with `tailwind.config.js` / `.ts` / `.cjs` / `.mjs`:

**Challenge:** Tailwind configs are JavaScript/TypeScript files, not JSON. Parsing them in Rust requires either:

- **Option A: Regex extraction** -- Extract the `colors:` object from the config file using pattern matching. Fragile but zero-dependency.
- **Option B: Node.js subprocess** -- Run a small Node.js script that `require()`s the config and outputs `JSON.stringify(resolvedConfig.theme.colors)`. Reliable but requires Node.js.
- **Option C: Parse as JSON5/relaxed JSON** -- Many tailwind configs have a simple enough `colors` object that it can be extracted and parsed as JSON5. Works for simple configs, fails for configs that use JavaScript expressions.

**Verdict: Option B (Node.js subprocess) with Option A as fallback.**

Most developer machines that have a Tailwind project will have Node.js installed. The subprocess approach is the most reliable:

```rust
use std::process::Command;

/// Extract theme colors from tailwind config using Node.js
pub fn extract_tailwind_colors(project_root: &Path) -> Result<HashMap<String, String>> {
    let script = r#"
        const path = require('path');
        const configPath = process.argv[2];
        try {
            const resolveConfig = require('tailwindcss/resolveConfig');
            const config = require(configPath);
            const fullConfig = resolveConfig(config);
            console.log(JSON.stringify(fullConfig.theme.colors));
        } catch (e) {
            // Fallback: try to read the file and extract colors manually
            const fs = require('fs');
            const content = fs.readFileSync(configPath, 'utf-8');
            // Basic extraction for simple configs
            console.log(JSON.stringify({ error: e.message }));
        }
    "#;

    let config_path = find_tailwind_config(project_root)?;
    let output = Command::new("node")
        .arg("-e")
        .arg(script)
        .arg(&config_path)
        .current_dir(project_root)
        .output()?;

    let colors: HashMap<String, serde_json::Value> =
        serde_json::from_slice(&output.stdout)?;

    flatten_color_palette(colors)
}
```

**Tailwind v4:** Does not use a JavaScript config file. Instead, theme values are defined directly in CSS using `@theme`. These appear as CSS custom properties prefixed with `--color-`, `--font-`, `--spacing-`, etc. on `:root`. The CSS extraction approach (Source 1) handles Tailwind v4 automatically.

#### Source 3: shadcn/shadcn-svelte Theme File

If the project itself uses shadcn (React) or shadcn-svelte, the theme is defined in the global CSS file with the exact same variable names OrqaStudio uses. Direct extraction with zero mapping required.

**Detection:** Look for `--primary:` and `--primary-foreground:` in any CSS file's `:root` block. If both exist, this is almost certainly a shadcn-style theme.

### Color Format Conversion

Extracted colors may be in any CSS color format. All must be normalized to OKLCH before storage.

**Rust crate: `csscolorparser`**

This crate parses CSS Color Level 4 values including: hex (`#rgb`, `#rrggbb`, `#rrggbbaa`), `rgb()`, `rgba()`, `hsl()`, `hsla()`, `hwb()`, `lab()`, `lch()`, `oklab()`, `oklch()`, and named colors. It returns an RGBA tuple that can be converted to OKLCH.

```rust
use csscolorparser::Color;

/// Convert any CSS color string to OKLCH format
pub fn to_oklch(css_color: &str) -> Result<String> {
    // Handle Tailwind color references like "colors.blue.500"
    if css_color.starts_with("colors.") || css_color.starts_with("theme(") {
        return Err(anyhow!("Tailwind reference, not a literal color"));
    }

    let color: Color = css_color.parse()?;
    let [l, c, h, _a] = color.to_oklch_array();

    // Round to 3 decimal places for readability
    Ok(format!("oklch({:.3} {:.3} {:.3})", l, c, h))
}
```

**OKLCH rationale:** OKLCH is a perceptually uniform color space. Two colors with the same lightness value in OKLCH will appear equally bright to the human eye (unlike HSL, where `hsl(60, 100%, 50%)` yellow appears much brighter than `hsl(240, 100%, 50%)` blue at the same lightness). This matters for:

- **Deriving foreground colors** -- Checking if `L > 0.7` reliably determines whether a color needs white or dark text on top.
- **Generating palette variants** -- Adjusting `L` by fixed amounts produces perceptually even steps.
- **shadcn-svelte compatibility** -- The current version of shadcn-svelte uses OKLCH for all its CSS variables.

### Token Mapping Algorithm

When OrqaStudio extracts raw colors from a project, it must map them to the semantic roles (`primary`, `secondary`, `accent`, etc.). This mapping depends on the source format:

**Case 1: shadcn-style variables detected (trivial mapping)**

```
--primary         -> primary
--primary-foreground -> primary-foreground
--secondary       -> secondary
(etc.)
```

**Case 2: Tailwind color palette detected (heuristic mapping)**

```
Most saturated color at shade 600   -> primary
Least saturated color at shade 100  -> secondary
Color with highest chroma           -> accent
Neutral shade 50                    -> background
Neutral shade 950                   -> foreground
Neutral shade 200                   -> border
Red shade 500                       -> destructive
```

**Implementation:**

```rust
/// Map a flat Tailwind color palette to shadcn-svelte semantic tokens
pub fn map_tailwind_to_tokens(
    palette: &HashMap<String, String>,
) -> HashMap<String, String> {
    let mut tokens = HashMap::new();

    // Step 1: Identify the "primary" color
    // Heuristic: the first custom (non-gray, non-standard) color at shade 500 or 600
    let primary = find_primary_color(palette);
    if let Some((name, color)) = primary {
        let oklch = to_oklch(&color).unwrap();
        tokens.insert("primary".into(), oklch.clone());
        tokens.insert("primary-foreground".into(), contrast_foreground(&oklch));
        tokens.insert("ring".into(), oklch);
    }

    // Step 2: Derive background/foreground from grays
    if let Some(bg) = palette.get("gray.50").or(palette.get("neutral.50")).or(palette.get("slate.50")) {
        tokens.insert("background".into(), to_oklch(bg).unwrap());
    }
    if let Some(fg) = palette.get("gray.950").or(palette.get("neutral.950")).or(palette.get("slate.950")) {
        tokens.insert("foreground".into(), to_oklch(fg).unwrap());
    }

    // Step 3: Destructive from red
    if let Some(red) = palette.get("red.500") {
        tokens.insert("destructive".into(), to_oklch(red).unwrap());
    }

    // Step 4: Derive border, input, muted from background
    // (adjust lightness of background color)
    derive_surface_tokens(&mut tokens);

    tokens
}
```

**Case 3: Generic CSS variables (best-effort mapping)**

For projects with custom variable names like `--brand-blue`, `--text-color`, `--bg-main`:

```rust
/// Heuristic mapping for arbitrary CSS variable names
const NAME_MAPPINGS: &[(&[&str], &str)] = &[
    (&["primary", "brand", "main", "accent-primary"], "primary"),
    (&["secondary", "accent-secondary"], "secondary"),
    (&["accent", "highlight", "emphasis"], "accent"),
    (&["background", "bg", "bg-main", "surface"], "background"),
    (&["foreground", "fg", "text", "text-primary", "text-main"], "foreground"),
    (&["muted", "subtle", "disabled", "bg-secondary"], "muted"),
    (&["border", "divider", "separator", "outline"], "border"),
    (&["error", "danger", "destructive", "red", "negative"], "destructive"),
];
```

This is best-effort. Unknown variables are stored in `unmapped` for the user to manually assign.

### Deriving Foreground Colors

Given a background color in OKLCH, derive an appropriate foreground (text) color:

```rust
/// Return white or dark foreground based on background lightness
fn contrast_foreground(oklch_bg: &str) -> String {
    let lightness = parse_oklch_lightness(oklch_bg);
    if lightness > 0.65 {
        "oklch(0.15 0 0)".to_string()  // dark text on light background
    } else {
        "oklch(0.98 0 0)".to_string()  // light text on dark background
    }
}

/// Derive surface tokens (border, input, muted) from background
fn derive_surface_tokens(tokens: &mut HashMap<String, String>) {
    if let Some(bg) = tokens.get("background").cloned() {
        let bg_l = parse_oklch_lightness(&bg);

        // Border: slightly darker than background
        let border_l = (bg_l - 0.08).max(0.0);
        tokens.entry("border".into())
            .or_insert_with(|| format!("oklch({:.3} 0 0)", border_l));

        // Input: same as border
        tokens.entry("input".into())
            .or_insert_with(|| tokens.get("border").unwrap().clone());

        // Muted: slightly darker than background
        let muted_l = (bg_l - 0.03).max(0.0);
        tokens.entry("muted".into())
            .or_insert_with(|| format!("oklch({:.3} 0 0)", muted_l));

        // Muted-foreground: mid-range
        tokens.entry("muted-foreground".into())
            .or_insert_with(|| format!("oklch({:.3} 0 0)", 0.556));
    }
}
```

## Q4: Per-Project Theme Persistence

**Question:** Where to store extracted/customized themes per project? How does cache invalidation work? Can users customize the auto-extracted theme?

### Storage: New `project_themes` Table in SQLite

**Verdict: Add a `project_themes` table to the existing schema.**

```sql
-- ============================================================
-- PROJECT THEMES (design token extraction results)
-- ============================================================
CREATE TABLE project_themes (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    -- Source file that tokens were extracted from
    source_path     TEXT NOT NULL,         -- e.g. "tailwind.config.ts", "src/app.css"
    source_type     TEXT NOT NULL
                    CHECK (source_type IN (
                        'tailwind-v3', 'tailwind-v4', 'css-variables',
                        'shadcn', 'custom'
                    )),
    -- SHA-256 of the source file for change detection
    source_hash     TEXT NOT NULL,
    -- JSON: extracted tokens in OKLCH format
    -- { "primary": "oklch(...)", "background": "oklch(...)", ... }
    tokens_light    TEXT NOT NULL DEFAULT '{}',
    -- JSON: dark mode token overrides (only tokens that differ from light)
    tokens_dark     TEXT NOT NULL DEFAULT '{}',
    -- JSON: raw extracted values before mapping (for debugging/re-mapping)
    raw_extracted   TEXT NOT NULL DEFAULT '{}',
    -- JSON: values that could not be mapped to shadcn variables
    unmapped        TEXT NOT NULL DEFAULT '{}',
    -- Whether this theme is the active theme for the project
    is_active       INTEGER NOT NULL DEFAULT 1,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE UNIQUE INDEX idx_themes_project_source ON project_themes(project_id, source_path);

-- ============================================================
-- PROJECT THEME OVERRIDES (user customizations per project)
-- ============================================================
CREATE TABLE project_theme_overrides (
    id              INTEGER PRIMARY KEY,
    project_id      INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    -- CSS variable name without '--' prefix
    token_name      TEXT NOT NULL,
    -- Override value in OKLCH format
    token_value     TEXT NOT NULL,
    -- Which mode this override applies to
    mode            TEXT NOT NULL DEFAULT 'light'
                    CHECK (mode IN ('light', 'dark')),
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);

CREATE UNIQUE INDEX idx_overrides_project_token ON project_theme_overrides(project_id, token_name, mode);
```

**Why two tables instead of one:**

1. **`project_themes`** stores the automated extraction result. It is regenerated whenever the source file changes. One project could have multiple theme sources (e.g., both a `tailwind.config.ts` and a `src/app.css`), though only one is `is_active = 1`.
2. **`project_theme_overrides`** stores user customizations. These survive re-extraction. The user should never lose their tweaks because the source file changed.

### Cache Invalidation via File Watcher

The `notify` crate is already planned for watching `.claude/` files (see `docs/research/onboarding.md`, Q2). The same file watcher can watch design token source files.

**Integration with existing watcher:**

```rust
use notify::{Watcher, RecursiveMode, Event, EventKind};

/// Files to watch for theme changes
const THEME_WATCH_PATTERNS: &[&str] = &[
    "tailwind.config.js",
    "tailwind.config.ts",
    "tailwind.config.cjs",
    "tailwind.config.mjs",
    "src/app.css",
    "src/global.css",
    "src/index.css",
    "src/styles/tokens.css",
    "ui/src/lib/styles/app.css",
    "app/globals.css",
];

fn on_file_change(event: Event, project_id: i64) {
    if let EventKind::Modify(_) = event.kind {
        for path in &event.paths {
            let filename = path.file_name().unwrap().to_str().unwrap();

            // Check if this is a theme-related file
            if is_theme_file(path) {
                // Re-hash the file
                let new_hash = sha256_file(path);
                let existing_hash = db::get_theme_source_hash(project_id, path);

                if new_hash != existing_hash {
                    // Re-extract tokens
                    let tokens = extract_tokens(path);
                    db::update_project_theme(project_id, path, &new_hash, &tokens);

                    // Emit Tauri event to frontend
                    app_handle.emit("theme-updated", ThemeUpdatedPayload {
                        project_id,
                        source: path.to_string_lossy().into(),
                    }).unwrap();
                }
            }
        }
    }
}
```

The frontend listens for `theme-updated` events and re-applies the theme:

```typescript
import { listen } from '@tauri-apps/api/event';

listen<ThemeUpdatedPayload>('theme-updated', (event) => {
  const { project_id } = event.payload;
  if (project_id === currentProjectId) {
    // Fetch updated theme from backend and re-apply
    invoke('get_project_theme', { projectId: project_id })
      .then(applyProjectTheme);
  }
});
```

### User Override Flow

The settings panel includes a "Theme" section where users can:

1. **See the current extracted palette** -- Color swatches for each mapped token.
2. **Toggle per-project theming** -- Enable/disable project theme (fall back to OrqaStudio defaults).
3. **Override individual tokens** -- Click a swatch to open a color picker. The chosen value is saved to `project_theme_overrides`.
4. **Reset overrides** -- "Reset to extracted" button clears all overrides for the project.
5. **Reset theme** -- "Use OrqaStudio defaults" button disables the project theme entirely.

```svelte
<!-- ThemeSettings.svelte (simplified) -->
<script lang="ts">
  import { invoke } from '@tauri-apps/api/core';
  import { applyProjectTheme } from '$lib/theme-manager';

  let projectId = $props<number>();
  let theme = $state<ProjectTheme | null>(null);
  let overrides = $state<Record<string, string>>({});

  $effect(() => {
    invoke('get_project_theme', { projectId }).then(t => theme = t);
    invoke('get_theme_overrides', { projectId }).then(o => overrides = o);
  });

  async function setOverride(tokenName: string, value: string) {
    await invoke('set_theme_override', { projectId, tokenName, value, mode: 'light' });
    overrides[tokenName] = value;
    if (theme) {
      theme.user_overrides = { ...overrides };
      applyProjectTheme(theme);
    }
  }

  async function resetOverrides() {
    await invoke('clear_theme_overrides', { projectId });
    overrides = {};
    if (theme) {
      theme.user_overrides = {};
      applyProjectTheme(theme);
    }
  }
</script>

{#if theme}
  <div class="grid grid-cols-4 gap-2">
    {#each Object.entries(theme.tokens) as [name, value]}
      <button
        class="h-10 w-full rounded border"
        style="background-color: {overrides[name] ?? value}"
        onclick={() => openColorPicker(name, overrides[name] ?? value)}
        title="{name}: {overrides[name] ?? value}"
      />
      <span class="text-sm text-muted-foreground">{name}</span>
    {/each}
  </div>
{/if}
```

| Question | Recommendation | Key Rationale |
|----------|---------------|---------------|
| Q1: Token Format & Storage | JSON object in SQLite keyed by shadcn-svelte variable names, all values in OKLCH | Direct 1:1 mapping to CSS variables. No translation layer. OKLCH is shadcn-svelte's native format. |
| Q2: Runtime Theme Application | `document.documentElement.style.setProperty()` for light mode, injected `<style>` for dark mode overrides. `mode-watcher` for dark/light/system toggle. | Sub-1ms application time. | Q3: Token Extraction Pipeline | Three-source extraction (CSS `:root` > Tailwind config > generic vars). `csscolorparser` for color parsing. Heuristic mapping to semantic roles. | Q4: Per-Project Persistence | `project_themes` table (extraction results) + `project_theme_overrides` table (user customizations). File watcher for cache invalidation. ### OrqaStudio Default Theme Tokens

The baseline OrqaStudio theme (used when no project theme is active) is the default shadcn-svelte "zinc" palette. It is defined in the global CSS file (`ui/src/app.css`) and requires no runtime processing. When a project theme is cleared, removing inline styles on `:root` instantly restores these defaults via the CSS cascade.
