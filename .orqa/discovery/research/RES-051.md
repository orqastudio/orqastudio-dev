## The Problem

AI CLI tools (Claude Code, Cursor, etc.) maintain their own per-project memory systems — files stored outside the project's governance structure. For OrqaStudio projects, this creates a fundamental tension:

1. **The artifact graph is supposed to be the single source of truth** — agents navigate it to find context, extend it through work, and the system self-reinforces through visibility.

2. **External memory is a parallel knowledge system** — it stores observations, decisions, feedback, and project context that agents carry between sessions, but it sits outside the graph. It has no relationships, no schema validation, no pipeline stage, and no enforcement.

3. **Duplication and drift are inevitable** — knowledge captured in memory at the moment it's learned eventually gets encoded in artifacts (rules, decisions, skills), but the memory entry remains. Future sessions load both, potentially with contradictions.

## The Genuine Gap

After filtering out duplicates and partial overlaps, the knowledge that has **no home in the artifact graph** falls into these categories:

### 1. Agent Behavioral Directives

How agents should work *with this user/team* — not what they should build, but how they should think and communicate.

Examples:
- "Conciseness over granularity in prompt injection"
- "No hard deadlines — structured approach over speed"
- "Don't use agents when the agentic structure itself is being refactored"
- "Orchestrator manages its own dev lifecycle"

These are not observations (they didn't come from mistakes). They're not principles (they're not architectural choices). They're not practices (they're not reusable methodology). They're **collaboration configuration**.

### 2. Product Philosophy Framing

The "why behind the why" — not what the product does or how it's built, but the mental model that guides decisions.

Examples:
- "OrqaStudio is structured therapy for problem-solving"
- "Self-enforcement through visibility, not more constraints"
- "The system should prevent losing sight of purpose during implementation"

The vision document captures *what* the product is. Pillars capture *what matters*. But the philosophical framing — the metaphors and mental models that help agents make judgment calls — sits between vision and pillars with no explicit home.

### 3. Context-Specific Operational Knowledge

Things that are true about the development environment but not about the product.

Examples:
- "`make restart` completing means the app has EXITED" (CLI-specific dev behavior)
- Svelte `$derived` syntax gotcha
- "Dogfood rules load via symlinks into CLI sessions where they shouldn't"

These are somewhere between lessons (they were learned from experience) and documentation (they describe current state). Some should be in coding standards or dev docs. Others are genuine operational notes that matter to agents but not to the codebase.

## Research Questions

### Q1: Where do behavioral directives enter the pipeline?

The knowledge maturity pipeline is:
```
Observation → Understanding → Principle → Practice → Enforcement → Verification
```

Behavioral directives feel like they enter at **Principle** (they're decisions about how agents should behave) but they don't follow the normal path — they weren't observed from mistakes and promoted. They were *declared by the user* as preferences.

Is there a missing entry point for "declared principles" vs "discovered principles"? Or should user directives be treated as Principles (decisions) that skip the observation/understanding stages because the user IS the understanding?

### Q2: What consumes this knowledge?

The orchestrator and all agents consume behavioral directives — they shape *every* interaction. This is different from domain skills (consumed by specific agents on specific tasks) or rules (consumed when specific patterns are detected).

Does "consumed by everything always" mean it belongs in the orchestrator prompt? In pillar definitions? In a new artifact type? Or is it just a property of certain rules/decisions that should be flagged as "universal context"?

### Q3: How does it mature?

Normal pipeline: observation → understanding → principle → practice → enforcement.

Behavioral directives seem to follow a different path:
- User states preference → agent records it → it guides behavior → it may eventually become a rule

But some directives never need to become rules. "No hard deadlines" is a collaboration preference, not an enforceable standard. It doesn't have a verification step. It's just... context.

Is there a class of knowledge that participates in the pipeline as *input* (it shapes how agents traverse the pipeline) but doesn't flow *through* the pipeline stages?

### Q4: Should external memory exist at all?

If the artifact graph is the single source of truth, should there be ANY knowledge in external memory? The ideal state (per user direction) is: memory is empty, artifacts have everything, the orchestrator prompt tells agents how to find it.

But external tools (Claude Code, Cursor) will always have their own memory systems. Can OrqaStudio make its artifact graph the *only* thing those tools need? Or does some adapter layer need to exist that syncs between the tool's memory format and the artifact graph?

### Q5: Is this actually about the orchestrator prompt?

The orchestrator prompt (`CLAUDE.md` / `orchestrator.md`) already serves as "universal agent context." Maybe behavioral directives just belong there — not as a new artifact type, but as a section of the orchestrator definition.

But the orchestrator prompt is already long (~2000 lines), and the vision for [EPIC-dc1e3e4b](EPIC-dc1e3e4b) is to shrink it to ~200 lines by moving content into the graph. Adding more content to it goes against that direction.

### Q6: Could pillars absorb this?

Pillars are "guiding principles." Behavioral directives are "guiding preferences." The gate mechanism (questions that evaluate alignment) could work for preferences too.

But pillars are product-focused ("does this feature serve clarity?") while behavioral directives are process-focused ("how should agents communicate?"). Mixing them would dilute the pillar concept.

## Observations So Far

1. **Most of what's in memory is already in artifacts.** The problem is not "we need a new artifact type" — it's "we're double-storing knowledge because the memory system doesn't know about the artifact graph."

2. **The genuine gap is small but real.** Agent behavioral directives and product philosophy framing don't have a clean home. But "small gap" doesn't automatically mean "new artifact type."

3. **The pipeline assumption may be wrong for this knowledge.** Not everything needs to mature through observation → verification. Some knowledge is declared (by the user) and consumed (by agents) without transformation. It's more like configuration than learning.

4. **The tool integration problem is distinct from the knowledge model problem.** Even if we solve "where does this knowledge live in the graph," we still need external tools to read from the graph instead of maintaining their own memory. That's a tooling/integration question, not an artifact design question.

## Conclusions

### Conclusion: No New Artifact Type Needed

All three categories of knowledge currently in external memory map to existing artifact types:

| Knowledge Category | Artifact Type | Examples |
|---|---|---|
| **Product philosophy framing** | Pillars (PILLAR-NNN) | "Structured therapy for problem-solving", "self-enforcement through visibility" |
| **Agent behavioral directives** | Architecture Decisions (AD-NNN) | "Conciseness over granularity", "no hard deadlines", "don't use agents when refactoring agent structure" |
| **Context-specific operational knowledge** | Skills | `make restart` behavior, Svelte $derived gotcha, dogfood symlink issue |

### Resolution of Research Questions

| Question | Answer |
|---|---|
| Q1: Where do directives enter the pipeline? | As ADs — declared principles that skip observation/understanding because the user IS the understanding |
| Q2: What consumes this knowledge? | Pillars: universal context. ADs: loaded when relevant. Skills: loaded per-task |
| Q3: How does it mature? | Pillars are foundational (don't flow through the pipeline). ADs follow proposed→accepted. Skills follow normal lifecycle |
| Q4: Should external memory exist? | No — everything maps to existing types. External memory should converge to empty as knowledge migrates to artifacts |
| Q5: Is this about the orchestrator prompt? | No — knowledge lives in artifacts. The orchestrator navigates the graph to find what it needs. This aligns with [EPIC-dc1e3e4b](EPIC-dc1e3e4b)'s goal of shrinking the orchestrator prompt |
| Q6: Could pillars absorb it? | Product philosophy yes (pillars may need review/expansion). Behavioral directives no (they're ADs). Operational knowledge no (it's skills) |
