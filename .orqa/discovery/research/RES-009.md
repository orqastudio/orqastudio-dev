## Aim: Compatibility

The documented aim of this research and the resulting architecture is **compatibility**. OrqaStudio should work with as many AI providers as possible — from fixed-cost subscriptions (Claude Max) to pay-per-token APIs (Anthropic, OpenAI, Gemini) to free/local models (Ollama, vLLM). The architecture must support both agent runtime SDKs (via sidecar) and direct API calls (via Rust) without the frontend knowing or caring which path is used. Provider choice is a user/org decision driven by cost, compliance, and ecosystem preferences — the app should never constrain that choice.

---

## Executive Summary

OrqaStudio needs a multi-provider architecture. Research identified two categories of AI providers with fundamentally different integration patterns:

1. **Agent runtime SDKs** (Claude Agent SDK, OpenAI Agents SDK, Google ADK) — full runtimes that manage tool loops, sessions, and streaming internally. Architecturally similar to each other. Require a Node.js/Bun sidecar process.

2. **Direct APIs** (Anthropic Messages API, OpenAI Chat Completions, Gemini Developer API, any OpenAI-compatible endpoint) — stateless HTTP APIs. Rust can call them directly via `reqwest` + SSE parsing. No sidecar needed.

**Recommendation:** Hybrid architecture — Provider interface inside a shared sidecar for agent SDKs (Option A) + direct HTTP path in Rust for API providers (Option C).

Additionally, business ecosystem AI (Microsoft 365 Copilot, Google Workspace Gemini) is **not usable** as a general-purpose AI provider. These subscriptions provide AI only within their own apps, not via APIs for third-party tools.

## Part 1: Agent Runtime SDKs

### Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`)

| Dimension | Details |
|-----------|---------|
| Architecture | Full runtime — spawns Claude Code CLI binary as subprocess |
| Tool loop | SDK-managed via `query()` — handles stop_reason, tool execution, result feeding, looping |
| Sessions | Built-in — UUID-based, resume by ID, includes context compaction |
| Streaming | Async iterator — raw Anthropic API events (`content_block_delta`, `text_delta`, `thinking_delta`) |
| Runtime | Node.js/Bun + CLI binary (`pathToClaudeCodeExecutable`) |
| Auth | OAuth (Claude Max subscription) — not API keys |
| MCP support | First-class — `createSdkMcpServer()` + `tool()` |
| Tool approval | Built-in — `canUseTool` callback pauses execution for human review |

**Unique characteristics:** Requires bundled CLI binary. OAuth-only auth model. Includes 14+ built-in tools (Read, Write, Bash, etc.). Fixed-cost subscription model.

### OpenAI Agents SDK (`@openai/agents`)

| Dimension | Details |
|-----------|---------|
| Architecture | Full runtime — `Runner.run()` manages complete agent loop |
| Tool loop | SDK-managed — loops through LLM calls, tool execution, handoffs |
| Sessions | Built-in — `MemorySession`, `OpenAIConversationsSession`, `OpenAIResponsesCompactionSession` |
| Streaming | Async iterator — `RunStreamEvent`, `RunItemStreamEvent`, `RunRawModelStreamEvent`. Also WebSocket support |
| Runtime | Node.js — standard npm package, no external binary |
| Auth | API key |
| MCP support | Yes — `MCPServerStdio`, `MCPServerSSE`, `MCPServerStreamableHttp` |
| Tool approval | Yes — `RunToolApprovalItem` for human-in-the-loop |

**Unique characteristics:** Supports agents-as-tools and multi-agent handoffs natively. Hosted tools (web search, code interpreter, image generation, computer use). No CLI binary dependency.

### Google ADK (`@google/adk`)

| Dimension | Details |
|-----------|---------|
| Architecture | Full runtime — cooperative yield-pause-resume event loop |
| Tool loop | SDK-managed — Runner mediates between agents, LLM calls, and tools |
| Sessions | Built-in — `InMemorySessionService`, `DatabaseSessionService` (SQLite/PostgreSQL), `VertexAiSessionService` |
| Streaming | Async iterable — yield-based with `partial=True` for immediate forwarding |
| Runtime | Node.js 24.13+ (high minimum version) |
| Auth | API key (Gemini API) or service account (Vertex AI) |
| MCP support | Via toolset adapters — converts MCP tools to ADK `BaseTool` |
| Tool approval | Via event actions |

**Unique characteristics:** Model-agnostic (supports Gemini, Claude, others). Requires Node.js 24.13+ which is a compatibility concern. Cooperative yield-based event model differs from async iterators.

### Agent SDK Comparison

| Feature | Claude SDK | OpenAI SDK | Google ADK |
|---------|-----------|------------|------------|
| Tool loop managed | Yes | Yes | Yes |
| Session resume | Yes | Yes | Yes |
| MCP support | First-class | First-class | Via adapters |
| Tool approval | Yes | Yes | Yes |
| Streaming pattern | Async iterator | Async iterator | Async iterable |
| External binary | Yes (CLI) | No | No |
| Auth model | OAuth | API key | API key |
| Min Node.js | 18+ | 18+ | 24.13+ |

**Conclusion:** Architecturally similar enough for a shared sidecar with a Provider interface. The differences are in event naming and API shape, not fundamental architecture.

## Part 2: Direct APIs

### Anthropic Messages API (`/v1/messages`)

| Dimension | Details |
|-----------|---------|
| Architecture | Stateless HTTP API — each request independent |
| Tool loop | Caller-managed — API returns `stop_reason: "tool_use"`, caller executes and re-sends |
| Sessions | Caller-managed — no server-side state |
| Streaming | SSE — `message_start`, `content_block_start`, `content_block_delta`, `message_stop` |
| Rust support | Yes — `anthropic-sdk-rust`, `anthropic-api` crates |
| Auth | API key |

### OpenAI Chat Completions / Responses API

| Dimension | Details |
|-----------|---------|
| Architecture | Stateless (Chat Completions) or stateful (Responses API) HTTP |
| Tool loop | Caller-managed (Chat Completions) or partially server-managed (Responses API) |
| Sessions | Caller-managed (Chat) or server-managed (Responses) |
| Streaming | SSE |
| Rust support | Yes — `async-openai` crate (mature) |
| Auth | API key |

### Gemini Developer API

| Dimension | Details |
|-----------|---------|
| Architecture | REST API with API key auth |
| Tool loop | Caller-managed |
| Sessions | Caller-managed |
| Streaming | SSE |
| Rust support | REST API callable via `reqwest` |
| Auth | API key only — no billing account needed for free tier |
| Free tier | 5-15 RPM, 100-1,000 RPD, 250K TPM — genuinely usable for light work |

### OpenAI-Compatible Endpoints (Ollama, OpenRouter, vLLM, etc.)

| Dimension | Details |
|-----------|---------|
| Architecture | Same as OpenAI Chat Completions API |
| Tool loop | Caller-managed |
| Auth | API key or none (local) |
| Rust support | Same `async-openai` crate works |

**Conclusion:** All direct APIs follow the same pattern — HTTP + SSE + caller-managed tool loop. Rust can handle all of them without a sidecar. The tool loop, session management, and context tracking would be implemented once in Rust and shared across all direct API providers.

## Part 3: Business Ecosystem AI

### Microsoft 365 Copilot

| Integration | General AI? | Desktop app? | Included in license? | Relevance |
|-------------|------------|-------------|---------------------|-----------|
| M365 Copilot Chat API | No (M365 data grounding only) | Yes | Yes ($30/user/mo add-on) | LOW |
| Azure OpenAI Service | Yes | Yes | No (separate Azure sub) | HIGH as provider |
| GitHub Copilot SDK | Yes (BYOK) / Limited (subscription) | Yes | With Copilot sub | WATCH (technical preview) |
| Copilot Studio | No (agent builder platform) | No | Partial | LOW |
| Semantic Kernel | SDK, not a service | Yes | N/A (open source) | LOW |

**Key finding:** M365 Copilot ($30/user/mo) does NOT give general-purpose AI API access. It provides AI inside Microsoft apps and a limited Chat API grounded in M365 data (SharePoint, OneDrive). For general-purpose AI, orgs need Azure OpenAI Service which requires a separate Azure subscription.

**GitHub Copilot SDK** is interesting — its BYOK mode lets any app use Copilot as an agent runtime with your own API keys. Still in technical preview (since Jan 2026). Worth monitoring.

### Google Workspace

| Integration | General AI? | Desktop app? | Included in license? | Relevance |
|-------------|------------|-------------|---------------------|-----------|
| Workspace Gemini features | N/A (UI only) | No | Yes (since Jan 2025) | NONE |
| Gemini Developer API | Yes | Yes | No (separate, has free tier) | HIGH as provider |
| Vertex AI | Yes | Yes | No (GCP billing) | MEDIUM |

**Key finding:** Google Workspace Gemini features are locked to Google's own apps. No API access. The Gemini Developer API is a completely separate product — a Workspace subscription does not grant API access. However, any Google account (including Workspace) can create a free-tier API key with no billing account.

### The Honest Answer for Charities / Small Companies

An M365 Copilot or Google Workspace subscription does **not** give general-purpose AI API access for third-party tools. To use AI in OrqaStudio, organizations need one of:

| Path | Cost | Complexity |
|------|------|------------|
| Claude Max subscription (via Agent SDK) | $100-200/mo fixed | Low (sidecar handles everything) |
| Gemini Developer API free tier | $0 | Low (API key only) |
| Gemini Developer API paid tier | Pay-per-token (cheap) | Low (needs GCP billing) |
| Anthropic API | Pay-per-token | Low (API key) |
| OpenAI API | Pay-per-token | Low (API key) |
| Azure OpenAI | Pay-per-token | Medium (Azure subscription + resource setup) |
| Local models (Ollama) | $0 (hardware cost) | Medium (install + run locally) |

The cheapest path is Gemini Developer API free tier (no cost, no billing account) or local models via Ollama.

## Part 4: Other Agent Frameworks

### Vercel AI SDK (`ai` npm package)

Provider-agnostic abstraction layer. Not a full agent runtime but a unified interface wrapping multiple providers behind a common API. `streamText()` works with OpenAI, Anthropic, or Google by swapping the model parameter. Supports tool loops via `maxSteps` and `ToolLoopAgent`. No built-in session management.

**Relevance:** Good reference for interface design but not worth adopting — it doesn't provide the agent runtime features we get from the agent SDKs.

### LangChain / LangGraph

Graph-based agent orchestration. Powerful but heavy — introduces graphs, nodes, edges, checkpoints. Built-in persistence and streaming. Model-agnostic.

**Relevance:** Overkill for our use case. The graph-based model adds complexity beyond what we need for "run an agent conversation with tools."

## Part 5: Recommended Architecture

### Hybrid: Shared Sidecar (Agent SDKs) + Direct Rust (APIs)

```
Agent SDK path (full runtime features):
  Svelte -> Rust -> NDJSON -> Sidecar -> [Claude SDK | OpenAI SDK | Google ADK] -> NDJSON -> Rust -> Svelte

Direct API path (simple, no sidecar):
  Svelte -> Rust -> reqwest HTTP/SSE -> [Anthropic | OpenAI | Gemini | Ollama | Azure OpenAI] -> Rust -> Svelte
```

Both paths converge at the same `ProviderEvent` types in Rust — the frontend doesn't know or care which path was used.

### Why Not Separate Sidecars Per Provider?

The agent SDKs are too architecturally similar to justify separate processes. They all follow: construct agent -> call run/query -> iterate streamed events -> handle tool approval. The differences are in event naming and API shape, not fundamental architecture.

### Why Not Force Everything Through the Sidecar?

Direct APIs are simple HTTP calls. Adding a sidecar process as middleware between Rust and an HTTP API adds latency, complexity, and a failure point with zero benefit. Rust's `reqwest` + SSE parsing handles these natively.

### Build Order

| Step | Provider | Path | Priority |
|------|----------|------|----------|
| 1 | Provider interface in sidecar + Claude adapter | Sidecar | NOW (composability gate) |
| 2 | `sdk_session_id` -> `provider_session_id` rename | Both | NOW (naming cleanup) |
| 3 | OpenAI Agents SDK adapter | Sidecar | HIGH |
| 4 | Direct Anthropic Messages API | Rust | HIGH |
| 5 | Direct OpenAI-compatible API (covers OpenAI, Ollama, OpenRouter, Azure OpenAI) | Rust | HIGH |
| 6 | Direct Gemini Developer API | Rust | MEDIUM |
| 7 | Google ADK adapter | Sidecar | LOW (Node.js 24.13+ concern) |
| 8 | GitHub Copilot SDK adapter | Sidecar | WATCH (still in preview) |

### Multi-Agent with Direct APIs

The direct API path fully supports multi-agent setups. Each "agent" is just:
- A system prompt
- A set of tool definitions
- A conversation history (stored in SQLite)

OrqaStudio's Rust layer manages the tool loop, orchestration, and handoffs. This gives us **more** control than the agent SDKs, not less.
