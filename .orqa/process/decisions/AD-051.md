## Decision

Artifacts exist in three layers. The distinction is between **type** (the definition — what kind of artifact) and **instance** (the data — a specific artifact with content). Each layer has different rules about what's fixed and what's configurable.

### Layer 1: App-Fixed (Immutable)

Type definitions, content, and relationships are all hardcoded into the app. These define what OrqaStudio IS as a product.

**What lives here:**
- Canonical type definitions (what "pillar" means, what "idea" means, what statuses each allows)
- Canonical relationship definitions (what `informs` means, what `delivers` means)
- Canonical status vocabulary and semantics
- Platform documentation (how to use OrqaStudio — for humans)
- Platform skills (how agents interact with OrqaStudio — for AI)

Cannot be edited through the app. Pre-commit hooks block file-based edits.

These are defined using the **same internal structures** as project-level definitions. The enforcement engine doesn't distinguish between canonical and project config — it enforces both. The difference is that canonical definitions are hardcoded rather than read from `project.json`.

### Layer 2: App-Required, Project-Authored (Required Existence, Configurable Content)

Certain artifact types must have instances for the project to function. The type is canonical (Layer 1). The instance must exist (the app requires it). The **content** is the project's to author.

| Type | Why Required | Content Is... |
|---|---|---|
| Pillars | Project must have guiding principles | What THIS project believes |
| Vision | Project must have direction | Where THIS project is going |
| Personas | Project must know who it serves | Who THIS project serves |
| Grounding | Agents must have behavioral anchors | What grounds THIS project's agents |

These define the project's purpose and identity within the framework.

**Required relationships at this layer:**
- Agents must be `grounded-by` at least one pillar or grounding artifact
- Rules must be `governed-by` at least one decision

### Layer 3: Project-Scoped (Fully Configurable)

Types, instances, content, and relationships are all defined by the project. The app enforces the rules the project configures, but doesn't prescribe what those rules are.

**Core types with project-scoped instances** — the types are canonical but instances are created freely:

| Type | Purpose |
|---|---|
| Ideas | Entry point for any change |
| Decisions | Resolutions that direct work and authorise enforcement |
| Rules | Standards we enforce |
| Lessons | What we've learned |
| Skills | Knowledge we teach |
| Agents | Roles that do work |

**Project-defined types** — the types themselves are defined by the project:

| Category | Examples |
|---|---|
| Discovery types | Research, wireframes, client discovery, experiments |
| Delivery types | Milestones, epics, tasks, phases, workstreams |

**Project relationships** — domain-specific connections (e.g., `depends-on` for software delivery). Defined in `project.json`, enforced by the app with the same rigour as canonical relationships.

**Project statuses** — additional statuses beyond the canonical vocabulary if needed for project-specific types.

### How the Layers Compose

The app combines all three layers into a single enforcement model:

```
┌─────────────────────────────────────────────────┐
│ Layer 1: App-Fixed                              │
│ Type definitions, canonical relationships,      │
│ status vocabulary, platform docs/skills         │
│ Immutable. Pre-commit hooks protect.            │
├─────────────────────────────────────────────────┤
│ Layer 2: App-Required, Project-Authored         │
│ Pillars, Vision, Personas, Grounding instances  │
│ Must exist. Content is the project's.           │
│ Required relationships enforced.                │
├─────────────────────────────────────────────────┤
│ Layer 3: Project-Scoped                         │
│ Ideas, decisions, rules, lessons, skills,       │
│ agents + project-defined discovery/delivery     │
│ types, relationships, statuses                  │
│ Fully configurable. App enforces config.        │
└─────────────────────────────────────────────────┘
```

## Rationale

The three-layer model ensures:
1. The systems thinking framework is non-negotiable — it IS the product
2. Projects must define their purpose (pillars, vision) before doing work
3. Everything beyond purpose and framework adapts to the domain
4. The enforcement engine is uniform — it doesn't care which layer a rule came from

## Consequences

- `project.json` defines project types, relationships, and statuses using the same schema as canonical definitions
- The app merges canonical + project config into a single enforcement model
- Layer 1 artifacts are read-only in the app; pre-commit hooks enforce immutability on disk
- Layer 2 artifacts are validated for existence on project load
- Layer 3 artifacts follow project-configured rules enforced by the app
- Research and wireframes are project types, not core types (corrects earlier assumption)
