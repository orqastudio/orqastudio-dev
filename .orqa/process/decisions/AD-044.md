## Decision

Operational standards flow through a defined pipeline. Each stage is represented by a specific artifact type. Standards enter as observations and mature through the pipeline until they are enforced and verified.

```
Observation (IMPL, maturity: observation)
  → Understanding (IMPL, maturity: understanding)
    → Principle (AD)
      → Practice (SKILL)
        → Enforcement (RULE)
          → Verification (VER)
```

### Stage Responsibilities

| Stage | Artifact | What Happens |
|-------|----------|-------------|
| **Observation** | IMPL (observation) | A pattern or anti-pattern is noticed. Raw finding. |
| **Understanding** | IMPL (understanding) | The observation is investigated. Root cause identified. |
| **Principle** | AD | An architectural decision codifies the understanding. |
| **Practice** | SKILL | A skill captures how to apply the principle. |
| **Enforcement** | RULE | A rule makes the practice mandatory. |
| **Verification** | VER | A verification record proves compliance. |

### Pipeline Flow Rules

1. **Forward flow** — Standards move forward through the pipeline, not backward
2. **Each stage is optional** — Not every observation becomes a rule
3. **Relationships track flow** — `observes`, `grounded`, `grounded-by`, `enforces`, `enforces` edges connect stages
4. **Bidirectional** — Every forward edge has an inverse edge on the target

### Examples

**Error handling standard:**
1. [IMPL-e53df28b](IMPL-e53df28b) observes: "unwrap() in production code caused a panic"
2. [IMPL-e53df28b](IMPL-e53df28b) matures to understanding: "All Rust production code must use Result types"
3. [AD-e513c9e4](AD-e513c9e4) codifies: "Error propagation via Result, thiserror for typed errors"
4. SKILL (orqa-error-composition) teaches: "How to compose errors across layers"
5. [RULE-b49142be](RULE-b49142be) enforces: "No unwrap()/expect()/panic!() in production"

**Component purity standard:**
1. [IMPL-5331c7b9](IMPL-5331c7b9) observes: "invoke() calls in display components cause coupling"
2. [AD-a334623b](AD-a334623b) codifies: "Display components receive props only"
3. SKILL (component-extraction) teaches: "How to extract pure components"
4. [RULE-b49142be](RULE-b49142be) enforces: "No invoke() in $lib/components/"

## Rationale

Without a formal pipeline, standards emerge ad-hoc. A rule might exist without the skill that teaches compliance. A skill might exist without the decision that justifies it. The pipeline ensures every standard has traceable provenance and every stage has the artifacts it needs.

## Consequences

**Positive:**
- Clear provenance for every standard
- Gaps in the pipeline are visible (a rule without a grounding decision is suspicious)
- New standards have a defined path from observation to enforcement
- The learning loop is mechanical, not aspirational

**Negative:**
- More artifacts to maintain
- Verification artifacts (VER-NNN) are a new concept to learn
- Over-formalization risk — not every observation needs to become a rule
