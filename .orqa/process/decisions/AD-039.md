## Decision

The artifact graph has two layers with different editability:

| Layer | Editable in App? | Updated By | Contains |
|-------|-----------------|-----------|----------|
| **Core (firmware)** | No — read-only | OrqaStudio update/release system | Artifact schemas, relationship definitions, traversal instructions, process model, role definitions, core skills, safety constraints |
| **Project** | Yes — freely editable | Users and agents during normal work | Project rules, project skills, docs, tasks, epics, research, decisions, lessons |

**Dogfood exception**: When `project.json` has `"dogfood": true`, core artifacts
are editable because the developer IS the update system.

### What's Core (Firmware)

1. **Artifact type schemas** — `schema.json` in every artifact directory. These
   define what fields exist, what types they have, what edges connect artifact types.
2. **Core skills** — `composability`, `planning`, `research-methodology`,
   `orqa-code-search`. Universal methodology applicable to all projects and domains.
3. **Orchestrator prompt** — `orchestrator.md`. Defines graph traversal and the
   process model.
4. **Role definitions** — Agent definitions in `.orqa/process/agents/`. The 7 universal
   roles and their ownership boundaries.
5. **Safety constraints** — NON-NEGOTIABLE rules that apply everywhere (no unwrap,
   no --no-verify, documentation-first, honest reporting).

### Three-Layer Protection

1. **[RULE-98682b5e](RULE-98682b5e) enforcement entries** — The enforcement engine blocks agents from
   writing to core files. Fires via the companion plugin's PreToolUse hook on
   `file` events. Agents are blocked before they can save the edit.

2. **Pre-commit hook** — Git commits modifying core files are blocked with a
   warning. Override via `ORQA_CORE_OVERRIDE=1` environment variable. Catches
   changes made outside the agent system.

3. **App UI** (future) — Core files render as read-only in the artifact editor.
   No edit affordance is shown.

All three respect the dogfood flag.

### Why Firmware Matters

The core graph is the foundation everything builds on. If a schema is changed:
- Existing artifacts may fail validation
- Graph traversal instructions become wrong
- The plugin injects incorrect context
- The thinking process breaks down

This is the same risk class as modifying a database schema without a migration.
Downstream systems depend on the contract.

### Update Model

- Core artifacts carry a version (tracked in `project.json` or a version file)
- Updates are distributed via OrqaStudio releases
- Users cannot modify core artifacts in the app
- The update system handles schema migrations when core schemas evolve

## Rationale

1. **Stability**: Projects depend on consistent graph structure. An accidental
   schema change could break all artifacts in a project.

2. **Domain agnosticism**: Core skills and process model must work for ANY domain.
   Allowing project-specific modifications to core skills would make them
   non-portable.

3. **Trust boundary**: Users trust that OrqaStudio's thinking methodology works.
   Core artifacts embody that methodology. Users customize via project-layer
   artifacts, not by modifying the engine.

4. **Self-enforcement**: [RULE-98682b5e](RULE-98682b5e) uses the enforcement engine to protect itself.
   The rule defines enforcement entries that block writes to core files. The
   enforcement engine reads these entries and applies them. The rule IS the
   protection, consumed by the same system it protects.

## Consequences

### Positive

- Projects are protected from accidental core modifications
- Core skills remain portable across all domains
- Update system can evolve core artifacts with migration support
- Self-enforcing — no external protection mechanism needed

### Negative

- Dogfood development requires awareness of the protection bypass
- Adding new core artifacts requires updating the protection list in [RULE-98682b5e](RULE-98682b5e)
- Users who want to customise the thinking methodology must use project-layer
  overrides, not modify core directly

### Risks

- **Over-protection**: Blocking too many files could make legitimate customisation
  difficult. Mitigated by keeping the core list minimal and making project-layer
  artifacts freely editable.
- **Bypass fatigue**: If developers routinely override the protection, it loses
  its signal value. Mitigated by only blocking truly foundational artifacts.
