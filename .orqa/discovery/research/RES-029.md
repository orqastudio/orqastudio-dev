## Problem Statement

OrqaStudio has 8 individual activity bar items and is about to add 6 more artifact types. The navigation is already crowded and will become unusable. Meanwhile, governance artifacts live in `.claude/` (a platform-specific directory) while planning artifacts live in `.orqa/`. This split means the viewer reads from two different roots, the project has no portability story, and the orchestrator config is spread across `CLAUDE.md`, `AGENTS.md`, and `.claude/settings.json`.

Three problems, one epic:
1. **Navigation bloat** — individual items need grouping
2. **Platform lock-in** — `.claude/` is Claude CLI-specific; `.orqa/` should be the universal source of truth
3. **Disconnected artifacts** — cross-references exist in frontmatter but aren't navigable

## Architectural Compliance

| Principle | How This Plan Complies |
|-----------|----------------------|
| Error propagation | All new Rust scan/read commands return `Result<T, OrqaError>`. Missing files, parse failures, and broken symlinks are typed errors. |
| IPC boundary | New artifact types follow the existing pattern: Rust `#[tauri::command]` → `invoke()` → store → component. No new communication channels. |
| Component purity | New viewer components receive artifact data as props. Stores call `invoke()`. Viewers render. |
| Type safety | Each artifact type gets a Rust struct with `Serialize`/`Deserialize` and a matching TypeScript interface. |
| Immutability | Artifact data is read-only in the viewer (editing is [EPIC-fe75b52c](EPIC-fe75b52c)). |
| UX-first | Navigation design drives the backend grouping, not the other way around. |
| End-to-end | Every new artifact type ships with: Rust scanner + reader → IPC type → store method → viewer component → navigation entry. |
| Svelte 5 only | All new components use runes. No Svelte 4 patterns. |

## Systems Architecture Checklist

### Data Persistence
No new SQLite tables. All artifacts are markdown files on disk read via Rust's `std::fs`. The scan commands return directory listings with parsed frontmatter; read commands return full file content.

### IPC Contract

| `milestone_read` | `rel_path: String` | `Artifact` | Read single milestone |
| `epic_list` | — | `Vec<ArtifactSummary>` | Scan `.orqa/epics/` |
| `epic_read` | `rel_path: String` | `Artifact` | Read single epic |
| `task_list` | — | `Vec<ArtifactSummary>` | Scan `.orqa/tasks/` |

| `idea_list` | — | `Vec<ArtifactSummary>` | Scan `.orqa/ideas/` |
| `idea_read` | `rel_path: String` | `Artifact` | Read single idea |
| `decision_list` | — | `Vec<ArtifactSummary>` | Scan `.orqa/decisions/` |

### State Management

### Cross-Link Navigation

Two cross-linking mechanisms:

**1. Frontmatter links** — Rendered as clickable chips in the structured header (described above).

**2. Body links** — In the markdown body, artifact references are detected by regex and rendered as clickable links:
- Pattern: `(MS|EPIC|TASK|IDEA|AD|IMPL)-\d{3}` → link to the artifact
- Pattern: `.orqa/path/to/file.md` → link to the artifact in the appropriate viewer

When a cross-link is clicked:
1. Parse the ID prefix to determine artifact type
2. Map type to group: `EPIC/TASK/IDEA/MS → Planning`, `AD/IMPL → Governance`, agent names → Team
3. Set the activity to that group
4. Set the sub-category to that type
5. Open the artifact in the viewer

- Try `std::os::windows::fs::symlink_dir` / `symlink_file` first
- If permission denied, fall back to junction points for directories (`junction` crate or `cmd /c mklink /J`)
- For files without symlink permission, fall back to a copy-on-change watcher (last resort)
- Document that developer mode enables symlinks without elevation

### 1d. Platform Status Command

`platform_status()` → `Vec<PlatformStatus>`:
- For each enabled platform, check every expected symlink
- Report: valid, broken, missing

## Phase 5: Cross-Linking

### 5a. Frontmatter Cross-Links

In `FrontmatterHeader`, any field containing an artifact ID or file path renders as `ArtifactLink`:
- `ArtifactLink` component: displays ID as a styled chip with type icon
- On click: calls `navigationStore.navigateToArtifact(id)`

`navigateToArtifact(id)` logic:
1. Parse prefix: `MS` → milestones, `EPIC` → epics, `TASK` → tasks, `IDEA` → ideas, `AD` → decisions, `IMPL` → lessons
2. Map to group: milestones/epics/tasks/ideas/research/plans → `planning`, decisions/lessons/rules/hooks → `governance`, agents/skills → `team`
3. Set `activeGroup`, `activeSubCategory`, open the artifact

### 5b. Body Cross-Links

In `MarkdownRenderer`, add a post-processing step:
- Regex: `\b(MS|EPIC|TASK|IDEA|AD|IMPL)-(\d{3})\b`
- Replace matched text with clickable `<button>` elements styled as inline links
- On click: same `navigateToArtifact(id)` flow
- Don't match inside code blocks or existing links

### 5c. File Path Links

References to `.orqa/` paths in frontmatter or body:
- Detect paths matching `.orqa/(research|plans|decisions|...)/...`
- Map to the appropriate group and sub-category
