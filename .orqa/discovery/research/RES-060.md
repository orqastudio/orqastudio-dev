## What Personas Represent

Personas are archetypal descriptions of user types or stakeholder profiles that a product serves. In OrqaStudio's context, they would capture:

- **User archetypes** -- the distinct modes of interaction with the product (solo developer, team lead, governance auditor, new user onboarding)
- **Stakeholder profiles** -- people or roles that influence product direction without being direct users (project sponsors, open-source contributors)
- **Behavioral patterns** -- goals, workflows, pain points, and decision-making criteria that differ across persona types

Personas exist at the intersection of product strategy (who are we building for?) and feature prioritisation (which persona does this epic serve?).

## Proposed Schema

```yaml
id: PERSONA-cda6edd6
title: "Solo Developer"
description: "Individual developer using OrqaStudio for personal project governance"
status: draft | validated | archived
created: 2026-03-14
updated: 2026-03-14
role: "Developer"
goals:
  - "Maintain project structure without overhead"
  - "Capture decisions as they happen"
pain-points:
  - "Governance feels heavy for one person"
  - "Context loss between sessions"
pillars:
  - PILLAR-569581e0
relationships:
  - type: serves
    target: EPIC-be023ed2
    rationale: "Feature designed for solo workflow"
```

### Key Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string (PERSONA-NNN) | Yes | Unique identifier |
| `title` | string | Yes | Persona name |
| `description` | string | Yes | One-sentence summary |
| `status` | enum | Yes | Lifecycle state |
| `role` | string | Yes | Professional role or archetype label |
| `goals` | string[] | Yes | What this persona wants to achieve |
| `pain-points` | string[] | Yes | Frustrations and obstacles |
| `pillars` | PILLAR-NNN[] | No | Which pillars this persona cares about most |
| `relationships` | array | No | Graph connections to features, epics, decisions |

## Proposed Lifecycle

```
draft --> validated --> archived
```

- **draft**: Persona is hypothesised but not yet confirmed through user research or usage data.
- **validated**: Persona has been confirmed through real usage patterns, interviews, or analytics. Goals and pain points are evidence-based.
- **archived**: Persona is no longer relevant (product pivot, market change). Historical record preserved.

## Proposed Location

Two options considered:

| Option | Path | Rationale |
|--------|------|-----------|
| A | `.orqa/delivery/personas/` | Personas are delivery-adjacent -- they inform what gets built |
| B | `.orqa/product/personas/` | Personas are product strategy artifacts |

**Recommendation: Option A (`.orqa/delivery/personas/`)** -- personas directly inform epic prioritisation and feature scoping, which are delivery concerns. They sit alongside ideas, epics, and research as inputs to the delivery pipeline.

## Relationship Types

New bidirectional relationship pair:

| Type | Inverse | Meaning |
|------|---------|---------|
| `serves` | `served-by` | "This feature/epic serves this persona" |
| `served-by` | `serves` | "This persona is served by this feature/epic" |

These would be added to the relationship type enums in all schema.json files and to the INVERSE_MAP in @orqastudio/types.

### Usage Examples

- Epic frontmatter: `relationships: [{ type: "serves", target: "PERSONA-cda6edd6" }]`
- Persona frontmatter: `relationships: [{ type: "served-by", target: "EPIC-be023ed2" }]`
- Priority scoring: epics that serve validated personas score higher than those serving draft personas

## Pros of Top-Level Artifact Type

1. **Graph connectivity** -- personas become nodes that can be traversed, queried, and visualised alongside epics and pillars
2. **Prioritisation signal** -- "which personas does this epic serve?" becomes a first-class query, not a comment in a doc
3. **Validation tracking** -- the draft-to-validated lifecycle forces evidence-based persona development
4. **Consistency** -- same frontmatter + schema.json pattern as all other artifact types
5. **Cross-referencing** -- bidirectional `serves`/`served-by` links create traceable persona-to-feature mappings

## Cons of Top-Level Artifact Type

1. **Schema overhead** -- another schema.json to maintain, another artifact type in the scanner, another ID prefix to track
2. **Low volume** -- most projects have 3-8 personas; the artifact machinery may feel heavy for such a small set
3. **Validation difficulty** -- unlike tasks or epics, persona validation is subjective and hard to automate
4. **Relationship noise** -- if every epic must declare which personas it serves, the relationship arrays grow without proportional insight
5. **Duplication risk** -- persona descriptions often overlap with user journey documentation that already exists in `.orqa/documentation/about/`

## Alternative: Documentation Pages

Personas could remain as markdown pages in `.orqa/documentation/about/personas/` without schema-validated frontmatter. They would be human-readable but not graph-connected.

| Dimension | Artifact Type | Documentation Page |
|-----------|--------------|-------------------|
| Graph queries | Yes | No |
| Schema validation | Yes | No |
| Bidirectional links | Yes | No (manual cross-refs only) |
| Maintenance cost | Higher | Lower |
| Sufficient for <5 personas | Overkill | Adequate |
| Sufficient for >10 personas | Necessary | Breaks down |

## Recommendation

**Start as documentation pages, promote to artifact type when needed.**

For most projects, 3-5 persona pages in `.orqa/documentation/about/personas/` with manual cross-references are sufficient. The overhead of a full artifact type (schema, scanner integration, relationship pairs, integrity checks) is not justified until:

1. The project has more than 8 personas
2. Persona-to-feature traceability is actively used for prioritisation
3. Multiple teams need to query "which features serve persona X?"

When those conditions are met, promote to a top-level artifact type using the schema and lifecycle proposed above. The `serves`/`served-by` relationship pair should be added to the relationship type enums at that point.

This is consistent with OrqaStudio's general philosophy: start lightweight, add structure when the lightweight approach breaks down.
