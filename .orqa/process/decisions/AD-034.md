## Decision

The artifact browser's filter panel and sort dropdown generate their options dynamically by reading each artifact type's `schema.json` file:

- **Filterable fields**: Any property with an `enum` array becomes a filter section. The section heading is the humanised field name, and the filter options are the enum values.
- **Sortable fields**: Any property with `type: "string"` and `format: "date"` becomes a date sort option. The `title` field is always sortable.
- **No hardcoded field lists**: The toolbar does NOT contain artifact-specific logic like "if epics, show priority filter". It operates purely on schema metadata.

This means adding a new enum field to any artifact type's schema automatically makes it available as a filter — no frontend code changes required.

## Rationale

Hardcoded filter/sort logic creates a maintenance burden: every new artifact type or new frontmatter field requires a code change to the toolbar. It also introduces type-specific UI logic into what should be a generic navigation component.

By deriving options from the schema:

- **New fields are free** — add an enum to `schema.json`, it appears in the filter panel automatically
- **New artifact types are free** — create a directory with a `schema.json`, the browser handles it
- **No type-specific UI code** — the toolbar component is generic across all artifact types
- **Schema is already the source of truth** — pre-commit validation uses the same schemas, so the filter options always reflect what's actually valid

Visual decorators (status dots, priority colours) are layered on top by field name convention — they enhance the generic rendering without replacing it.

## Consequences

- The Rust artifact scanner must extract `filterable_fields` and `sortable_fields` from each type's `schema.json` and include them in the `NavType` response
- `DocNode` must carry a `frontmatter` bag (`Record<string, unknown>`) so the frontend can access any field for filtering, not just `status`
- Schema changes have UI side effects — adding or removing enum values changes the filter panel
- Fields that are technically enums but shouldn't be filterable (if any arise) would need a schema-level annotation to exclude them
