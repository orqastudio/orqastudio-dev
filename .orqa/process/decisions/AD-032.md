## Decision

SQLite is used **only for conversation persistence**. All other structured data in OrqaStudio lives in the file-based artifact system.

### What Goes in SQLite

- Conversation sessions (high-volume, append-heavy)
- Messages and tool calls within sessions
- Streaming state and metrics
- Any future conversation-adjacent data (e.g., message search indexes)

### What Does NOT Go in SQLite

- Governance artifacts (rules, lessons, decisions, hooks) — `.orqa/process/`
- Planning artifacts (epics, tasks, ideas, research, milestones, pillars) — `.orqa/delivery/`
- Project configuration and settings — `.orqa/project.json`
- Artifact relationships and cross-references — the artifact node graph (in-memory, built from files)
- User preferences — `project.json` fields
- Agent, skill, and team definitions — `.orqa/process/`

### The File-Based Architecture

The artifact system is the database:

- **Storage**: Markdown files with YAML frontmatter in `.orqa/`
- **Schema enforcement**: JSON Schema validation + body template linting via pre-commit hooks
- **Query layer**: The bidirectional artifact node graph [EPIC-0a8a5e72](EPIC-0a8a5e72) provides relationship queries, backreferences, orphan detection, and type-based lookups
- **Live updates**: File watcher rebuilds the graph on `.orqa/` changes

This means governance data is human-readable, git-versioned, diff-friendly, and portable across tools. The node graph provides the relational capabilities that would otherwise require a database.

### Gate for New SQLite Usage

Any proposal to store data in SQLite MUST answer:

1. **Why can't this be a file with frontmatter?** If the data has identity, status, and relationships, it belongs in the artifact system.
2. **Why can't the node graph provide the query?** If the need is relational lookups or backreferences, the graph already handles this.
3. **Is this high-volume append data?** Conversation messages are written hundreds of times per session. Governance artifacts are written occasionally. Volume is the key differentiator.
4. **Does this need to survive without `.orqa/`?** Conversation history is app-internal state. Governance data is project-portable.

If the answer to questions 1-2 is "it can be file-based" and the answer to question 3 is "no", the data belongs in files, not SQLite.

## Rationale

- The governance layer is inherently document-oriented — each artifact is a self-contained markdown file with typed frontmatter
- Git versioning provides history, branching, and collaboration for free
- The artifact node graph [EPIC-0a8a5e72](EPIC-0a8a5e72) provides relational query capabilities without a separate database
- Conversation data is fundamentally different: high-frequency writes, sequential access patterns, no need for human editing or git diffing
- Keeping SQLite scoped prevents scope creep where convenience leads to duplicating file-based data in the database

## Known Violations (Pending Migration)

The following SQLite tables predate this decision and violate it:

- **`governance_analyses`** — AI-generated governance scan results. Should produce research artifacts in `.orqa/delivery/research/` that can be promoted to epics/tasks via the standard artifact lifecycle.
- **`governance_recommendations`** — AI-generated governance recommendations. Should be inline within the research artifact, not a separate table.

These tables and their associated repo/command code should be removed. Governance scan results flow through the artifact pipeline: scan → research artifact → promote to epic/task.

### Artifacts Table Clarification

The **`artifacts`** table is a **read-through cache** of file-based artifacts, not a source of truth. Files in `.orqa/` are authoritative. The table stores metadata (path, hash, last-scanned timestamp) to optimize scanning and avoid re-reading unchanged files. If the table is deleted, the system rebuilds it from disk on next scan.

## Consequences

- New features that need structured storage default to file-based artifacts, not SQLite tables
- The artifact node graph becomes the primary query mechanism for governance data
- SQLite schema stays small and focused (sessions, messages, metrics, artifact cache)
- Any future "should we put this in SQLite?" discussion starts from "no" and requires justification
