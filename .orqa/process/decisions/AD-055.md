# AD-055: Organisation-mode multi-project architecture

## Context

OrqaStudio currently operates on a single project (one `.orqa/` directory, one git repo). As the ecosystem grows — with multiple library repos, the app, plugins, and the product governance itself — there's a need for cross-project visibility without forcing a specific directory structure.

Additionally, the distinction between **core artifacts** (always present in the app) and **plugin-provided artifacts** (registered at runtime) needs to be formalised. This decision defines the architecture for both.

## Decision

### Organisation mode

A project can declare `"organisation": true` in `project.json`. When enabled:

1. **Child projects are configured by path** in `project.json`:
   ```json
   {
     "organisation": true,
     "projects": [
       { "name": "orqastudio-app", "path": "../orqa-studio" },
       { "name": "orqastudio-sdk", "path": "../orqa-sdk" },
       { "name": "orqastudio-types", "path": "../orqa-types" },
       { "name": "delivery-plugin", "path": "../orqastudio-delivery-plugin" }
     ]
   }
   ```

2. **No directory nesting required.** Projects can live anywhere on the filesystem. Git submodules are one way to co-locate them but not mandatory. The organisation project references them by path.

3. **The graph builds sub-graphs per child project.** Each child project's `.orqa/` artifacts are scanned and added as a sub-graph connected back to the organisation root. The graph engine knows which sub-graph each artifact belongs to via a `project` property on each node.

4. **The SDK traverses the unified graph.** When listing artifacts by type, the SDK crawls all sub-graphs, returning results tagged with their source project. This enables:
   - **Organisation view**: show all epics across all projects
   - **Filtered view**: show only epics from the SDK project
   - **Single-project mode**: drill down into one project, hiding others
   - **Cross-project relationships**: an epic in the app project can reference a task in the SDK project

5. **The organisation project has its own `.orqa/` artifacts** — product-level vision, pillars, decisions, milestones that span multiple child projects.

### Core vs plugin artifact split

| Layer | Artifacts | Owner | Where |
|---|---|---|---|
| **Core (app-bundled)** | Pillars, vision, personas, grounding, decisions, rules, lessons, skills, agents | App repo | Always present |
| **Plugin (registered)** | Milestones, epics, tasks, ideas, research + delivery views | `@orqastudio/plugin-delivery` | Loaded via plugin manifest |
| **Plugin (registered)** | Any future artifact types | Third-party plugins | Loaded via plugin manifest |

Core artifacts are defined in the platform layer (`project.json` schema) and cannot be removed. Plugin artifacts are registered at runtime via `orqa-plugin.json` manifests and appear in the navigation, graph, and views only when their plugin is active.

### Plugin loading flow

1. App starts → loads core artifact type definitions from platform config
2. Plugin manifests (`orqa-plugin.json`) are discovered (from `plugins/` directory or organisation config)
3. Each plugin's `provides.artifactTypes` are registered with the type system
4. Each plugin's `provides.views` are registered with the navigation system
5. Each plugin's `provides.icons` are merged into the icon registry
6. The graph engine scans `.orqa/` directories using the merged type definitions

## Consequences

- **Projects are portable.** Moving a repo on disk only requires updating the path in the organisation's `project.json`.
- **No git submodule requirement.** Submodules are convenient for the dev environment but not architecturally required.
- **Plugin extraction has a clear target.** The delivery plugin is the first extraction — milestones, epics, tasks, roadmap views, kanban views all move out of the app.
- **The organisation graph is eventually consistent.** When a child project's artifacts change, the organisation re-scans that sub-graph.
- **Claude context advantage.** An organisation project with all repos as submodules gives Claude full context across the ecosystem in one session.

## Alternatives considered

1. **Monorepo with workspaces** — Forces all code into one repo. Rejected because OrqaStudio repos are intentionally separate for independent versioning and licensing.
2. **Nested `.orqa/` directories** — Child projects must be subdirectories. Rejected because it imposes filesystem structure and makes git submodules mandatory.
3. **Registry-based project discovery** — Projects register themselves with a central service. Rejected as over-engineered for the current stage.
