## Overview

This document specifies three interconnected schemas that form the plugin-provider-capability system:

1. **Extended plugin.json** -- declares what a plugin is, what it needs, and what defaults it provides
2. **Provider definition** (`.orqa/providers/<name>.json`) -- declares an AI provider's identity, detection, and plugin requirements
3. **Capability routing** (in `project.json`) -- user-configurable per-project capability fulfilment preferences

These three schemas interact through a resolution chain: the app detects active providers, loads matching plugins, merges capability defaults, and applies user overrides.

---

## Schema 1: Extended plugin.json

### Location

`.orqa/plugins/<plugin-name>/.claude-plugin/plugin.json` (for Claude Code plugins) or `.orqa/plugins/<plugin-name>/plugin.json` (generic).

### Schema

```jsonc
{
  // Identity (existing fields, unchanged)
  "name": "orqastudio-claude-plugin",
  "version": "0.1.0",
  "description": "OrqaStudio governance companion plugin for Claude Code.",
  "author": {
    "name": "OrqaStudio"
  },

  // NEW: Plugin type taxonomy
  // Array — a single plugin may span multiple types.
  // At least one type is required.
  "type": ["ai-provider-integration", "governance-extension"],

  // NEW: Requirements — shape depends on declared types.
  // Only fields relevant to declared types are valid.
  "requires": {
    // Present when type includes "ai-provider-integration"
    "ai-providers": {
      // "any-of" = any one listed provider satisfies the requirement
      // "all" = every listed provider must be present
      "any-of": ["anthropic"]
    }
    // Future type-specific requires fields would appear here:
    // "data-sources": { ... }  -- for "data-connector" type
    // "renderers": { ... }     -- for "visualisation" type
  },

  // NEW: Default capability mappings this plugin provides.
  // These are the plugin's recommended defaults for how capabilities
  // should be fulfilled. Users can override per-project in project.json.
  "default-capabilities": {
    // Each key is a capability name from the capability vocabulary (RULE-92dba0cb).
    // The value specifies HOW that capability is fulfilled.
    "code_search_regex": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "search_regex"
    },
    "code_search_semantic": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "search_semantic"
    },
    "code_research": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "code_research"
    }
  },

  // NEW: Capabilities this plugin ADDS beyond the app baseline.
  // These extend the capability vocabulary for this plugin's provider.
  "extended-capabilities": {
    "skill_load": {
      "description": "Load a skill into agent context via CLI slash command",
      "provider": "native",
      "tool": "Skill"
    }
  },

  // NEW: Installation wiring — what this plugin contributes when installed.
  // The plugin installer uses this to wire capabilities into the project.
  "installs": {
    // Skills contributed by this plugin
    "skills": [
      "chunkhound",
      "orqa-code-search"
    ],
    // Agent capability additions — appended to matching agents
    "agent-capabilities": {
      "*": ["code_search_regex", "code_search_semantic", "code_research"]
    },
    // Hooks contributed (reference to hooks config)
    "hooks": "hooks/hooks.json"
  }
}
```

### Type Taxonomy

| Type | Description | Requires Fields |
|------|-------------|-----------------|
| `ai-provider-integration` | Bridges an AI provider's tooling with OrqaStudio | `requires.ai-providers` |
| `governance-extension` | Adds governance rules, skills, hooks, or enforcement | (none type-specific) |
| `visualisation` | Adds rendering or display capabilities | (future: `requires.renderers`) |
| `data-connector` | Connects external data sources | (future: `requires.data-sources`) |

The type array is open -- new types can be added without schema changes. The `requires` object is a union of all type-specific requirement fields. Only fields matching declared types are validated.

### Capability Fulfilment Value Shape

Each capability mapping value follows one of these shapes:

```jsonc
// MCP server tool
{
  "provider": "mcp",
  "server": "<server-name>",
  "tool": "<tool-name>"
}

// Native provider tool (built into the AI provider's CLI/app)
{
  "provider": "native",
  "tool": "<tool-name>"
}

// App-provided tool (OrqaStudio's own implementation)
{
  "provider": "app",
  "tool": "<tauri-command-name>"
}
```

### Validation Rules

1. `type` must be a non-empty array of strings
2. If `type` includes `"ai-provider-integration"`, `requires.ai-providers` must be present
3. `requires.ai-providers` must have exactly one key: either `any-of` or `all`, containing a non-empty array of provider IDs
4. `default-capabilities` keys must be valid capability names from the capability vocabulary
5. `extended-capabilities` keys must NOT collide with existing vocabulary entries (they extend, not override)
6. A plugin without `requires.ai-providers` is valid -- it simply is not provider-dependent (e.g., a pure `governance-extension` plugin)

## Schema 2: AI Provider Definition

### Location

`.orqa/providers/<provider-id>.json` -- one file per provider. These are app-native config files, not plugin artifacts. They define what AI providers exist and how to detect them.

### Schema

```jsonc
{
  // Identity
  "id": "anthropic",
  "name": "Anthropic Claude",
  "description": "Claude AI models via Anthropic API or Claude Code CLI",

  // How to detect this provider is available/active
  "detection": {
    // Environment variable check — if this env var exists and is non-empty,
    // the provider is considered available
    "env": "ANTHROPIC_API_KEY",

    // Process check (optional) — if a process matching this name is running
    "process": "claude",

    // Config file check (optional) — if this file exists
    "config-file": "~/.claude.json"
  },

  // Plugins this provider requires to function in OrqaStudio.
  // The plugin installer ensures these are present.
  "required-plugins": ["orqastudio-claude-plugin"],

  // Plugins that enhance this provider but aren't required.
  "optional-plugins": [],

  // The native capability vocabulary this provider supports
  // (tools the provider's CLI/app gives you for free, without plugins)
  "native-capabilities": {
    "file_read": "Read",
    "file_write": "Write",
    "file_edit": "Edit",
    "file_search": "Glob",
    "content_search": "Grep",
    "shell_execute": "Bash",
    "web_fetch": "WebFetch",
    "web_search": "WebSearch",
    "notebook_edit": "NotebookEdit"
  }
}
```

### Example: OpenAI Provider

```jsonc
{
  "id": "openai",
  "name": "OpenAI",
  "description": "OpenAI models via API",
  "detection": {
    "env": "OPENAI_API_KEY"
  },
  "required-plugins": [],
  "optional-plugins": ["orqastudio-openai-plugin"],
  "native-capabilities": {}
}
```

### Example: OrqaStudio App Provider

```jsonc
{
  "id": "orqa-app",
  "name": "OrqaStudio Built-in",
  "description": "OrqaStudio's native Agent SDK — the app IS the provider",
  "detection": {
    "config-file": ".orqa/project.json"
  },
  "required-plugins": [],
  "optional-plugins": [],
  "native-capabilities": {
    "file_read": "read",
    "file_write": "write",
    "file_edit": "edit",
    "file_search": "glob",
    "content_search": "grep",
    "shell_execute": "bash",
    "code_search_regex": "search_regex",
    "code_search_semantic": "search_semantic",
    "code_research": "code_research",
    "skill_load": "load_skill",
    "web_fetch": "web_fetch",
    "web_search": "web_search",
    "notebook_edit": "notebook_edit"
  }
}
```

### Validation Rules

1. `id` must be a unique lowercase string matching `^[a-z][a-z0-9-]*$`
2. `detection` must have at least one detection mechanism (`env`, `process`, or `config-file`)
3. `required-plugins` entries must reference plugins that exist in `.orqa/plugins/`
4. `native-capabilities` keys must be valid capability vocabulary entries
5. The filename must match `<id>.json`

### Detection Semantics

Detection mechanisms are evaluated in order: `env` > `process` > `config-file`. The first match determines availability. Multiple mechanisms are OR-ed -- any one passing means the provider is detected.

Detection answers "is this provider available?" not "is this the active provider." A system may have multiple providers detected (e.g., both `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` set). The active provider is determined by the session context (which CLI tool launched the session, or which model the user selected in the app).

## Schema 3: Capability Routing Config

### Location

`project.json` gains a new top-level `capabilities` field. This is the user-override layer.

### Schema

```jsonc
{
  // ... existing project.json fields ...

  // NEW: Per-project capability routing overrides.
  // Resolution order: project config > plugin defaults > app baseline
  "capabilities": {
    // Override specific capability fulfilment
    "code_search_regex": {
      "provider": "app",
      "tool": "search_regex"
    },
    // Use the plugin default for everything else by omitting it
  }
}
```

### Resolution Chain

When the system needs to fulfil a capability:

```
1. Check project.json "capabilities" for an explicit override
   → If found, use it

2. Check active plugins' "default-capabilities" for a mapping
   → If found, use it
   → If multiple plugins provide defaults for the same capability,
     use the plugin with higher specificity (ai-provider-integration
     over governance-extension)

3. Fall back to the detected provider's "native-capabilities"
   → If found, use the native tool name

4. Capability is unavailable — report gap explicitly
```

### Example: Full Resolution

Given:
- Provider detected: `anthropic` (via `ANTHROPIC_API_KEY`)
- Plugin active: `orqastudio-claude-plugin`
- Project config has no capability overrides

Resolving `code_search_regex`:
1. `project.json` capabilities: not present -- skip
2. Plugin `default-capabilities`: `{ "provider": "mcp", "server": "chunkhound", "tool": "search_regex" }` -- use this
3. Result: `mcp__chunkhound__search_regex`

Resolving `file_read`:
1. `project.json` capabilities: not present -- skip
2. Plugin `default-capabilities`: not present -- skip
3. Provider `native-capabilities`: `"Read"` -- use this
4. Result: `Read`

### Example: User Override

A user wants to use the app's native search instead of ChunkHound:

```jsonc
// project.json
{
  "capabilities": {
    "code_search_regex": { "provider": "app", "tool": "search_regex" },
    "code_search_semantic": { "provider": "app", "tool": "search_semantic" },
    "code_research": { "provider": "app", "tool": "code_research" }
  }
}
```

Now `code_search_regex` resolves to the app's native `search_regex` Tauri command instead of ChunkHound, even though the Claude plugin is active and provides ChunkHound defaults.

## Interaction Model

### Plugin Installation Flow

```
1. Plugin placed in .orqa/plugins/<name>/
2. System reads plugin.json:
   a. Validate type array
   b. Check requires against detected providers
   c. If requires not met, plugin is installed but inactive (greyed out in UI)
3. If active:
   a. Register default-capabilities in the resolution chain
   b. Wire installs.skills into the project's skill index
   c. Wire installs.agent-capabilities into agent definitions
   d. Register installs.hooks with the hook engine
4. Plugin is now live
```

### Provider Detection Flow

```
1. System scans .orqa/providers/*.json
2. For each provider, run detection checks
3. Build list of detected (available) providers
4. Determine active provider from session context
5. Filter plugins: only plugins whose requires.ai-providers
   includes the active provider are activated
6. Plugins without requires.ai-providers are always active
   (they're provider-independent)
```

### Capability Resolution Flow

```
1. Agent needs capability X
2. Resolution chain: project config > plugin defaults > provider native
3. If resolved: tool name is returned
4. If unresolved: gap reported in delegation prompt
```

### Bidirectional Relationship

The provider-plugin relationship is declared from both sides:

- **Provider side:** `required-plugins` and `optional-plugins` list plugin names
- **Plugin side:** `requires.ai-providers` lists provider IDs

This bidirectionality serves different purposes:
- **Provider -> Plugin:** "To use this provider in OrqaStudio, you need these plugins installed"
- **Plugin -> Provider:** "This plugin only activates when one of these providers is detected"

The relationship is pragmatic, not a formal constraint. Per [RES-cd3d33bf](RES-cd3d33bf) decisions, this bidirectional declaration becomes unnecessary when [IDEA-1c7ebe74](IDEA-1c7ebe74) (sidecar-as-plugin) lands and the provider IS the plugin.

## Migration Path for Existing Plugin

The current `orqastudio-claude-plugin` plugin.json:

```json
{
  "name": "orqa-plugin",
  "version": "0.1.0",
  "description": "OrqaStudio governance companion plugin for Claude Code.",
  "author": { "name": "OrqaStudio" }
}
```

Would become:

```json
{
  "name": "orqastudio-claude-plugin",
  "version": "0.2.0",
  "description": "OrqaStudio governance companion plugin for Claude Code. Enforces rules, loads agent context, and provides governance commands.",
  "author": { "name": "OrqaStudio" },
  "type": ["ai-provider-integration", "governance-extension"],
  "requires": {
    "ai-providers": {
      "any-of": ["anthropic"]
    }
  },
  "default-capabilities": {
    "code_search_regex": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "search_regex"
    },
    "code_search_semantic": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "search_semantic"
    },
    "code_research": {
      "provider": "mcp",
      "server": "chunkhound",
      "tool": "code_research"
    }
  },
  "extended-capabilities": {
    "skill_load": {
      "description": "Load a skill into agent context via CLI slash command",
      "provider": "native",
      "tool": "Skill"
    }
  },
  "installs": {
    "skills": ["chunkhound", "orqa-code-search"],
    "agent-capabilities": {
      "*": ["code_search_regex", "code_search_semantic", "code_research"]

  }
```

## Open Questions for Implementation

1. **Schema validation:** Should plugin.json get a formal JSON Schema like artifact frontmatter does? Recommendation: yes, stored alongside the plugin schema documentation.
2. **Plugin load order:** When multiple plugins provide `default-capabilities` for the same capability, the specificity rule (ai-provider-integration wins) handles most cases. What about two plugins of the same type? Recommendation: alphabetical by plugin name as tiebreaker, with a user-visible warning.
3. **Hot reload:** Can plugins be activated/deactivated without restarting? For CLI context, this happens naturally (each session re-evaluates). For app context, this requires the plugin loader to support dynamic filtering.
4. **Detection caching:** Provider detection (env vars, process checks) should be cached per session, not per capability resolution. Environment changes mid-session are not expected.
