## Decision

The `statuses` array in `project.json` defines not just the vocabulary and icons, but also:
- **`transitions`**: which statuses a human can manually move to from this status
- **`auto_rules`**: conditions the system evaluates to propose or apply transitions automatically

## Schema

```json
{
  "key": "active",
  "label": "Active",
  "icon": "circle-dot-dashed",
  "spin": true,
  "transitions": ["hold", "blocked", "review"],
  "auto_rules": [
    { "condition": "all-children-completed", "target": "review" }
  ]
}
```

## Auto Rule Conditions

| Condition | Meaning | Applied To |
|---|---|---|
| `all-children-completed` | All child artifacts (tasks for epics, epics for milestones) have status `completed` | Epics, Milestones |
| `dependency-blocked` | Any `depends-on` artifact is not `completed` | Tasks |
| `dependencies-met` | All `depends-on` artifacts are `completed` | Tasks (blocked → ready) |
| `recurrence-threshold` | Lesson recurrence >= 2 | Lessons |

## Rationale

Hardcoded transitions break when the status vocabulary is customised. A project that adds a "testing" status between "active" and "review" would need code changes. Config-driven transitions let any project define its own lifecycle without touching code.

## Consequences

- Rust transition engine reads rules from config, not hardcoded conditions
- PipelineStepper reads valid transitions from config, not a local map
- Plugin graph-guardian can validate transitions as well as statuses
- New projects can define entirely custom lifecycles
