## Decision

The OrqaStudio core UI is limited to three capabilities for artifact interaction:

1. **Navigate** — browse, sort, group, and filter artifacts through the sidebar and viewer
2. **Search** — AI-driven cross-artifact search that infers intent and presents structured results
3. **Edit** — create, modify, and delete artifacts in-app

All higher-level visualizations — roadmap timelines, process dashboards, dependency graphs, kanban boards, milestone progress views, priority matrices — are **plugin territory**. They are NOT built into the core app.

## Rationale

Different projects have fundamentally different visualization needs. A software project wants sprint boards and dependency graphs. A research project wants knowledge maps and timeline views. A personal project wants simple lists and calendars.

Baking visualization assumptions into the core app:
- Creates bloat for projects that don't need those views
- Forces a single mental model of "how work is organized" onto all users
- Makes the core app harder to maintain as visualization requirements diverge
- Contradicts the domain-agnostic vision — OrqaStudio is not a project management tool, it's a clarity engine

The plugin architecture already supports custom views over the artifact graph. Visualization plugins consume the same `ArtifactGraphSDK` that the core app uses, with full access to the graph's query capabilities (resolve, references, type/status filtering, broken refs, orphans).

### AI-Driven Search

Cross-artifact search uses AI rather than Elasticsearch-style keyword matching. By routing search queries through the AI provider and presenting results in a structured way, we get:
- Semantic understanding of search intent ("show me everything blocking the next milestone")
- Flexible result presentation (the AI structures results, not a fixed template)
- Natural language queries rather than filter syntax
- Context-aware ranking that understands artifact relationships

This aligns with Pillar 1 (Clarity Through Structure) — the search doesn't just find text matches, it helps users understand the *structure* of what they're looking at.

## Consequences

- The core artifact browser must have excellent sorting, grouping, and filtering — since there's no dashboard to fall back on, the browser IS the primary way to find things
- The plugin SDK must be rich enough that visualization plugins can build compelling views without modifying the core app
- AI search quality depends on the prompt engineering and the AI provider's capabilities — this is a feature that improves as models improve
- First-party visualization plugins (roadmap, dependency graph) should ship as `layer: plugin` artifacts alongside the core app, not baked into it
